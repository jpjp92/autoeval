## 🔄 실패/보류 데이터 재생성 설계

### 배경

평가 결과에서 QA 항목은 3가지 상태로 분류됨:

| 상태 | 기준 |
|------|------|
| 성공 | quality_avg ≥ 0.7 AND rag_avg ≥ 0.7 |
| 보류 | 둘 중 하나만 < 0.7 |
| 실패 | quality_avg < 0.7 AND rag_avg < 0.7 |

### 전체 흐름

```
평가 완료
  └── 실패/보류 항목 선택 (체크박스 or 일괄 선택)
        └── 재생성 전략 선택
              └── 재생성 job 시작 → 재평가 → 결과 확인
```

### 재생성 전략

| 전략 | 적용 케이스 | 방법 |
|------|------------|------|
| 같은 모델 재시도 | 일시적 품질 저하 (랜덤성 기대) | 동일 설정 그대로 |
| 모델 교체 | 모델 자체 한계 | 다른 모델 선택 후 재생성 |
| 컨텍스트 확장 | RAG 실패 (근거성·관련성 낮음) | 청크 범위 확대 |

### 결과 처리 방식

| 방식 | 설명 | 장단점 |
|------|------|--------|
| **A. 신규 job** | 재생성 결과를 별도 job으로 저장 → 재평가 후 비교 | 단순, 기존 데이터 보존 |
| **B. 기존 병합** | 기존 job의 실패 항목을 재생성 결과로 교체 | 이력 관리 복잡 |

→ **1단계: A 방식(신규 job)으로 구현**, 이후 필요 시 B 방식 추가

### 구현 단계

**Phase 1 — 기본 재생성 (신규 job)**
- 평가 페이지 실패/보류 항목에 체크박스 추가
- 전체 선택 단축 버튼 (실패만 / 보류만 / 전체)
- "선택 재생성" 버튼 → 해당 **`qa_id` 목록** + 원래 설정(model, lang, hierarchy) 전달
  > ⚠️ `docId` 단위 재생성은 잘못된 구조 — docId 1개에 QA가 여러 개일 경우 전체 재생성됨
  > 재생성 단위 = QA (qa_id 기준)

**Phase 2 — 전략 선택 UI**
- 재생성 전 모달: 모델 교체 / 컨텍스트 확장 옵션 선택
- `regeneration_mode` 선택:

| mode | 설명 | 적용 failure_type |
|------|------|-----------------|
| `reuse_context` | 기존 청크 재활용, 프롬프트/모델만 변경 | `hallucination`, `ambiguous_question` |
| `re_retrieve` | 청크 재검색 후 재생성 (top-k 증가 등) | `retrieval_miss`, `bad_chunk`(조건부) |

**Phase 3 — 결과 비교 뷰**
- 원본 QA vs 재생성 QA 점수 비교 (delta_quality / delta_rag / delta_total)

### Failure Type 분류

재생성 전 각 QA 항목에 실패 원인을 분류하여 전략 자동 분기에 활용

**스키마 (복합 실패 대응)**
```json
{
  "failure_types": ["hallucination", "retrieval_miss"],
  "primary_failure": "hallucination",
  "failure_reason": "사실성 점수 0.42 — 컨텍스트에 없는 내용 포함",
  "confidence": null
}
```
> `failure_type` 단일값 → `failure_types` 배열 + `primary_failure` 로 수정
> 실제는 복합 문제 발생 가능 (hallucination + retrieval_miss 동시 발생)
> `confidence`는 초기 `null` — 점수 역산(1-score) 방식은 확신도와 무관하여 위험, 추후 LLM classifier로 보강

**분류 기준 및 전략**

| failure_type | 판단 기준 | 재생성 전략 |
|--------------|-----------|------------|
| `hallucination` | `factuality < 0.6` | grounding 강화 프롬프트 / 모델 교체 (`reuse_context`) |
| `retrieval_miss` | `groundedness < 0.6` or `relevance < 0.6` | 컨텍스트 확장 (`re_retrieve`, top-k 증가 / reranking) |
| `ambiguous_question` | `clarity < 0.6` | 명확성 강조 프롬프트 재생성 (`reuse_context`) |
| `bad_chunk` | `context 길이 < 100자` → ❌ 재생성 불가 / `chunk 이상(heading/colophon 등)` → `re_retrieve` 허용 | 조건부 허용 (아래 참고) |
| `evaluation_error` | 평가 점수 필드 누락 or 파싱 오류 | 재평가 시도 |

> `bad_chunk` 기준은 현재 길이/타입 기반이나 추후 semantic quality evaluator로 고도화 필요
> (overlap 부족, 의미 단절, 질문 생성 불가 구조 등 semantic 판단 포함)

**bad_chunk 조건부 재생성**

| 조건 | 처리 |
|------|------|
| `context 길이 < 100자` | ❌ 재생성 불가 — 청크 자체가 너무 짧아 QA 생성 의미 없음 |
| `chunk_type = heading/colophon` 등 구조적 불량 | ✅ `re_retrieve` 허용 — 다른 청크로 재검색 후 재생성 가능 |
| `rag_avg < 0.4` (청크 내용은 있으나 품질 매우 낮음) | ✅ `re_retrieve` 허용 |

**primary_failure 결정 규칙**

복수 failure_type 발생 시 우선순위 기반으로 `primary_failure` 자동 선택:

```
우선순위: hallucination > retrieval_miss > ambiguous_question > bad_chunk > evaluation_error
```

동점(같은 우선순위 내) 시: **가장 낮은 점수의 차원** 기준으로 선택
```python
# 예) factuality=0.42, groundedness=0.45 → hallucination (factuality가 더 낮음)
primary_failure = min(failing_types, key=lambda t: dimension_scores[t])
```

**분류 로직 위치**
- 백엔드 `pipeline.py`에서 평가 완료 후 분류 → `qa_scores`에 저장
- `bad_chunk` 해당 항목은 재생성 대상에서 자동 제외 + UI에 이유 표시

### 재생성 전략 (확장)

| 전략 분류 | 전략 | 적용 failure_type |
|----------|------|-----------------|
| **Prompt** | grounding 강화 (citation 강제) | `hallucination` |
| **Prompt** | chain-of-thought 유도 | `hallucination`, `completeness 낮음` |
| **Prompt** | 명확성 강조 | `ambiguous_question` |
| **모델** | 모델 교체 | `hallucination`, 일반 품질 낮음 |
| **Retrieval** | top-k 증가 (청크 범위 확대) | `retrieval_miss` |
| **Retrieval** | reranking 적용 | `retrieval_miss` |

**Auto Strategy** (추후 구현)
- `primary_failure` 기반으로 전략 자동 추천
- 예) `hallucination` 감지 → "grounding 프롬프트 + 모델 교체" 자동 제안

### 재생성 추적 (기존 구조 확장)

> 전용 experiment 테이블 불필요 — 현 단계에서는 `qa_gen_results.metadata` 필드 추가로 충분
> experiment 테이블은 실험이 반복적으로 쌓이고 통계 분석이 필요해지는 시점에 도입

**`qa_gen_results.metadata`에 추가할 필드**
```json
{
  "generation_model": "claude-sonnet",
  "parent_eval_id": "eval_20260320_...",
  "strategy_type": "grounding_prompt",
  "retry_for": ["hallucination"]
}
```

**개선 효과 측정 (delta_score 분리)**
```
delta_quality = new_quality_avg - old_quality_avg
delta_rag     = new_rag_avg     - old_rag_avg
delta_total   = new_final_score - old_final_score
```
- 별도 저장 없이 비교 시점에 계산
- UI 표시: `품질 +0.12 ↑ | RAG +0.05 ↑ | 최종 +0.09 ↑`
- 단일 `delta_score`로 합산하면 품질/RAG 중 어느 쪽이 개선되었는지 불명확 → 분리 필수

### 평가 Reasoning 저장 설계

통과/실패 여부와 무관하게 각 QA 평가 시 LLM이 점수 근거를 함께 반환 → `pipeline_results`에 저장

**적용 레이어**: Layer 2 (RAG Triad), Layer 3 (Quality)
Layer 1-A/B는 규칙/통계 기반이라 이미 수치로 설명됨

### Reasoning → Failure Type 연계 설계

**흐름**
```
LLM 평가 응답 (차원별 reason 포함)
  └── failure_type 분류 로직
        ├── factuality_reason + factuality 점수 → hallucination 여부 판단
        ├── groundedness_reason + groundedness 점수 → retrieval_miss 여부 판단
        ├── clarity_reason + clarity 점수 → ambiguous_question 여부 판단
        └── context 길이/chunk_type → bad_chunk 여부 판단
              └── failure_type + reason + confidence 저장
```

**차원 → failure_type 매핑**

| 낮은 차원 | 기준 | failure_type |
|-----------|------|--------------|
| `factuality` | < 0.6 | `hallucination` |
| `groundedness` or `relevance` | < 0.6 | `retrieval_miss` |
| `clarity` | < 0.6 | `ambiguous_question` |
| context 길이 < 100자 or chunk_type=heading/colophon | — | `bad_chunk` |
| 점수 필드 누락 or 파싱 오류 | — | `evaluation_error` |

**confidence 산출** (추가 LLM 호출 없이 점수 역산)
```
confidence = 1.0 - 해당 차원 점수   # 점수가 낮을수록 failure 확신도 높음
예) factuality = 0.42 → confidence = 0.58
    groundedness = 0.31 → confidence = 0.69
```

**최종 저장 구조** (`pipeline_results.layers.quality/rag.qa_scores[*]`)
```json
{
  "qa_index": 0,
  "factuality": 0.42,  "factuality_reason": "컨텍스트에 없는 수치를 임의로 생성함",
  "completeness": 0.80, "completeness_reason": "질문의 핵심 조건을 모두 다룸",
  "specificity": 0.75,  "specificity_reason": "구체적 예시 포함되어 있으나 단위 누락",
  "conciseness": 0.88,  "conciseness_reason": "간결하고 불필요한 반복 없음",
  "avg_quality": 0.71,
  "pass": true,
  "failure_type": "hallucination",
  "failure_reason": "컨텍스트에 없는 수치를 임의로 생성함",
  "confidence": 0.58
}
```

> `failure_type`은 pass 여부와 무관하게 저장 — 통과 항목도 낮은 차원이 있으면 기록하여 골든셋 후보 검토 시 활용

**구현 파일**
- `backend/evaluators/qa_quality.py` — 프롬프트에 `*_reason` 필드 추가
- `backend/evaluators/rag_triad.py` — 프롬프트에 `*_reason` 필드 추가
- `backend/evaluators/pipeline.py` — failure_type 분류 로직 추가 (차원 점수 + reason 기반)
- `frontend/.../QAEvaluationDashboard.tsx` — QA 상세 뷰에 reason + failure_type 표시

**구현 순서**
1. `qa_quality.py` / `rag_triad.py` 프롬프트 수정 → reason 필드 응답 추가
2. `pipeline.py`에 failure_type 분류 함수 추가
3. DB 저장 구조 확장 (기존 `qa_scores` 배열에 필드 추가, 스키마 변경 불필요)
4. 프론트엔드 QA 상세 뷰에 reasoning + failure_type 배지 표시

#### 1. QAPreviewItem 타입 확장

interface QAPreviewItem {

  failure_reason?: string;       // 주요 실패 차원의 reason 문장

  // 차원별 reason

  completeness_reason?: string;

  relevance_reason?: string;
  groundedness_reason?: string;
  clarity_reason?: string;
}

#### 2. QA 목록 테이블 — failure_type 배지 추가

현재: `상태(성공/보류/실패)` + `품질 점수` + `RAG Triad` 컬럼
변경: 상태 옆에 `failure_type` 배지 추가

```
[ 성공 ]                          → 배지 없음
[ 보류 ] [ retrieval_miss 0.69 ]  → 주황 배지 + confidence
[ 실패 ] [ hallucination 0.88 ]   → 빨간 배지 + confidence
```

**failure_type 배지 색상**
| type | 색상 |
|------|------|
| `hallucination` | rose (빨강) |
| `retrieval_miss` | amber (주황) |
| `ambiguous_question` | yellow (노랑) |
| `bad_chunk` | slate (회색) |
| `evaluation_error` | purple (보라) |

#### 3. QA 상세 뷰 (`QADetailView`) — reasoning 섹션 추가

현재 상세 뷰 구성:
```
헤더 (번호 + 의도 배지 + 상태 배지)
질문 / 답변 / 컨텍스트
점수 카드 2개 (품질 평가 평균 | RAG Triad 평균)
```

변경 후:
```
헤더 (번호 + 의도 배지 + 상태 배지 + failure_type 배지)
질문 / 답변 / 컨텍스트
점수 카드 2개 (품질 평가 평균 | RAG Triad 평균)
─────────────────────────────────────
[NEW] 평가 근거 섹션
  ├── RAG Triad 차원별 점수 + reason
  │     관련성  0.82  "질문의 핵심 키워드가 답변에 포함됨"
  │     근거성  0.31  "컨텍스트에 없는 외부 정보 사용됨"  ← 낮은 차원 강조
  │     명확성  0.78  "질문이 구체적이고 모호하지 않음"
  └── 품질 평가 차원별 점수 + reason
        사실성  0.42  "수치 데이터가 컨텍스트와 불일치"   ← 낮은 차원 강조
        완전성  0.80  "주요 조건을 대부분 다룸"
        구체성  0.75  "예시 포함, 단 단위 누락"
        간결성  0.88  "불필요한 반복 없음"
```

**낮은 차원 강조 기준**: 점수 < 0.7 → 행 배경 rose-50, 텍스트 rose-600
