## 1. DB 재설계 계획 (Option B 전환)

### 1.1 현재 테이블 구조 실태 (코드 레벨 확인)

#### `doc_chunks`

- `document_id`는 업로드 시 `str(uuid4())`로 생성 ([ingestion_api.py:123](../backend/api/ingestion_api.py))
- `metadata` JSONB 안에 `document_id` 키로 저장 — 전용 컬럼 없음
- 모든 필터 쿼리가 `metadata->>'document_id'` 패턴 사용 ([doc_chunk_repo.py:188](../backend/db/doc_chunk_repo.py))
- **리스크**: 인제스션 초기(document_id 도입 이전) 청크는 해당 키 없음 → 전용 컬럼 추출 시 NULL 발생 가능

#### `doc_metadata`

- `document_id` 전용 컬럼 존재 (text UNIQUE) — Option A에서 이미 설계됨
- `/analyze-hierarchy` 실행 시에만 row 생성 → **analyze-hierarchy 미실행 문서는 doc_metadata 행 없음**
- `doc_chunks`와는 애플리케이션 레벨 일치만 보장, DB FK 없음

#### `qa_gen_results`

- `source_doc` (text) 컬럼: filename 저장 — 전용 컬럼으로 존재
- `doc_chunk_ids` (text[]): 생성에 사용된 청크 UUID 배열 — FK 없음
- **`document_id` 미저장**: `save_qa_generation_to_supabase()` 호출 시 metadata에 `generation_model`, `lang`, `source_doc`, `prompt_version`만 포함 — document_id 없음 ([generation_api.py:639-643](../backend/api/generation_api.py))
- `linked_evaluation_id` UUID FK → `qa_eval_results.id` (유일하게 FK 존재)

#### `qa_eval_results`

- `generation_id` 컬럼 없음 — 역방향 탐색 불가
- `qa_gen_results.linked_evaluation_id`로 단방향 참조만 존재

**현재 연결 상태 요약**

| 연결 | 방식 | 문제 |
|------|------|------|
| `doc_chunks` ↔ `doc_metadata` | `metadata->>'document_id'` 문자열 일치 | FK 없음, 삭제 연쇄 수동 |
| `qa_gen_results` → `doc_chunks` | `doc_chunk_ids text[]` | FK 없음, 무결성 미보장 |
| `qa_gen_results` → `doc_metadata` | `source_doc` filename 기반 | FK 없음, filename 중복 시 모호 |
| `qa_gen_results` → `qa_eval_results` | `linked_evaluation_id` UUID FK | ✅ FK 존재 |
| `qa_eval_results` → `qa_gen_results` | 없음 | 역방향 탐색 불가 |

### 1.3 기존 데이터 처리 문제 및 마이그레이션 계획

#### Step 0 (선행): 전체 백업 + 데이터 현황 점검

```sql
-- doc_chunks 중 document_id 없는 행 수 확인
SELECT COUNT(*) FROM doc_chunks WHERE metadata->>'document_id' IS NULL;

-- doc_chunks의 document_id 중 doc_metadata에 없는 것 확인
-- (analyze-hierarchy 미실행 문서)
SELECT DISTINCT metadata->>'document_id' AS doc_id
FROM doc_chunks
WHERE metadata->>'document_id' IS NOT NULL
  AND metadata->>'document_id' NOT IN (SELECT document_id FROM doc_metadata);

-- qa_gen_results의 source_doc 목록
SELECT DISTINCT source_doc FROM qa_gen_results;
```

이 3개 쿼리 결과에 따라 이후 처리 방식이 달라짐.

---

#### Step 1: `doc_chunks` — document_id 전용 컬럼 추출

```sql
-- 1-a: 컬럼 추가
ALTER TABLE doc_chunks ADD COLUMN document_id text;

-- 1-b: 기존 데이터 JSONB → 컬럼 복사 (NULL 허용)
UPDATE doc_chunks SET document_id = metadata->>'document_id';

-- 1-c: 인덱스 생성
CREATE INDEX idx_doc_chunks_document_id ON doc_chunks(document_id);
```

**기존 데이터 처리**
- `metadata->>'document_id'` 없는 구행 → `document_id = NULL` (정상, FK 제약 시 NULLABLE로 허용)
- NULL 행은 어떤 문서에도 귀속되지 않는 고아 청크 — 운영 중 재인제스션 또는 별도 정리 필요

---

#### Step 2: `doc_metadata` 선행 보완 후 FK 연결

FK를 걸기 전에 `doc_chunks`에 존재하는 모든 `document_id`에 대해 `doc_metadata` row가 있어야 함.
`/analyze-hierarchy` 미실행 문서는 `doc_metadata` row가 없으므로 최소 정보로 선행 삽입 필요.

```sql
-- 2-a: doc_chunks에 있지만 doc_metadata에 없는 document_id를 최소 row로 삽입
INSERT INTO doc_metadata (document_id, filename)
SELECT DISTINCT
    metadata->>'document_id' AS document_id,
    metadata->>'filename'    AS filename
FROM doc_chunks
WHERE metadata->>'document_id' IS NOT NULL
  AND metadata->>'document_id' NOT IN (SELECT document_id FROM doc_metadata)
ON CONFLICT (document_id) DO NOTHING;

-- 2-b: FK 연결 (NULL 허용 — 고행 청크는 FK 대상 없음)
ALTER TABLE doc_chunks
  ADD CONSTRAINT fk_doc_chunks_metadata
  FOREIGN KEY (document_id)
  REFERENCES doc_metadata(document_id)
  ON DELETE SET NULL;
```

> `ON DELETE CASCADE` 대신 `ON DELETE SET NULL` 채택 이유:
> CASCADE 시 `doc_metadata` row 삭제 → 해당 청크 전체 삭제 위험.
> 현재 삭제 기능이 UI에 없으므로 SET NULL이 더 안전.

---

#### Step 3: `qa_gen_results` — document_id 컬럼 추가 및 백필

`qa_gen_results`에 `document_id`가 한 번도 저장되지 않았으므로 직접 백필 불가.
`doc_chunk_ids` 배열을 이용해 청크로부터 역추적하는 방식 사용.

```sql
-- 3-a: 컬럼 추가
ALTER TABLE qa_gen_results ADD COLUMN document_id text;

-- 3-b: doc_chunk_ids[1]로 첫 청크 조회 → document_id 역추적 백필
-- (동일 generation의 모든 청크는 같은 document_id를 가짐)
UPDATE qa_gen_results g
SET document_id = (
    SELECT metadata->>'document_id'
    FROM doc_chunks
    WHERE id::text = g.doc_chunk_ids[1]
    LIMIT 1
)
WHERE g.doc_chunk_ids IS NOT NULL
  AND array_length(g.doc_chunk_ids, 1) > 0
  AND g.document_id IS NULL;

-- 3-c: 백필 안 된 행은 source_doc(filename)으로 doc_metadata 조회하여 보완
-- (doc_chunk_ids가 비어있거나 청크가 이미 삭제된 경우)
UPDATE qa_gen_results g
SET document_id = (
    SELECT m.document_id
    FROM doc_metadata m
    WHERE m.filename = g.source_doc
    ORDER BY m.created_at DESC
    LIMIT 1
)
WHERE g.document_id IS NULL
  AND g.source_doc IS NOT NULL;

-- 3-d: FK 연결
ALTER TABLE qa_gen_results
  ADD CONSTRAINT fk_qa_gen_doc
  FOREIGN KEY (document_id)
  REFERENCES doc_metadata(document_id)
  ON DELETE SET NULL;
```

**주의**: 동일 filename으로 여러 번 업로드된 경우 `ORDER BY created_at DESC LIMIT 1`로 최신 document_id를 취함. 이 경우 이전 업로드 버전과의 연결은 정확도가 떨어질 수 있음 — 허용 범위로 간주.

---

#### Step 4: `qa_eval_results` — generation_id 역방향 FK 추가

`qa_gen_results.linked_evaluation_id`가 이미 존재하므로 이를 역으로 뒤집어 백필.

```sql
-- 4-a: 컬럼 추가
ALTER TABLE qa_eval_results ADD COLUMN generation_id uuid;

-- 4-b: 기존 linked_evaluation_id 역방향으로 백필 (신뢰도 높음)
UPDATE qa_eval_results e
SET generation_id = g.id
FROM qa_gen_results g
WHERE g.linked_evaluation_id = e.id;

-- 4-c: FK 연결
ALTER TABLE qa_eval_results
  ADD CONSTRAINT fk_qa_eval_gen
  FOREIGN KEY (generation_id)
  REFERENCES qa_gen_results(id)
  ON DELETE SET NULL;
```

## 1-B. anchor_ids 제거 및 document_id 단일 관리 계획

### 현황

`anchor_ids`는 현재 두 가지 역할:

1. `/analyze-hierarchy` — LLM에 전달할 청크 텍스트 구성 (서버에서 완결, localStorage 불필요)
2. `/generate` — QA 생성 시 청크 조회 시작점 (프론트 localStorage → 백엔드 전달)

역할 1은 서버 내부에서 완결되므로 localStorage와 무관.
역할 2만 프론트 localStorage 의존 — `document_id`로 대체 가능.

### 현재 문제

- `anchor_ids`를 localStorage에 저장 → 브라우저/도메인 단위 저장이라 배포 환경에서 지속성 없음
- 다른 브라우저, 시크릿 모드, 다른 사용자 접속 시 anchor_ids 없음 → fallback 경로 사용
- `document_id`도 localStorage에 저장 → 동일 문제

### 마이그레이션 후 목표 흐름

```
현재:
  프론트 localStorage(anchor_ids, document_id)
    → POST /generate { anchor_ids, document_id, h1, h2, h3 }
    → get_doc_chunks_by_ids(anchor_ids)

마이그레이션 후:
  프론트 localStorage(document_id)  ← anchor_ids 제거
    → POST /generate { document_id, h1, h2, h3, samples }
    → sample_doc_chunks(document_id, n=samples*3)  ← 서버에서 균등 샘플링
    → doc_chunks (document_id FK)로 직접 조회
```

`document_id`는 doc_metadata FK로 유효성이 DB 레벨 보장 → 애플리케이션 레벨 검증 단순화.

장기적으로는 localStorage의 `document_id`도 서버 세션이나 DB 상태로 대체해 프론트 상태 의존도 최소화 가능.

### 코드 변경 범위 (Option B 완료 후)

| 파일 | 변경 내용 |
|------|-----------|
| `backend/api/generation_api.py` | `anchor_ids` 파라미터 제거, `sample_doc_chunks(document_id, n)` 직접 호출로 대체 |

## 2. 평가 파이프라인 병목 분석

### 2.1 구조 확인 결과

#### RAG Triad (Layer 2) — 이미 단일 호출

`_rag_worker`는 `evaluate_all_with_reasons()` **1회 호출**로 relevance + groundedness + context_relevance 3개 점수를 JSON으로 동시에 받아옴. 3회 개별 호출 구조가 아님 — 이미 최적화된 상태.

```python
# pipeline.py — _rag_worker
result = rag_evaluator.evaluate_all_with_reasons(question, answer, context)
# → 단일 LLM 호출, JSON 응답으로 3개 점수 반환
```

`rag_triad.py`의 개별 메서드 `evaluate_relevance()`, `evaluate_groundedness()`, `evaluate_context_relevance()`는 파이프라인에서 호출되지 않음 (dead code).

#### QA Quality (Layer 3) — 단일 호출, completeness 1개 지표

`_quality_worker`도 `evaluate_all()` **1회 호출**. completeness + coverage + missing_aspects 반환.
현재 단일 지표(completeness)만 평가 — 원래 설계(4개 지표)에서 의도적 축소.

### 2.2 실제 병목 원인

RAG/Quality 모두 이미 단일 호출 구조임에도 느린 이유:

| 원인 | 상세 |
|------|------|
| **workers 제한** | Anthropic provider 선택 시 `max_api_workers=2` — QA 20개 기준 사실상 순차 처리와 유사 |
| **groundedness CoT** | CoT + context 최대 8,000자 전달 → 응답 토큰 많아 레이턴시 증가 |
| **evaluator 모델 선택** | Claude 사용 시 TPM 50K 제한 |
| **QA 생성 (Claude Sonnet)** | workers=2 (429 빈발 방지로 하향) → 청크 10개 기준 ~125초 |

**provider별 workers 비교**

| Provider | Generation workers | Eval workers |
|----------|--------------------|--------------|
| Anthropic (Claude) | 2 | 2 |
| Google (Gemini) | 5 | 10 |
| OpenAI (GPT) | 5 | 8 |

### 2.3 개선 방향 (우선순위 순)

1. **evaluator 모델 Gemini 고정** — workers=10, RPM 1,000. 현재 기본값이 `gemini-2.5-flash`이므로 사용자가 Claude 선택 시에만 느려짐 — UI 안내 또는 강제 고정 검토.

2. **groundedness context 전달량 축소** — 현재 최대 8,000자. 줄이면 레이턴시 감소하지만 정확도 트레이드오프 있음.

3. **QA 생성 workers 상향 실험** — Claude Sonnet 2→3. 429 빈발 원인이 TPM이면 효과 제한적.

### 2.4 QA Quality specificity 추가 검토

| 지표 | 현재 | RAG Layer 커버 여부 | 추가 필요성 |
|------|------|---------------------|------------|
| factuality | 미측정 | groundedness가 커버 | 낮음 |
| completeness | ✅ 측정 중 | — | — |
| specificity | 미측정 | 미커버 | 중간 — 답변 구체성 공백 |
| conciseness | 미측정 | 미커버 | 낮음 |

specificity 추가 시 단일 호출 JSON에 필드 추가 방식으로 LLM 호출 횟수 증가 없이 확장 가능.

### 3.5 타입/상수 문제

- `any` 타입 사용 — `IntentTreemapTooltip` (line 320), `RadarTooltip` (line 344) 등
- **매직 넘버 산재**:
  - `0.85`, `0.7` (점수 임계값) → `const SCORE_THRESHOLDS = { high: 0.85, mid: 0.7 }`
  - `/10` (게이지 스케일 정규화) → 상수화 또는 데이터 레이어에서 정규화
- 소트 컬럼 문자열 리터럴 (`'id'`, `'intent'`, `'q'` 등) → enum 또는 `as const` 객체로 추출
- `QAPreviewItem` (lines 122-152): 28개 optional 필드, 레거시/신규 포맷 구분 타입 분리 필요

### 3.6 리팩토링 우선순위

**High — 유지보수 차단 수준**
1. 6개 서브 컴포넌트 분리 (`DashboardHeader`, `SummaryStatsCards`, `ChartsGrid`, `QATable`, `ExportMenu`, `DashboardStates`)
2. `buildChartData` / `buildChartDataFromHistory` 단일 함수 통합
3. `useMemo` 추가 (`filteredQA`, `sortedQA`, `evaluationData`)
4. 중복 스타일/배지 유틸 컴포넌트 추출

**Medium — 코드 품질**
1. 커스텀 훅으로 상태 분리 (`useQATableState`, `useHistoryManagement`)
2. `any` → 명시적 타입
3. 매직 넘버 상수화 (`SCORE_THRESHOLDS` 등)

**Low — 개선 사항**
1. 접근성 개선 (aria-label, 키보드 지원, SVG 대체 텍스트)
2. 차트 애니메이션 최적화 (ResizeObserver → IntersectionObserver)

### 3.7 작업 방향

파일 크기와 복잡도를 고려해 **단계적 분리** 권장:

1. **Phase 1**: 유틸 함수 분리 (`chartDataUtils.ts`, `scoreUtils.ts`) — 사이드이펙트 없음
2. **Phase 2**: 단순 표시 컴포넌트 추출 (`SummaryStatsCards`, `DashboardStates`) — props 단방향
3. **Phase 3**: 복잡 컴포넌트 추출 (`QATable`, `ChartsGrid`) — 상태 설계 필요
4. **Phase 4**: 커스텀 훅 도입 + `useMemo` 추가
