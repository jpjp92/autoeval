# Development Log: Prompt Optimization Review (2026-04-03)

본 문서는 `backend/generators/prompts.py`의 QA 생성 프롬프트 품질을 점검하고, 향후 고도화 방향을 정리한 리포트입니다.

## 1. 개요

AutoEval 프로젝트의 핵심인 QA 생성 엔진의 품질을 제어하는 프롬프트 시스템을 검토했습니다. 설계상 강점과 실무 적용 시 발생 가능한 잠재적 리스크를 식별하고 이를 해결하기 위한 개선안을 도출했습니다.

## 2. 주요 설계 강점 (Keep)

- **QA 품질 제어 축 명확화**: Groundedness, Single-scope, Depth, Completeness 등의 지표가 RAG 평가와 잘 연결되어 있음.
- **Intent Taxonomy**: 실제 downstream task와 정렬이 잘 된 6가지 의도 유형 정의.
- **Domain Adaptive Engine**: 문서 도메인별 가중치 및 샘플 제공 방식.
- **Trivial QA 방지**: 단순 날짜, 번호, 명칭 등 학습 가치가 낮은 데이터 생성을 엄격히 제한.

## 3. 핵심 개선 과제 (Improve)

### 3.1 [Priority 1] 사전 검증 및 Context Screening 유연화

- **문제**: 현재 "완성된 서술 문서 필수" 기준이 표/불렛 데이터를 불필요하게 스크리닝할 가능성이 높음.
- **해결**: 문법적 완결성보다 **"정보 간 상관관계(Entity-Value-Condition)"** 유무를 기준으로 스크리닝 완화.
- **실행**: 수정 전, 현재 프롬프트가 표/불렛 데이터를 실제 `{"qa_list": []}`로 반환하는지 테스트 코드로 먼저 확인.

### 3.2 Reasoning 필드: '해석' -> '증거 추적(Evidence Trace)'

- **구조적 차이**:
  - **이전 (Interpretation)**: 질문의 배경이나 주관적 논리를 설명. 할루시네이션 유발 포인트.
  - **이후 (Evidence Trace)**: 질문이 컨텍스트의 **어느 구절(Mapping)**에서 왔고, 이를 **어떻게 변형(Formulation)**했는지 기술.
- **효과**: 생성 모델이 답변(A)을 쓸 때 원문의 팩트에 더 집중하게 되어, 결과적으로 성능 평가 점수(Groundedness)가 향상됨.

### 3.3 Depth vs Single-scope 원칙 충돌 해결

- **문제**: "2개 이상 연결(Depth)" 하려다 "차원을 섞는(Mixed Dimension)" 실수 발생.
- **해결**: Depth의 범위를 **"동일한 질문 차원 내에서의 복수 요소 결합"**으로 한정.

### 3.4 다국어(EN/KO) 정합성 및 Diversity Rules

- **Diversity**: 생성 전 intent 분포 계획 수립 단계를 프롬프트 상단에 배치.
- **Multilingual**: 한국어의 '답변 완전성(Answer Completeness)' 규칙을 영어 프롬프트에도 이식.

## 3.5 핵심 개선 과제 및 충돌 지침 분석

### 1. Context Screening: Ingestion과 Generation의 불일치
*   **현상**: Ingestion 단계에서 PDF/Docx의 표를 **Markdown Table** 형식으로 충실히 추출하고 있음.
*   **충돌**: 현재 V1 프롬프트의 "완성된 서술 문장(주어+서술어) 필수" 지침이 Markdown 형식을 '단순 항목 나열'로 오인하여 우수한 정보원을 차단함.
*   **해결**: "문장 형태" 대신 "의미 있는 정보 관계(Entity-Value-Condition)"가 성립하면 수용하도록 지침 완화.

### 2. Depth vs Single-scope: 차원 혼합 방지
*   **현상**: "2개 이상의 요소 연결(Depth)" 지침이 "질문 차원 하나만 다룰 것(Single-scope)" 지침과 충돌하여 모델이 질문을 복잡하게 꼬는 경향 발생.
*   **해결**: Depth를 "동일 질문 차원 내의 복수 항목/조건 결합"으로 재정의하여 의미 있는 정보 밀도는 높이되 차원 혼합은 금지.

### 3. Reasoning: 해석에서 증거 매핑으로
*   **현상**: 기존의 "해석" 요구 사항은 모델의 주관적 개입을 유도하여 할루시네이션(Grounding 실패)의 원인이 됨.
*   **해결**: Reasoning을 **"컨텍스트 문장/항목 매핑 및 구성 방식 서술(Evidence Trace)"**로 변경하여 근거성을 강제함.
