# 2026-04-01: anchor_ids 제거 및 document_id 단일 관리 전환

---

## 1. 작업 배경

Option B DB 마이그레이션(2026-03-31 완료) 이후 `document_id`가 FK로 정식 편입됐으나,
청크 조회 경로는 여전히 `anchor_ids`(localStorage → 백엔드 전달)가 1순위였고 `document_id`는 fallback 역할에 그쳤다.

**기존 문제**
- `anchor_ids`를 localStorage에 저장 → 브라우저/사용자/도메인 단위로 지속성 없음
- 시크릿 모드, 다른 브라우저 접속 시 `anchor_ids` 없음 → fallback 경로로만 동작
- 인제스션 완료 시점에 `document_id`가 DB에 정착돼 있으므로 `anchor_ids`가 불필요한 중간 상태였음

## 2. 변경 내용

### 2.1 Backend

#### `backend/api/generation_api.py`

- `GenerateRequest`에서 `anchor_ids: Optional[list]` 필드 제거
- 청크 조회 우선순위 변경
  - **이전**: `anchor_ids` 있으면 `get_doc_chunks_by_ids(anchor_ids)` → coverage gap 보충
  - **이후**: `document_id + filename` 있으면 `get_doc_chunks_sampled(document_id, n)` RPC 균등 샘플링 → h1/h2/h3 후처리 필터링
  - fallback: `get_doc_chunks_by_filter(document_id, ...)`
- 태깅 완료율 검증 조건에서 `anchor_ids` 의존 제거 (모든 청크 대상으로 일관 적용)
- `analyze_domain()` 호출 시 `anchor_ids=anchor_ids` 인자 제거

#### `backend/api/ingestion_api.py`

- `HierarchyAnalysisResponse` 모델에서 `anchor_ids: Optional[List[str]]` 필드 제거
- `analyze-hierarchy` 핸들러에서 `anchor_ids = [c["id"] for c in anchor_chunks]` 계산 코드 제거
- 응답 구성에서 `anchor_ids=anchor_ids` 제거
- `document_id`만 응답으로 반환 (기존 동작 유지)

#### `backend/generators/domain_profiler.py`

- `analyze_domain()` 시그니처에서 `anchor_ids: Optional[list]` 파라미터 제거
- 내부 조회를 `get_doc_chunks_by_filter()` 단일 경로로 통일
- 불필요한 `get_doc_chunks_by_ids` import 제거

### 2.2 Frontend

#### `frontend/src/components/standardization/DataStandardizationPanel.tsx`

- `AnalysisResult` 인터페이스에서 `anchor_ids?: string[]` 필드 제거
- `anchorKey()`, `saveAnchorIds()` 헬퍼 함수 제거

```
  └─ 프론트: { filename, hierarchy_h1/h2/h3, document_id } 전달
  └─ sample_doc_chunks(document_id, n=samples*3) 균등 샘플링
  └─ h1/h2/h3 후처리 필터링
  └─ QA 생성 → qa_gen_results 저장 (document_id FK 포함)
```

### 5.2 `update_chunk_metadata` Broken pipe / Cloudflare 400 반복 (이슈 2 → 3차 수정)

#### 1차 수정 (chunked gather)

**원인**: `process_batch` 내부에서 update 태스크 전체를 `asyncio.gather` 로 동시 발사.
5배치 병렬 실행 × 배치당 5청크 = 최대 25개 `asyncio.to_thread` 동시 실행 →
supabase-py sync 클라이언트 HTTP/2 커넥션 풀 경합 → Broken pipe / `deque mutated during iteration`

**수정**: `gather(*update_tasks)` → 3개씩 chunked gather (동시 커넥션 25 → 15)

#### 2차 수정 (전역 세마포어)

**원인**: chunked gather 이후에도 5배치 동시 실행 × 3 = 최대 15개 동시 → 여전히 `ConnectionTerminated error_code:9`

**수정 내용** (`backend/db/doc_chunk_repo.py`)

```python
_DB_WRITE_CONCURRENCY = 2
_db_write_sem: asyncio.Semaphore | None = None

def _write_sem() -> asyncio.Semaphore:
    global _db_write_sem
    if _db_write_sem is None:
        _db_write_sem = asyncio.Semaphore(_DB_WRITE_CONCURRENCY)
    return _db_write_sem
```

- `update_chunk_metadata` 내부에 `async with _write_sem()` 적용 → 전체 동시 DB 쓰기 최대 2개 제한
- retry 2회 → 3회, 고정 대기 → 지수 백오프(0.5→1.0→2.0→4.0 초)
- `ingestion_api.py` LLM 배치 세마포어 10 → 5, chunked gather 제거(세마포어가 조절하므로 단순 gather)

#### 3차 수정 (patch_chunk_hierarchy RPC)

**신규 오류**: `{'message': 'JSON could not be generated', 'code': 400}` — Cloudflare가 요청 차단

**원인**: `update_chunk_metadata` 가 기존 metadata 전체를 재직렬화해 전송. 페이로드가 커서 Cloudflare WAF/rate-limit에 걸림.

**수정 내용**

- Supabase에 `patch_chunk_hierarchy` RPC 함수 추가 — DB side에서 `jsonb ||` 로 3개 필드만 merge

```sql
CREATE OR REPLACE FUNCTION patch_chunk_hierarchy(
  p_chunk_id UUID, p_h1 TEXT, p_h2 TEXT, p_h3 TEXT
) RETURNS VOID AS $$
BEGIN
  UPDATE doc_chunks
  SET metadata = metadata || jsonb_build_object(
    'hierarchy_h1', p_h1, 'hierarchy_h2', p_h2, 'hierarchy_h3', p_h3
  )
  WHERE id = p_chunk_id;
END;
$$ LANGUAGE plpgsql;
```

- `backend/db/doc_chunk_repo.py` — `patch_chunk_hierarchy()` 함수 추가 (세마포어·retry 동일 적용)
- `backend/api/ingestion_api.py` — `process_batch` 에서 `update_chunk_metadata` → `patch_chunk_hierarchy` 호출로 교체
- `backend/config/supabase_client.py` — `patch_chunk_hierarchy` export 추가

**결과**: 페이로드 대폭 축소 (메타데이터 전체 → 파라미터 4개), Cloudflare 400 방지

### 6.2 ITO 브랜치 분기

ITO 환경은 QA 생성 없이 실데이터 QA를 직접 평가에 투입하는 별도 흐름이 필요하다.

**브랜치 전략**
```
main        ← 상용 솔루션 트랙 (QA 생성 + 평가 풀 파이프라인)
feature/ito ← ITO 운용 트랙 (외부 QA 입력 + 평가 전용)
```

**ITO 트랙 주요 변경 범위**
- 외부 QA 입력 전용 엔드포인트 (`POST /api/evaluation/evaluate-external`) 신규 개발
  - 요청 body: `{ qa_list: [{q, a, context, intent}], metadata: {} }`
  - 기존 4-layer 평가 파이프라인 재사용 (syntax → stats → rag → quality)
  - `qa_gen_results` 저장 생략, `qa_eval_results`에 직접 저장
- 프론트엔드: QA 파일 업로드(CSV/JSON) UI 추가 — 생성 패널 우회
