## 1. 작업 배경

Option B DB 마이그레이션(2026-03-31 완료) 이후 `document_id`가 FK로 정식 편입됐으나,
청크 조회 경로는 여전히 `anchor_ids`(localStorage → 백엔드 전달)가 1순위였고 `document_id`는 fallback 역할에 그쳤다.

**기존 문제**
- `anchor_ids`를 localStorage에 저장 → 브라우저/사용자/도메인 단위로 지속성 없음
- 시크릿 모드, 다른 브라우저 접속 시 `anchor_ids` 없음 → fallback 경로로만 동작
- 인제스션 완료 시점에 `document_id`가 DB에 정착돼 있으므로 `anchor_ids`가 불필요한 중간 상태였음

### 5.2 `update_chunk_metadata` Broken pipe / Cloudflare 400 반복 (이슈 2 → 3차 수정)

#### 1차 수정 (chunked gather)

**원인**: `process_batch` 내부에서 update 태스크 전체를 `asyncio.gather` 로 동시 발사.
5배치 병렬 실행 × 배치당 5청크 = 최대 25개 `asyncio.to_thread` 동시 실행 →
supabase-py sync 클라이언트 HTTP/2 커넥션 풀 경합 → Broken pipe / `deque mutated during iteration`

**수정**: `gather(*update_tasks)` → 3개씩 chunked gather (동시 커넥션 25 → 15)

#### 2차 수정 (전역 세마포어)

**원인**: chunked gather 이후에도 5배치 동시 실행 × 3 = 최대 15개 동시 → 여전히 `ConnectionTerminated error_code:9`

**수정 내용** (`backend/db/doc_chunk_repo.py`)

```python
_DB_WRITE_CONCURRENCY = 2
_db_write_sem: asyncio.Semaphore | None = None

def _write_sem() -> asyncio.Semaphore:
    global _db_write_sem
    if _db_write_sem is None:
        _db_write_sem = asyncio.Semaphore(_DB_WRITE_CONCURRENCY)
    return _db_write_sem
```

- `update_chunk_metadata` 내부에 `async with _write_sem()` 적용 → 전체 동시 DB 쓰기 최대 2개 제한
- retry 2회 → 3회, 고정 대기 → 지수 백오프(0.5→1.0→2.0→4.0 초)
- `ingestion_api.py` LLM 배치 세마포어 10 → 5, chunked gather 제거(세마포어가 조절하므로 단순 gather)

#### 3차 수정 (patch_chunk_hierarchy RPC)

**신규 오류**: `{'message': 'JSON could not be generated', 'code': 400}` — Cloudflare가 요청 차단

**원인**: `update_chunk_metadata` 가 기존 metadata 전체를 재직렬화해 전송. 페이로드가 커서 Cloudflare WAF/rate-limit에 걸림.

**수정 내용**

- Supabase에 `patch_chunk_hierarchy` RPC 함수 추가 — DB side에서 `jsonb ||` 로 3개 필드만 merge

```sql
CREATE OR REPLACE FUNCTION patch_chunk_hierarchy(
  p_chunk_id UUID, p_h1 TEXT, p_h2 TEXT, p_h3 TEXT
) RETURNS VOID AS $$
BEGIN
  UPDATE doc_chunks
  SET metadata = metadata || jsonb_build_object(
    'hierarchy_h1', p_h1, 'hierarchy_h2', p_h2, 'hierarchy_h3', p_h3
  )
  WHERE id = p_chunk_id;
END;
$$ LANGUAGE plpgsql;
```

- `backend/db/doc_chunk_repo.py` — `patch_chunk_hierarchy()` 함수 추가 (세마포어·retry 동일 적용)
- `backend/api/ingestion_api.py` — `process_batch` 에서 `update_chunk_metadata` → `patch_chunk_hierarchy` 호출로 교체
- `backend/config/supabase_client.py` — `patch_chunk_hierarchy` export 추가

**결과**: 페이로드 대폭 축소 (메타데이터 전체 → 파라미터 4개), Cloudflare 400 방지

## 7. Supabase VIEW 적용 완료

Supabase Table Editor에서 즉시 조회 가능한 통계/점검 뷰. **전체 적용 완료.**
supabase-py에서도 동일하게 `sb.table("v_xxx").select("*")` 로 사용 가능.

> **적용 완료**: Supabase SQL Editor에서 실행 → Table Editor `Views` 탭에서 확인 가능

---

### 7.1 파이프라인 완성도 (`v_pipeline_status`)

문서별 인제스션 → hierarchy 분석 → QA 생성 → 평가 완료 여부를 한 줄로 확인.

```sql
CREATE OR REPLACE VIEW v_pipeline_status AS
SELECT
    dm.filename,
    dm.document_id,
    dm.created_at::date                                        AS ingested_date,
    COUNT(dc.id)                                               AS total_chunks,
    COUNT(dc.id) FILTER (
        WHERE dc.metadata->>'hierarchy_h1' IS NOT NULL
          AND dc.metadata->>'hierarchy_h1' != '__admin__'
    )                                                          AS tagged_chunks,
    ROUND(
        COUNT(dc.id) FILTER (
            WHERE dc.metadata->>'hierarchy_h1' IS NOT NULL
              AND dc.metadata->>'hierarchy_h1' != '__admin__'
        )::numeric / NULLIF(COUNT(dc.id), 0) * 100, 1
    )                                                          AS tagging_rate_pct,
    CASE WHEN dm.domain_profile IS NOT NULL THEN 'O' ELSE 'X' END AS domain_analyzed,
    CASE WHEN dm.h2_h3_master   IS NOT NULL THEN 'O' ELSE 'X' END AS hierarchy_built,
    COUNT(DISTINCT qg.id)                                      AS qa_gen_count,
    COUNT(DISTINCT qe.id)                                      AS qa_eval_count
FROM doc_metadata dm
LEFT JOIN doc_chunks dc
    ON dc.metadata->>'document_id' = dm.document_id
LEFT JOIN qa_gen_results qg
    ON qg.source_doc = dm.filename
    AND qg.doc_chunk_ids && ARRAY(
        SELECT id FROM doc_chunks
        WHERE metadata->>'document_id' = dm.document_id
    )
LEFT JOIN qa_eval_results qe
    ON qe.id = qg.linked_evaluation_id
GROUP BY
    dm.document_id, dm.filename, dm.created_at,
    dm.domain_profile, dm.h2_h3_master
ORDER BY dm.created_at DESC;
```

### 7.2 hierarchy 태깅 분포 (`v_hierarchy_coverage`)

문서별 H1/H2/H3 조합과 청크 수 분포 확인. 태깅 누락 구간 파악용.

```sql
CREATE OR REPLACE VIEW v_hierarchy_coverage AS
SELECT
    metadata->>'filename'      AS filename,
    metadata->>'document_id'   AS document_id,
    metadata->>'hierarchy_h1'  AS h1,
    metadata->>'hierarchy_h2'  AS h2,
    metadata->>'hierarchy_h3'  AS h3,
    COUNT(*)                   AS chunk_count
FROM doc_chunks
WHERE metadata->>'hierarchy_h1' IS NOT NULL
  AND metadata->>'hierarchy_h1' != '__admin__'
GROUP BY
    metadata->>'filename',
    metadata->>'document_id',
    metadata->>'hierarchy_h1',
    metadata->>'hierarchy_h2',
    metadata->>'hierarchy_h3'
ORDER BY filename, h1, h2, h3;
```

### 7.3 평가 결과 요약 (`v_eval_summary`)

평가 점수·등급·레이어별 세부 점수를 평탄화해서 조회.

```sql
DROP VIEW IF EXISTS v_eval_summary;

CREATE VIEW v_eval_summary AS
SELECT
    qg.source_doc                                              AS filename,
    qe.id                                                      AS eval_id,
    qe.job_id,
    qe.total_qa,
    qe.valid_qa,
    qe.final_score,
    qe.final_grade,
    qe.created_at::date                                        AS eval_date,
    (qe.scores->'syntax' ->>'pass_rate')::numeric              AS syntax_pass_rate,
    (qe.scores->'rag'    ->>'avg_score')::numeric              AS rag_avg_score,
    (qe.scores->'rag'    ->>'avg_relevance')::numeric          AS rag_relevance,
    (qe.scores->'rag'    ->>'avg_groundedness')::numeric       AS rag_groundedness,
    (qe.scores->'rag'    ->>'avg_context_relevance')::numeric  AS rag_context_relevance,
    (qe.scores->'quality'->>'avg_quality')::numeric            AS quality_avg,
    (qe.scores->'quality'->>'avg_completeness')::numeric       AS quality_completeness,
    (qe.scores->'stats'->'diversity'->>'score')::numeric       AS stats_diversity,
    (qe.scores->'stats'->'duplication_rate'->>'score')::numeric AS stats_dedup
FROM qa_eval_results qe
JOIN qa_gen_results qg ON qg.linked_evaluation_id = qe.id
ORDER BY qe.created_at DESC;
```

### 7.4 DB 정합성 점검 (`v_db_health`)

이상 상태(null 값·고아 레코드·깨진 참조)를 항목별로 감지. 정기 점검용.

```sql
DROP VIEW IF EXISTS v_db_health;

CREATE VIEW v_db_health AS

SELECT
    'doc_metadata: domain/master null' AS check_item,
    COUNT(*)                           AS count,
    CASE WHEN COUNT(*) = 0 THEN 'OK' ELSE 'WARN' END AS status,
    CASE WHEN COUNT(*) = 0 THEN NULL
         ELSE string_agg(filename || ' (' || document_id::text || ')', ', ')
    END AS detail
FROM doc_metadata
WHERE domain_profile IS NULL OR h2_h3_master IS NULL

UNION ALL

SELECT
    'doc_chunks: hierarchy_h1 null',
    COUNT(*),
    CASE WHEN COUNT(*) = 0 THEN 'OK' ELSE 'WARN' END,
    CASE WHEN COUNT(*) = 0 THEN NULL
         ELSE string_agg(DISTINCT
             (metadata->>'filename') || ' / doc_id=' || LEFT(metadata->>'document_id', 8) || '...'
         , ' | ')
    END
FROM doc_chunks
WHERE metadata->>'hierarchy_h1' IS NULL

UNION ALL

SELECT
    'doc_chunks: orphan (no doc_metadata)',
    COUNT(DISTINCT metadata->>'document_id'),
    CASE WHEN COUNT(DISTINCT metadata->>'document_id') = 0 THEN 'OK' ELSE 'WARN' END,
    CASE WHEN COUNT(DISTINCT metadata->>'document_id') = 0 THEN NULL
         ELSE string_agg(DISTINCT
             (metadata->>'filename') || ' / doc_id=' || LEFT(metadata->>'document_id', 8) || '...'
         , ' | ')
    END
FROM doc_chunks dc
WHERE NOT EXISTS (
    SELECT 1 FROM doc_metadata dm
    WHERE dm.document_id::text = dc.metadata->>'document_id'
)

UNION ALL

SELECT
    'qa_gen_results: broken chunk refs',
    COUNT(*),
    CASE WHEN COUNT(*) = 0 THEN 'OK' ELSE 'WARN' END,
    CASE WHEN COUNT(*) = 0 THEN NULL
         ELSE string_agg(source_doc || ' (gen_id=' || LEFT(id::text, 8) || '...)', ' | ')
    END
FROM qa_gen_results qg
WHERE NOT EXISTS (
    SELECT 1 FROM doc_chunks dc
    WHERE dc.id = ANY(qg.doc_chunk_ids)
    LIMIT 1
);
```

### 7.5 RLS 참고사항

- VIEW는 RLS 정책을 직접 설정 불가
- Supabase 기본값은 `SECURITY DEFINER` → 하위 테이블 RLS 우회
- 추후 팀 공유 또는 외부 접근 시 `security_invoker = true` 옵션 추가 필요

```sql
-- 공유 전환 시 적용
CREATE OR REPLACE VIEW v_pipeline_status
WITH (security_invoker = true)
AS SELECT ...;
```

현재(내부 단일 사용자)는 기본값 유지.

## 10. 평가 대시보드 Hierarchy 표시 추가 (2026-04-01)

### 10.1 배경

QA 생성 시 사용자가 선택한 H1/H2/H3 필터가 `qa_gen_results.metadata`에 저장되지 않아,
평가 결과 화면에서 어떤 계층 범위를 대상으로 생성했는지 확인 불가했음.

### 10.2 변경 내용

#### Backend — `backend/api/generation_api.py`

`save_qa_generation_to_supabase` 호출 시 `metadata` 딕셔너리에 필드 추가:

```python
"hierarchy_h1": h1 or "",
"hierarchy_h2": h2 or "",
"hierarchy_h3": h3 or "",
```

- 수정 이전 레코드: `hierarchy_h*` 필드 없음 (구형 데이터)
- 수정 이후 생성 job부터만 insert — 하위 호환 필요

#### Frontend — `frontend/src/components/evaluation/QAEvaluationDashboard.tsx`

| 위치 | 변경 내용 |
|------|-----------|
| `EvalReport.metadata` 타입 | `hierarchy_h1?`, `hierarchy_h2?`, `hierarchy_h3?` 필드 추가 |
| `HistoryItem.metadata` 타입 | 동일 |
| 리포트 헤더 칩 row | Hierarchy breadcrumb 칩 추가 — `LayoutGrid` 아이콘 + `H1 › H2 › H3` 형태, 빈 값이면 칩 미렌더링 |

**Hierarchy 칩 표시 조건**: `hierarchy_h1/h2/h3` 중 하나라도 값이 있으면 표시. 구형 레코드는 모두 빈 문자열이므로 칩 숨김 처리됨 — 별도 마이그레이션 불필요.

### 11.3 UI 개선 — 데이터 통계 점수 표기 / 헤더 다크모드 토글 이동 (2026-04-01)

#### 데이터 통계 레이더 차트 점수 표기 통일

**대상**: `frontend/src/components/evaluation/QAEvaluationDashboard.tsx`

`10.0 / 10` 처럼 분자는 소수, 분모는 정수로 혼용되어 어색했던 표기를 소수 1자리로 통일.

| 위치 | 변경 전 | 변경 후 |
|------|---------|---------|
| `RadarTooltip` hover | `X.X / 10` | `X.X / 10.0` |
| `MetricRadialGauge` hover tooltip | `X.X / 10` | `X.X / 10.0` |
| `MetricRadialGauge` 게이지 내부 숫자 | `toFixed(1)` (그대로) | `toFixed(1)` (그대로) |

결과: `9.1 / 10.0`, `10.0 / 10.0`, `0.0 / 10.0` 형태로 일관됨.

> 중복성 0.0 표시는 표시 버그가 아님 — near-duplicate 40% → `score = max(0, 10 - 40/2) = 0` 정상 계산값.
