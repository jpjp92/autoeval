# 2026-03-30: QA 품질 평가 및 통계 분석 고도화 산출물

이 문서는 QA 품질 평가 엔진의 `완전성(Completeness)` 지표 개선 사항과 `데이터셋 통계(Layer 1-B)`의 산출 방법론 및 지표 정의를 상세히 정리합니다.

---

## 1. 품질 평가: 완전성 (QA Quality: Completeness)

LLM 답변이 질문의 모든 요구사항을 충실히 반영했는지 측정하는 지표입니다.

### 1.1 측정 원리: 질문 분해 (Decomposition)
- **원리**: 복합적인 질문을 원자 단위의 서브 질문(Atomic sub-questions)으로 분해하여 평가의 객관성을 확보함.
- **예시**: "A와 B의 차이점은 무엇이며, 각각의 장점은?"
  - 서브 질문 1: A와 B의 차이점이 설명되었는가?
  - 서브 질문 2: A의 장점이 언급되었는가?
  - 서브 질문 3: B의 장점이 언급되었는가?

### 1.2 채점 기준 및 커버리지 (Coverage)
- **커버리지 계산**: `답변된 서브 질문 수 / 전체 서브 질문 수` 비율로 산출.
- **점수 산출 (Scaling)**: 
  - 커버리지 100%: 10점 (완벽)
  - 커버리지 80% 이상: 8-9점 (사소한 누락)
  - 커버리지 50% 미만: 5점 이하 (핵심 내용 누락)
- **누락 요소 탐지**: 답변에서 다루지 못한 측면을 `missing_aspects`로 추출하여 사용자에게 구체적인 피드백 제공.

## 2. 데이터셋 통계: Layer 1-B 산출 방법론

데이터셋 자체의 건전성과 다양성을 물리적으로 측정하는 지표입니다.

### 2.1 다양성 (Diversity)
*   **정의**: 데이터셋이 얼마나 다양한 주제와 어휘를 포함하고 있는가.
*   **산출 방식**:
    - **인텐트 엔트로피 (60%)**: Shannon Entropy($-\sum p_i \log_2 p_i$)를 통해 인텐트 분포의 균형도를 측정. (균등할수록 고득점)
    - **어휘 다양도 (TTR, 40%)**: 고유 토큰 수 대비 전체 토큰 수 비율($Unique / Total$). (중복 단어가 적을수록 고득점)
*   **목표**: 특정 유형의 질문에만 매몰되지 않은 균형 잡힌 데이터셋 확보.

### 2.2 중복도 (Duplication)
*   **정의**: 서로 다른 질문 쌍 간의 텍스트 유사성 정도.
*   **산출 방식**: `SequenceMatcher` 알고리즘을 사용해 질문 텍스트 간 유사도가 70% 이상인 쌍(Near-duplicate)을 탐지.
*   **감점 로직**: `10 - (중복률 / 2)`. 중복률이 20%를 넘어가면 0점 처리하여 데이터 중복을 엄격히 관리.

### 2.3 편향도 (Skewness)
*   **정의**: 특정 소스 문서(docId)에 질문이 과도하게 집중되어 있는 정도.
*   **산출 방식**: 전체 데이터셋 중 단일 문서가 차지하는 최대 비중(`doc_max_ratio`)을 기준으로 구간 채점.
*   **기준**: 
    - 30% 이하: 10점 (이상적 분포)
    - 50% 이하: 8점
    - 80% 이하: 5점
    - 80% 초과: 2점

### 2.4 충족성 (Sufficiency)
*   **정의**: 데이터 분석 및 학습에 필요한 필수 정보가 물리적으로 갖춰졌는가.
*   **산출 방식**:
    - **필드 채움률**: 질문, 답변, 컨텍스트, 문서ID, 인텐트 등 5개 필수 필드의 기입 여부 전수 조사.
    - **물리적 볼륨**: 질문/답변의 평균 길이를 통해 텍스트가 분석하기에 충분한 정보량을 담고 있는지 보조적으로 판단.

## 3. UI/UX 및 데이터 매핑 원칙

- **용어 통일**: 품질 평가의 **'완전성'**과 통계의 **'충족성'**을 엄격히 구분하여 사용자 혼선 방지.
- **감사 리포트 스타일**: 모든 분석 결과는 [헤더 바 + 액센트 라인 + 컨텐츠] 구조의 리포트 형식으로 시각화.
- **데이터 투명성**: 정량적 점수(Metrics) 뒤에 항상 구체적인 근거 데이터(Stats/Reason)를 배치하여 분석 결과의 신뢰도 확보.

## 4. 평가 상태 판정 로직 개선 (2026-03-30)

### 4.1 개요
사례 #4와 같이 점수는 높지만(0.7 이상) 실제로는 '근거 오류' 등의 결함이 감지된 경우, 기존의 단순 점수 기반 '성공' 판정을 지양하고 **'보류(Hold)'**로 분류하여 사용자 검토를 유도하도록 로직을 개선함.

### 4.2 하이브리드 판정 로직 (Hybrid Score-Error Logic)
- **실패 (Fail)**: 품질 및 Triad 평균 점수가 모두 0.7점 미만인 경우.
- **보류 (Hold)**: 
  - 점수가 0.7점 미만인 지표가 하나라도 있는 경우.
  - **예외 처리**: 점수가 모두 0.7점 이상이나, 백엔드 판정 결과 결함(`pass: false` 또는 `failure_types` 존재)이 발견된 경우.
- **성공 (Success)**: 모든 점수가 0.7점 이상이며, 어떠한 결함도 감지되지 않은 경우.

### 4.3 주요 수정 및 수정 내역
- **대시보드 UI (`getQAStatus`)**: 백엔드 전달 데이터(`pass`, `failure_types`)를 최우선으로 고려하는 판정 로직 적용 및 Null Guard 강화.
- **엑셀(XLSX) 내보내기**: 대시보드 UI와 동일한 상태 판정 로직으로 동기화하여 리포트 일관성 확보.
- **HTML 리포트**: 
  - 내장 스크립트(`qaStatus`)를 고도화하여 동일 로직 적용.
  - 구형 브라우저 호환성을 위해 `??` 연산자 제거 및 표준 JavaScript 문법(`indexOf`, `function`) 사용.
  - 루프 내 정의되지 않은 변수 참조(`ReferenceError`) 수정 완료.

## 5. 개별 지표 시각화 및 리포트 고도화 (2026-03-30)

평가 결과의 투명성을 높이기 위해 단순 평균 점수 노출을 넘어 각 지표별 세부 점수와 근거를 통합 시각화하도록 개선함.

### 5.1 QA 상세 보기 UI 개선 (3컬럼 레이아웃)
- **레이아웃 리팩토링**: `[지표명] [점수] [사유]`의 3개 열 구조를 도입하여 정보의 위계 정립.
  - **지표명 (Label)**: 관련성, 근거성, 맥락성, 완전성 등 명시적 레이블링.
  - **점수 (Score)**: 소수점 3자리까지 표시하며 모노스페이스 폰트 적용.
  - **사유 (Reasoning)**: 점수에 대한 LLM의 판단 근거를 우측에 배치하여 가독성 확보.
- **점수대별 가시성(Color Logic)**:
  - `Emerald (≥0.85)`: 우수 (Pass)
  - `Amber (0.70 ~ 0.85)`: 보통 (Attention)
  - `Rose (<0.70)`: 주의 (Fail/Hold)

### 5.2 HTML 내보내기 리포트 동기화
- **오프라인 리포트 고도화**: 대시보드 UI와 동일한 3컬럼 레이아웃을 오프라인 HTML 파일에서도 그대로 구현.
- **데이터 무결성 확보**: 리포트 생성 시 단순 `avg_score` 뿐만 아니라 개별 항목 점수를 전수 매핑하여 전달하도록 `exportUtils.ts` 로직 보강.
- **하위 호환성**: 개별 점수 필드가 없는 구형 데이터 로드 시 점수 열에 `-` 실장하여 안정적인 렌더링 보장.

### 5.3 백엔드 데이터 필드 보강
- **`pipeline.py` / `evaluation_api.py`**:
  - `qa_preview` 배열 및 상세 조회 API(`_build_export_detail`) 응답에 `relevance`, `groundedness`, `context_relevance`, `completeness` 점수 필드를 추가하여 프론트엔드와 데이터 모델 동기화 완료.
