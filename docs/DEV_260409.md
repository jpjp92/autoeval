## FIX-B: `extract_text_by_page()` — `_KOR_CONT` y_gap 가드

### 문제

`_KOR_CONT`/`_KOR_TAIL` 병합 보정 로직이 heading + 본문 첫 문장을 잘못 결합.

```
# 현재 출력 (잘못된 병합)
"1.2.1. 등장배경추론형 AI 데이터는 AI 발전에서 지식 기반 시스템..."

# 기대 출력 (정상 분리)
"1.2.1. 등장배경"
"추론형 AI 데이터는 AI 발전에서 지식 기반 시스템..."
```

**오발동 조건**:
- `prev_text` = `"1.2.1. 등장배경"` → 끝이 한글("경") → `_KOR_TAIL` 매칭
- `cur_text`  = `"추론형 AI 데이터는..."` → 3자 한글+공백 → `_KOR_CONT` 매칭
- → 어절 절단으로 오인, 병합됨

**y_gap 분포 (실측)**:

| 케이스 | y_gap | 판정 |
|--------|-------|------|
| 정상 어절 절단 | 8~10px | 병합 허용 |
| heading + 본문 | 17~35px | **병합 차단** |
| heading 2줄 분할 (예: "제1장" + "총론") | 21px | font_diff=0이므로 통과 |

### 수정 (`backend/ingestion/parsers.py:688`)

```python
# 변경 전
is_kor = (
    _KOR_CONT.match(stripped)
    and _KOR_TAIL.search(prev_text.rstrip())
)

# 변경 후 — y_gap > 15 AND font_diff > 1.5이면 별개 단락으로 간주
is_kor = (
    _KOR_CONT.match(stripped)
    and _KOR_TAIL.search(prev_text.rstrip())
)
if is_kor:
    prev_y1   = prev_block.get("bbox", [0, 0, 0, 0])[3]
    y_gap_val = y_pos - prev_y1
    prev_font = prev_block.get("font_size", 10.0)
    cur_font  = block_dict.get("font_size", 10.0)
    font_diff = abs(prev_font - cur_font)
    if y_gap_val > 15.0 and font_diff > 1.5:
        is_kor = False  # heading+본문 오병합 방지
```

**조건 설계 근거**:
- `y_gap > 15` 단독: heading 2줄 분할(y_gap=21, font_diff=0)을 차단해 회귀 발생
- `font_diff > 1.5` 단독: 같은 font heading+본문을 차단 못함
- **두 조건 AND**: heading(font 14)+본문(font 11)은 gap=30, diff=3 → 차단 / heading 2줄은 diff=0 → 허용

### anchor 샘플링 — 실제 Supabase 함수 확인 및 NTILE 검토 결과

#### 실제 Supabase 함수 (stride 방식)

세션 초반 `setup_vector_db.sql`의 `ORDER BY random()`이 현재 구현으로 오인했으나, **실제 Supabase에 배포된 함수는 이미 등간격 stride 방식**임을 확인.

- `document_id` 미지정 시 `created_at DESC`로 최신 버전 자동 고정 — 재업로드 버전 혼재 방지
- `step_size = GREATEST(total_count / p_n, 1)` 등간격 stride → 완전 결정적 샘플링
- `chunk_index::int` 기준 정렬 — `pipeline.py`에서 `batch_start + idx`로 전역 순번 할당

#### NTILE 대안 검토 결과 — 불채택 (2026-04-09)

| 장점 항목 | 실익 | 이유 |
|-----------|:---:|------|
| `LENGTH >= 100` 필터 | ❌ | heading 청크 제거 → hierarchy 분석 목적에 역효과 |
| `(page, chunk_index)` 복합 정렬 | ❌ | `chunk_index`가 이미 전역 순번 — `page` 추가 불필요 |
| heading 우선 선택 | △ | LLM 청킹에서 `chunk_type='heading'` 독립 청크 희소 |
| 버킷 균일성 | △ | 말미 누락 미미 / document_id 버전 고정·결정적 샘플링 장점 상실 |

**결론**: 현재 stride 방식 유지. `setup_vector_db.sql`을 실제 Supabase 함수와 동기화 필요.

#### 1. 비동기화 Job ID — 기존 Generation 패턴 재사용

**현재 상태**

`POST /api/ingestion/upload` — `await process_and_ingest(...)` 직접 호출, 단일 HTTP 요청에서 동기 처리. 타임아웃 위험.

**인제스션 전용 설계**

인제스션은 단계가 세분화되므로 별도 `IngestionJobManager`와 `IngestionStatus` enum 분리 권장 (기존 `job_manager` 오염 방지):

```python
# ingestion/job_manager.py (신규)
class IngestionStatus(str, Enum):
    PENDING    = "pending"
    EXTRACTING = "extracting"   # PDF/DOCX 텍스트 추출
    CHUNKING   = "chunking"     # LLM 청킹 (배치 진행률 가능)
    EMBEDDING  = "embedding"    # 임베딩 + DB 저장
    COMPLETED  = "completed"
    FAILED     = "failed"

@dataclass
class IngestionJob:
    job_id:    str
    status:    IngestionStatus
    progress:  int = 0          # 현재 배치
    total:     int = 0          # 전체 배치 수
    message:   str = ""
    error:     Optional[str] = None
    filename:  str = ""
    started_at:   str = ""
    completed_at: Optional[str] = None
```

**API 흐름**

```
POST /api/ingestion/upload
  └─ ingestion_job_manager.create_job(job_id)
  └─ background_tasks.add_task(process_and_ingest, job_id=job_id, ...)
  └─ 즉시 반환: { job_id, status: "pending" }

GET /api/ingestion/{job_id}/status
  └─ { status, progress, total, message }
```

`process_and_ingest()`에 `job_id` 인자 추가 후 단계별 `ingestion_job_manager.update_job()` 호출.

**구현 범위**: `ingestion/job_manager.py` 신규 / `ingestion_api.py` BackgroundTasks 전환 / `pipeline.py` job_id 전파 / 프론트엔드 `DataStandardizationPanel` 폴링 추가.

#### 2. Semaphore 계층 설계

**계층별 설계 및 근거**

| 계층 | 변수 | 권장값 | 근거 |
|------|------|:---:|------|
| 동시 인제스션 수 | `INGESTION_SEM` | 2 | 대용량 파일 동시 처리 시 메모리·CPU 보호. `ingestion_api.py` 상단 전역 |
| LLM 청킹 API | `LLM_CHUNK_SEM` | 5 | llm_chunker.py `parallel` 티어 파라미터와 역할 중복 — **불필요**. 티어 시스템이 이미 제어함 |
| 임베딩 API | `EMBED_SEM` | 5 | `asyncio.gather(*embed_tasks)` 무제한 → Semaphore 추가. 현재 최대 배치 수는 5개 미만이나 동시 업로드 시 필요 |

**LLM 청킹 Semaphore는 불필요**: `recommend_params()` / `recommend_params_docx()`가 이미 `parallel` 값(3~5)으로 LLM 호출 동시 수를 제어함. 중복 제어는 오히려 처리 지연.

**임베딩 Semaphore 구현**

```python
# pipeline.py 상단
_EMBED_SEM = asyncio.Semaphore(int(os.getenv("EMBED_CONCURRENCY", "5")))

async def _embed_and_save_guarded(*args, **kwargs):
    async with _EMBED_SEM:
        await _embed_and_save(*args, **kwargs)

# 기존 gather 교체
await asyncio.gather(*[
    _embed_and_save_guarded(i // _EMBED_BATCH_SIZE + 1, i,
                            all_chunks_to_embed[i: i + _EMBED_BATCH_SIZE],
                            **embed_kwargs)
    for i in range(0, total_chunks, _EMBED_BATCH_SIZE)
])
```

#### 3. 임베딩 재시도 — tenacity 불필요

**현재 상태**

**tenacity 도입 검토 — 불채택**

`tenacity`는 `pyproject.toml`에 없는 신규 의존성. 기존 코드베이스 전체가 수동 retry 루프 패턴을 사용하므로 일관성 유지 및 의존성 최소화를 위해 동일 패턴 적용.

```python
# pipeline.py — _embed_and_save 내 embed_content 재시도 추가
for attempt in range(3):
    try:
        res = await gemini_client.aio.models.embed_content(
            model=_EMBED_MODEL, contents=batch_texts,
            config=google_genai.types.EmbedContentConfig(
                task_type="RETRIEVAL_DOCUMENT",
                output_dimensionality=1536,
            ),
        )
        break
    except Exception as e:
        if attempt == 2:
            raise
        wait = 2 ** attempt   # 1s → 2s → raise
        logger.warning(f"embed_content retry {attempt + 1}/3 (wait={wait}s): {e}")
        await asyncio.sleep(wait)
```

### 실패 데이터 선별 재처리 — 설계 구상

#### 현재 실패 구조의 핵심 문제

`process_and_ingest`는 매 호출마다 `doc_id = str(uuid4())` 신규 생성. 실패 후 재시도하면 **새 doc_id**로 전체 재처리 — 실패 시 저장된 부분 청크는 DB에 orphan으로 잔류.

```
1차 시도 (doc_id=A):  청크 0~49 저장 성공 → 청크 50 임베딩 실패 → FAILED
2차 시도 (doc_id=B):  전체 청크 재처리, 재삽입
                       doc_id=A의 청크 0~49 → orphan (DB에 남아 있음)
```

dedup 키는 `(content_hash, document_id)` 쌍이므로 새 doc_id에서는 기존 저장 여부를 판별하지 못함.

---

#### 실패 유형 분류 및 재처리 가능성

| 단계 | 실패 원인 | DB 잔류 데이터 | 재처리 가능 |
|------|-----------|:--------------:|:-----------:|
| EXTRACTING | 이미지 PDF, 손상 파일, 지원 외 형식 | 없음 | ❌ 파일 교체 필요 |
| EXTRACTING | 일시적 파싱 오류 | 없음 | ✅ 즉시 재시도 |
| CHUNKING | Gemini API 429/500 | doc_metadata만 (청크 없음) | ✅ 재시도 가능 |
| EMBEDDING | API 오류 (배치 N 이후 실패) | **부분 청크 orphan** | ✅ orphan 정리 후 재시도 |
| DB SAVE | Supabase 연결 오류 | 임베딩 완료 청크 일부 누락 | ✅ orphan 정리 후 재시도 |

---

#### 설계: Same doc_id 재시도 + orphan 정리

재시도 시 기존 doc_id 재사용 + 부분 저장된 청크 삭제 후 전체 재실행.

**핵심 아이디어**: dedup 키 `(content_hash, doc_id)`를 이용해 이미 저장된 청크는 skip — 단, 중간 실패라면 어느 청크가 저장됐는지 불확실하므로 **정리 후 재시도**가 안전.

```
재시도 흐름:
  1. IngestionJob에서 실패 job의 doc_id 조회
  2. DELETE doc_chunks WHERE document_id = failed_doc_id  (orphan 정리)
  3. process_and_ingest(doc_id=failed_doc_id, ...)  (기존 doc_id 재사용)
  4. upsert_doc_metadata는 idempotent → 재호출 무해
```

**장점**
- doc_metadata의 document_id는 유지 → 기존 hierarchy 태깅 결과 재사용 가능
- 불완전 저장 없이 클린 재시작 보장

---

#### 구현 범위

**백엔드 신규/수정 파일**

```
backend/db/doc_chunk_repo.py
  + async def delete_chunks_by_document_id(doc_id: str) -> int
      DELETE doc_chunks WHERE document_id = doc_id
      반환: 삭제된 청크 수 (로깅용)

backend/ingestion/pipeline.py
  - process_and_ingest(...) 시그니처에 doc_id: Optional[str] = None 추가
      None이면 uuid4() 생성 (기존 동작), 값이 있으면 그대로 사용

backend/ingestion/job_manager.py  (1번 비동기화 구현 시 신규)
  - IngestionJob에 doc_id: str 필드 추가
  - status에 실패 단계 기록 (failed_stage: str)

backend/api/ingestion_api.py
  + POST /api/ingestion/{job_id}/retry
      1. job 조회 → status == FAILED 검증
      2. delete_chunks_by_document_id(job.doc_id)
      3. background_tasks.add_task(process_and_ingest, doc_id=job.doc_id, ...)
      4. job status → PENDING 리셋
```

**프론트엔드 (`DataStandardizationPanel.tsx`)**

업로드 상태 UI에 FAILED 시 "재시도" 버튼 추가 → `POST /api/ingestion/{job_id}/retry` 호출.

아래 케이스는 재시도 API 호출 시 즉시 `400 Bad Request` 반환:

| 조건 | 메시지 |
|------|--------|
| `failed_stage == "extracting"` AND `error` 에 "이미지 기반" / "텍스트 없음" 포함 | "파일 자체 문제 — 다른 파일을 업로드하세요" |
| 동일 job_id 재시도 3회 초과 | "최대 재시도 횟수 초과" |

### QA 품질 기반 재처리 — 설계 구상 및 실현 가능성 검토

#### 현재 데이터 구조 확인

재처리 설계 전 코드베이스 실제 상태 점검.

**개별 QA 식별자 문제 — 핵심 선행 조건**

`qa_gen_results.qa_list` 내 개별 QA 항목에 안정적 UUID가 없음. 현재 `qa_id`는 평가 파이프라인(`pipeline.py:519`)에서 위치 기반으로 합성:

```python
"qa_id": qa.get("qa_id", f"qa_{result_idx}_{qa_idx}")
```

재처리 후 qa_list 순서가 바뀌면 `qa_0_2`가 다른 QA를 가리킬 수 있어 **히스토리 추적 불가**. 모든 Phase의 선행 조건.

**평가 피드백 데이터 이미 존재**

`qa_eval_results.pipeline_results.layers.rag.qa_scores`:
```json
{
  "qa_index": 3,
  "groundedness": 0.45,
  "groundedness_reason": "답변에 컨텍스트에 없는 수치가 포함됨",
  "relevance_reason": "...",
  "failure_types": ["hallucination"],
  "primary_failure": "hallucination"
}
```

Phase B 피드백 주입에 필요한 reason 텍스트는 이미 저장됨. 조회 경로만 구현하면 됨.

**원천 컨텍스트 재조회 가능**

`qa_list[].docId` = chunk UUID → `doc_chunks` 테이블에서 원문 재조회 가능.

---

#### 재처리 대상 3가지 — 실현 가능성 분류

**① 기술적 실패 (Technical Failure)**

LLM 응답 거부, API 타임아웃, JSON 파싱 에러.

현황: `qa_list[].qa_list`가 빈 배열인 결과가 저장됨. `total_qa == 0 → FAILED`는 이미 처리됨. 개별 청크 수준 실패(특정 청크의 QA가 0개)는 추적되지 않음.

실현 가능성: ✅ 중간 — docId 재조회 + 해당 청크만 재생성 가능. qa_id 안정화 선행 필요.

**② 품질 미달 (Low Quality)**

RAG Triad 또는 Completeness 임계치 이하.

현황: `groundedness < 0.6` 등 failure_types가 이미 `pipeline_results`에 기록됨. 프론트엔드에서 필터링도 가능. reason 텍스트도 저장되어 있어 피드백 주입 재료 확보됨.

실현 가능성: ✅ 높음 — 데이터 기반이 가장 충실.

**③ 커버리지 갭 (Coverage Gap)**

특정 H1/H2에서 목표 수량 미달.

현황: "목표 수량" 개념 자체가 현재 시스템에 없음. 생성 시 `samples` 파라미터만 존재하고 H1/H2별 목표 분배 로직 없음.

실현 가능성: ❌ 낮음 — 목표 수량 관리 레이어 신규 설계 필요. 현재 스코프 밖.

---

#### Phase별 설계 및 실현 방안

**Phase A: 실패 원인 분석 (조회)**

이미 `pipeline_results`에 per-QA failure 데이터가 있으므로 별도 분석 로직 불필요. 필요한 것은 조회 인터페이스:

```
GET /api/evaluate/{eval_job_id}/failed-qa
  → qa_index, q, a, context, failure_types, primary_failure,
     groundedness_reason, docId (원천 청크 ID)
  필터: ?min_groundedness=0.6 또는 ?failure_type=hallucination
```

**Phase B: 타겟팅 재생성**

*피드백 주입*: `prompts.py`에 `build_recovery_user_template(negative_constraint: str)` 추가. 기존 `build_user_template()`에 실패 reason을 prefix로 삽입:

```python
def build_recovery_user_template(domain_profile, chunk_type, total_chunks,
                                  failure_reason: str) -> str:
    base = build_user_template(domain_profile, chunk_type, total_chunks)
    constraint = (
        f"[재생성 제약] 이전 생성 실패 사유: {failure_reason}\n"
        f"위 문제를 반드시 해결하여 재생성하세요.\n\n"
    )
    return constraint + base
```

*모델 업그레이드*: `recovery_model` 파라미터를 `POST /api/generate` 또는 별도 recovery API에 추가. 기존 generation worker에 model 파라미터 이미 있으므로 값만 교체.

**Phase C: 병합 및 교체**

현재 `qa_list`는 job 단위 JSONB — 개별 QA 교체가 구조적으로 어려움.

현실적 접근: **교체가 아닌 보완 생성** — 재생성 결과를 별도 `qa_gen_results` row로 저장하고, 원본 row의 `linked_recovery_job_id` FK로 연결. 평가 시 두 집합을 합산하거나 선택.

DB 최소 변경:
```sql
ALTER TABLE qa_gen_results
  ADD COLUMN retry_count     INT     DEFAULT 0,
  ADD COLUMN parent_job_id   TEXT,   -- 재생성인 경우 원본 job_id
  ADD COLUMN is_recovery     BOOLEAN DEFAULT FALSE;
```

`qa_id` UUID화 (선행):
```python
# qa_generator.py — QA 생성 시 UUID 부여
for qa in parsed_qa_list:
    qa["qa_id"] = str(uuid4())
```

#### 구현 로드맵

```
Step 1 (선행, 독립)
  qa_generator.py — 생성 시 qa_id UUID 부여
  → 이후 모든 재처리 추적의 기반

Step 2
  GET /api/evaluate/{eval_job_id}/failed-qa
  → failure_types 필터로 재처리 대상 목록 반환

Step 3
  prompts.py — build_recovery_user_template() 추가
  POST /api/generate/recovery — docId 목록 + failure_reason + recovery_model 수신
  → 해당 청크만 재생성 + qa_gen_results에 is_recovery=true로 저장

Step 4
  qa_gen_results 스키마 확장 (retry_count, parent_job_id, is_recovery)
  프론트엔드 재처리 UI + Diff View
```

**커버리지 갭은 현재 스코프 제외** — H1/H2별 목표 수량 관리 레이어 설계가 선행되어야 함.
