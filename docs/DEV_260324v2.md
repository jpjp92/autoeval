### 7-4. get_doc_chunks_sampled — Supabase 인덱스 기반 균등 샘플링

Supabase(PostgreSQL)는 row 인덱스 직접 조회 불가.
`doc_chunk_repo.py`의 `get_document_chunks()`는 `order("created_at").limit(N)` — 앞부분만 반환.

#### chunk_index 충돌 문제 (2026-03-24 확인)

`ingestion_api.py:265`에서 `"chunk_index": i + idx` 로 저장됨.
이는 **배치 내 상대 인덱스**이므로 동일 파일 재인제스천 시 중복 chunk_index 발생.

```
1차 인제스천: chunk_index = 0, 1, 2, ... (batch 상대값)
2차 인제스천: chunk_index = 0, 1, 2, ... (동일 값 → 충돌)
```

→ filename + chunk_index로 청크를 유니크 식별할 수 없음.
→ RPC 정렬 기준을 `document_id` + `chunk_index` 복합으로 변경 필요.

> **`document_id` 확인**: `ingestion_api.py:123` `doc_id = str(uuid4())`, `:255` `"document_id": doc_id` — 인제스천당 유일한 uuid4로 저장 확인됨.

#### 방법 A: Python 측 OFFSET 루프 (N+1 쿼리)

```python
async def get_doc_chunks_sampled(filename: str, n: int = 30) -> list:
    # 1. 전체 개수 조회 (1회)
    count_res = (
        supabase.table("doc_chunks")
        .select("id", count="exact")
        .eq("metadata->>filename", filename)
        .execute()
    )
    total = count_res.count or 0
    if total == 0:
        return []

    step = max(total // n, 1)
    offsets = list(range(0, total, step))[:n]

    # 2. OFFSET별 조회 (N회) → 비효율적, 총 N+1 쿼리
    results = []
    for offset in offsets:
        row = (
            supabase.table("doc_chunks")
            .select("id, content, metadata")
            .eq("metadata->>filename", filename)
            .order("created_at")
            .offset(offset)
            .limit(1)
            .execute()
        )
        if row.data:
            results.append(row.data[0])
    return results
```

> ⚠️ N+1 쿼리 — 30개 샘플링 시 31회 조회. 느리고 비효율적.

#### 방법 B: Supabase RPC (SQL 함수) — 권장

Supabase SQL Editor에 함수 1개 등록, 이후 단일 RPC 1회 호출:

```sql
-- Supabase SQL Editor에 등록
-- chunk_index 충돌 대응:
--   - 동일 파일 재인제스천 시 chunk_index 중복 → p_document_id(uuid)로 인제스천 구분
--   - document_id 미지정 시 가장 최근 인제스천(MAX created_at) 기준으로 정렬
CREATE OR REPLACE FUNCTION sample_doc_chunks(
    p_filename    TEXT,
    p_n           INT DEFAULT 30,
    p_document_id TEXT DEFAULT NULL   -- ← 인제스천 uuid (anchor_ids 발급 시 함께 전달)
)
RETURNS SETOF doc_chunks AS $$
DECLARE
    total_count INT;
    step_size   INT;
    v_doc_id    TEXT;
BEGIN
    -- document_id 미지정 시 가장 최근 인제스천의 document_id 자동 선택
    IF p_document_id IS NULL THEN
        SELECT metadata->>'document_id' INTO v_doc_id
        FROM doc_chunks
        WHERE metadata->>'filename' = p_filename
        ORDER BY created_at DESC
        LIMIT 1;
    ELSE
        v_doc_id := p_document_id;
    END IF;

    IF v_doc_id IS NULL THEN RETURN; END IF;

    SELECT COUNT(*) INTO total_count
    FROM doc_chunks
    WHERE metadata->>'document_id' = v_doc_id;

    IF total_count = 0 THEN RETURN; END IF;
    step_size := GREATEST(total_count / p_n, 1);

    RETURN QUERY
    SELECT * FROM (
        SELECT *, ROW_NUMBER() OVER (
            ORDER BY (metadata->>'chunk_index')::int
        ) AS rn
        FROM doc_chunks
        WHERE metadata->>'document_id' = v_doc_id
    ) t
    WHERE (t.rn - 1) % step_size = 0
    LIMIT p_n;
END;
$$ LANGUAGE plpgsql STABLE;
```

> **변경 요점**:
> - `p_filename` 필터 → `metadata->>'document_id'` 필터 (재인제스천 chunk_index 충돌 방지)
> - `p_document_id` 미지정 시 `created_at DESC` 기준 최신 인제스천 자동 선택
> - `chunk_index` 정렬 유지 (`ingestion_api.py:265` 저장 확인됨)

```python
# doc_chunk_repo.py — RPC 호출 1회
async def get_doc_chunks_sampled(filename: str, n: int = 30) -> list:
    if not supabase:
        return []
    try:
        response = supabase.rpc(
            "sample_doc_chunks",
            {"p_filename": filename, "p_n": n},
        ).execute()
        return response.data if response.data else []
    except Exception as e:
        logger.error(f"❌ get_doc_chunks_sampled failed: {e}")
        return []
```

| 방법 | 쿼리 수 | 구현 복잡도 | 권장 |
|------|--------|-----------|------|
| A (OFFSET 루프) | N+1 (30→31회) | 낮음 | ❌ 비효율 |
| B (RPC) | 1회 | SQL 함수 등록 필요 | ✅ 권장 |

> **착수 조건**: Supabase SQL Editor에서 `sample_doc_chunks` 함수 등록 확인 후 구현.
