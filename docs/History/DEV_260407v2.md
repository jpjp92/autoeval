# 백엔드 개발 로그 (2026-04-07 v2)

## 백엔드 흐름 검증 및 업데이트 사항 (`REF_BACKFLOW.md` 기준)

오늘 `docs/Guide/REF_BACKFLOW.md` 문서에 정의된 백엔드 전체 흐름과 실제 구현 코드(`backend/`)를 비교 검증한 결과, 대부분의 핵심 로직이 문서와 일치하게 구현되어 있음을 확인했습니다.

### 1. 주요 검증 결과

* **진입점 (Entry Point)**: `backend/main.py`에서 CORS 설정, Ingestion/Generation/Evaluation 라우터 등록 및 `/health` 엔드포인트가 문서대로 구현됨.
* **Ingestion API**:
  * PDF(300자) 및 DOCX(100자) 최소 길이 제한 로직 확인.
  * `analyze-hierarchy` (Pass 1+2 통합) 및 `apply-granular-tagging` (Pass 3) 로직이 Gemini 모델을 통해 정상적으로 작동함.
  * 태깅 시 `batch_size=5`, `concurrency=5` 설정이 문서와 일치함.
* **Generation API**:
  * 비동기 작업(`run_qa_generation_real`)으로 구현되었으며, 워커 수(Anthropic: 2, Gemini: 5, GPT: 5) 및 429 에러 발생 시의 Fallback 로직이 정확히 반영됨.
  * 청크 간 중복 제거(`_dedup_across_chunks`) 시 `SequenceMatcher` 0.75 임계값 사용 확인.
* **Evaluation API**:
  * 4단계 파이프라인(Syntax, Stats, RAG Triad, Quality)이 모두 구현되어 있으며, 최종 점수 가중치 및 등급 산정 로직이 문서와 일치함.

### 2. 발견된 미세한 불일치 및 재검토 결과

> 코드와 `config/models.py`를 직접 비교하여 아래와 같이 재판단함.

| 항목                                                              | 위치                            | 판단                                                                                                                                               |
| ----------------------------------------------------------------- | ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `analyze-hierarchy` hardcoded `"gemini-3-flash-preview"`      | `api/ingestion_api.py:191`    | **정상** — `MODEL_CONFIG["gemini-3-flash"]["model_id"]`와 동일한 값. 단, MODEL_CONFIG를 경유하지 않아 모델 ID 변경 시 수동 동기화 필요. |
| `evaluation_api.py` 기본 `"gemini-2.5-flash"`                 | `api/evaluation_api.py:78,90` | **정상** — `MODEL_CONFIG["gemini-flash"]["model_id"]`와 동일한 값. 역시 MODEL_CONFIG 미경유 패턴.                                         |
| `qa_quality.py` 기본 `"gpt-5.1"`                              | `evaluators/qa_quality.py:36` | **충돌 아님** — Completeness 레이어 전용 evaluator 기본값이며, 전체 평가 파이프라인 기본 모델(`gemini-2.5-flash`)과 별개.                 |
| 문서 표기 `Gemini 3.1 Flash` vs 코드 `gemini-3-flash-preview` | `docs/`                       | **문서 표기 문제** — 모델의 공식 표시 이름은 `Gemini 3.1 Flash`이고 API model_id는 `gemini-3-flash-preview`. 코드는 정상.               |

**결론**: 런타임 동작상 버그 없음. 다만 `ingestion_api.py:191`, `evaluation_api.py:78,90` 두 곳에서 MODEL_CONFIG를 우회하는 패턴이 존재하며, 향후 모델 ID 변경 시 누락될 수 있어 유지보수성 개선 여지가 있음.

### 3. 향후 계획

* `api/ingestion_api.py:191` 및 `api/evaluation_api.py:78,90`의 raw 모델 ID 문자열을 `MODEL_CONFIG[...]["model_id"]` 참조로 통일하는 것 고려 (선택적 개선).
* 프론트엔드(`frontend/`)와의 연동 상태 추가 점검.

## Ingestion 동기 처리 구조 검토

### 현재 구조

`/api/ingestion/` 세 단계 모두 **동기(blocking)** 처리.

| 엔드포인트 | 처리 방식 | 특징 |
|-----------|-----------|------|
| `POST /upload` | 동기 | 청킹 + 임베딩 + DB 저장 완료 후 200 응답 |
| `POST /analyze-hierarchy` | 동기 | LLM 단일 호출 완료 후 응답 |
| `POST /apply-granular-tagging` | 동기 | 청크 전체 배치 태깅 완료 후 응답 |

Generation은 `background_tasks.add_task()`로 즉시 `job_id`를 반환하는 비동기와 대비됨.

### 장단점

**동기 처리 (현재)**

| 장점 | 단점 |
|------|------|
| 구현 단순 — job 관리, 폴링 로직 불필요 | 대용량 문서 시 HTTP 타임아웃 위험 |
| 완료 시점이 명확 — 응답 받으면 바로 다음 단계 진행 가능 | 프론트가 응답 대기 중 UI 블로킹 (스피너 외 피드백 없음) |
| 에러 핸들링 직관적 — HTTP 상태코드로 즉시 전달 | 서버 재시작 시 진행 중 작업 소실 |
| — | 여러 사용자가 동시 업로드 시 요청 큐잉 없이 처리됨 |

**비동기 처리 (Generation 방식)**

| 장점 | 단점 |
|------|------|
| 타임아웃 없이 장시간 작업 가능 | job 상태 관리 + 폴링 구현 필요 |
| 진행률 표시 가능 (`progress` 필드) | 에러 전달 경로가 복잡 (job status 안에 포함) |
| 서버 부하 분산 유리 | 구현 복잡도 증가 |

### 결론

현재 문서 규모(단일 PDF/DOCX)에서는 동기 처리가 실용적이며 문제없음. 단, 향후 대용량 배치 업로드나 다중 파일 동시 처리가 필요해지면 Generation과 동일한 비동기 job 패턴으로 전환 검토 필요.

## 프론트엔드 버그 수정 — 파일 업로드 후 화면 공백(White Screen)

### 증상

파일 업로드 및 태깅 완료 후 프론트엔드 화면이 완전히 하얗게 빈 화면으로 전환됨. 백엔드 로그에는 정상 처리(DB 저장 완료)가 찍히나 프론트는 응답 없음.

### 원인 분석

`frontend/src/lib/api.ts`의 `getHierarchyList` 함수에서 백엔드 flat JSON 응답을 잘못 파싱한 것이 근본 원인.

```
업로드 완료
  → setCurrentFilename(filename)
  → useEffect[currentFilename] 트리거
  → loadHierarchyList() 호출
  → getHierarchyList() 내부에서 ...result.data! spread
  → result.data === undefined (백엔드가 flat JSON 반환, data 키 없음)
  → h1_list: undefined, h2_by_h1: undefined 상태 저장
  → setHierarchyLoaded(true) → 리렌더
  → undefined.map() → TypeError
  → Error Boundary 없음 → White Screen
```

백엔드 `/api/ingestion/hierarchy-list`는 `{ success, h1_list, h2_by_h1, h3_by_h1_h2 }` 형태의 flat JSON을 반환하지만, 기존 코드는 `ApiResponse<T>` 패턴(`{ success, data: T }`)을 가정하고 `result.data`를 spread 하고 있었음.

### 수정 내용 (`frontend/src/lib/api.ts`)

```typescript
// BEFORE — data 키가 없으면 undefined spread → 런타임 crash
const result = await apiFetch(`...`);
return { success: true, ...result.data! };

// AFTER — flat JSON을 직접 타입 캐스팅하여 안전하게 추출
const result = await apiFetch(`...`) as unknown as {
  success: boolean;
  h1_list?: string[];
  h2_by_h1?: Record<string, string[]>;
  h3_by_h1_h2?: Record<string, string[]>;
  error?: string;
};
return {
  success: true,
  h1_list:     result.h1_list     ?? [],
  h2_by_h1:    result.h2_by_h1    ?? {},
  h3_by_h1_h2: result.h3_by_h1_h2 ?? {},
};
```

### 비고

* 백엔드 자체는 정상 동작 — 이슈는 순수 프론트엔드 파싱 버그.
* `App.tsx`에 Error Boundary가 없어 TypeError가 화면 전체를 날렸음. 추후 Error Boundary 추가 검토.
