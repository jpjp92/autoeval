# DEV_260327 — LLM 기반 청킹 테스트 결과

## 배경

기존 rule-based 청킹(`chunk_blocks_aware`)이 생성한 청크에서 QA 품질 문제 발생:
- ο (U+03BF Greek omicron) 불릿 잔존으로 문장 흐름 단절
- `y` 문자 (폰트 인코딩 오류 → 불릿 기호) 오염
- 글자 사이 공백 (`산 업 자 원 부`) 미복원
- 단일 PDF 블록에 복수 섹션 헤더 혼합 → 청크 경계 혼란
- 목차 페이지(p12 이후) TOC 필터 미적용으로 불필요한 청크 생성

→ **Gemini 2.5 Flash를 직접 활용한 LLM 청킹으로 전환 테스트** 실시

## 구현: `test/chunk_with_llm.py`

### 파이프라인

```
PDF
 └─ extract_text_by_page()         # rawdict + _span_text_with_spaces (공백 복원)
     └─ 원시 블록 목록 (page, y0, text)
         └─ LLM 청킹 (배치 단위)
             ├─ CTX 블록: 이전 배치 마지막 N개 (문맥 참조용, 출력 제외)
             └─ NEW 블록: 이번 배치 청킹 대상
                 └─ merge_short_chunks()     # <100자 청크 인접 청크에 흡수
                     └─ deduplicate_chunks() # Jaccard>0.5 중복 제거
```

### 핵심 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| 공백 복원 | `parsers.extract_text_by_page()` 재사용 | rawdict + `_span_text_with_spaces` 로 40% 임계값 기반 한글 공백 복원 |
| 노이즈 제거 | LLM 프롬프트 규칙으로 처리 | ο, y, Smart, •, Ÿ 등 불릿 패턴 기술 → rule 함수 없이 제거 |
| 배치 overlap | CTX/NEW 태깅 방식 | 이전 배치 마지막 3블록을 CTX로 전달, stride=batch_size로 NEW 중복 없음 |
| 제목 통합 | LLM 프롬프트 규칙 | `제목\n내용` 형식으로 독립 헤더 청크 방지 |
| 짧은 청크 처리 | `merge_short_chunks(min=100)` | LLM이 100자 미달 생성 시 인접 청크에 흡수 (반복 수렴) |
| 중복 제거 | Jaccard(source_blocks) > 0.5 | 배치 경계에서 동일 블록 참조 청크 제거 |

### LLM 프롬프트 주요 규칙

프롬프트는 XML 태그 방식으로 구조화 (`<role>`, `<input_format>`, `<noise_correction>`, `<chunking_rules>`, `<section_title_rules>`, `<output_format>`):

- `<role>`: 범용 청킹 전문가 (문서 유형 무관)
- `<input_format>`: `[CTX-N]` 참고용, `[NEW-N]` 청킹 대상
- `<noise_correction>`: 문장 단절, 불릿 제거, 글자 공백, 숫자·단위 분리, 구두점 위치
- `<chunking_rules>`:
  - 제목 블록(Ⅰ Ⅱ □ 1. 1.1 1단계 등) → 다음 청크 첫 줄로 통합
  - 최소 **200자**, 최대 **800자** (의미 단위 보존 우선, 표 포함 청크 수용)
  - 청크 끝맺음: 접속사·연결어(또한, 그러나, 따라서 등)로 끝나는 경우 다음 청크 앞으로 이동
- `<table_rules>`: `[표]` 마크다운 표 블록 → 파이프·구분선 제거 후 유형별 산문 변환 (정의형/조항형/비교형/목록형)
- `<section_title_rules>`: 청크 전체 내용 기반 핵심 주제 15자 이내 요약
- `<output_format>`: 순수 JSON 배열, 마크다운 감싸기 금지, source_blocks에 NEW 인덱스만 기록

---

### 청크 출력 필드 (chunk response body)

```json
{
  "chunk_index": 4,
  "text": "광역 감시/제어 시스템 개발 목적\n본 사업은 전력계통의...",
  "section_title": "광역 감시/제어 시스템 개발 목적",
  "source_blocks": [10, 11, 12, 13],
  "page": 4
}
```

- `section_title`: LLM이 컨텍스트 기반 요약 추출 (15자 이내), 누락 시 chunk 첫 줄 fallback
- `page`: source_blocks 첫 번째 블록의 page 번호
- `source_blocks`: NEW 블록 인덱스만 (CTX 제외)

## 평가 스코어 공식 변경 (2026-03-27)

### 변경 이유
품질 평가가 완전성(completeness) 단일 지표로 축소됨에 따라 기존 곱셈형 게이트 방식이 completeness에 과도한 영향력 부여 → additive 방식으로 전환 및 비율 재조정.

### 변경 내용

| 레이어 | 이전 | 변경 |
|--------|------|------|
| Syntax | 10% | **5%** |
| Stats | 10% | **5%** |
| RAG Triad | ~80% (completeness 곱셈 게이트) | **65%** (독립 가중) |
| Completeness | 곱셈 게이트 | **25%** (독립 가중) |

```python
# 이전 (곱셈 게이트)
final_score = syntax*0.1 + stats*0.1 + (rag_avg * quality_avg) * 0.8

# 변경 (additive)
final_score = syntax*0.05 + stats*0.05 + rag_avg*0.65 + quality_avg*0.25
```

**점수 비교 예시**

| 케이스 | rag | completeness | 이전 점수 | 변경 점수 |
|--------|-----|-------------|-----------|-----------|
| 우수 | 0.87 | 1.00 | 0.796 | 0.916 |
| 완전성 낮음 | 0.87 | 0.60 | 0.618 | 0.816 |
| RAG 낮음 | 0.65 | 0.90 | 0.568 | 0.690 |

## 대규모 문서 처리 전략

### 문서 규모 판단 기준

스크립트 실행 시 PDF 메타데이터(총 페이지 수)를 먼저 읽어 규모 티어를 자동 감지하고 파라미터를 추천한다.

```python
doc.page_count  # fitz.open() 후 즉시 확인 가능 (블록 추출 전)
```

### 문서 티어별 파라미터 정책

| 티어 | 페이지 | 예상 블록 | batch_size | parallel | overlap | max_output_tokens | 예상 소요 | 예상 비용 |
|------|--------|-----------|------------|----------|---------|-------------------|-----------|-----------|
| S (소형) | ≤ 20p | ~80블록 | 30 | 3 | 3 | 8,192 | ~1분 | ~$0.002 |
| M (중형) | 21–50p | ~150블록 | 30 | 3 | 3 | 8,192 | ~2분 | ~$0.004 |
| L (대형) | 51–100p | ~300블록 | 40 | 5 | 3 | 12,288 | ~3분 | ~$0.007 |
| XL (초대형) | 101–200p | ~600블록 | 50 | 5 | 5 | 16,384 | ~5분 | ~$0.013 |
| XXL (극대형) | 200p 초과 | ~1,000블록+ | 50 | 5 | 5 | 16,384 | ~8분+ | ~$0.022+ |

> **비용 산정 기준**: Gemini 2.5 Flash, thinking OFF (`thinking_budget=0`) 기준. Input $0.075/1M tok, Output $0.30/1M tok. 배치당 입력 ~3,900tok + 출력 ~900tok ≈ $0.0006. thinking ON 대비 약 5-7배 절감.

> **batch_size 상한을 50으로 제한하는 이유**: 표 블록(~1,600자)이 포함된 배치는 블록 30개 기준으로도 ~7,000tok 이상 발생할 수 있음. 50블록을 초과하면 단일 배치 응답 지연이 150초를 넘어 타임아웃 위험 증가.

### 문자 수 기반 배치 분할 (token-aware)

페이지 수 외에 **블록당 평균 문자 수**도 배치 크기 결정에 반영한다:

```
배치 누적 문자 수 > 4,000자  →  해당 배치 마감, 다음 배치 시작
```

표 블록(500~1,600자)이 많은 문서는 block 수 기준 batch_size에 도달하기 전에 먼저 문자 수 기준이 발동되어 배치 크기가 자동 축소된다.

### 단계별 처리 흐름 (대규모 문서)

```
1. PDF 열기 → page_count 확인 → 티어 판정 → 파라미터 자동 설정
2. 블록 추출 → 블록 수·평균 문자 수 확인 → 배치 목록 구성 (문자 수 기준 우선)
3. 비동기 병렬 청킹 (Semaphore로 parallel 제한)
4. [추후] 체크포인트: 배치 완료마다 .checkpoint.json 저장 → 실패 시 재개 가능
5. merge_short_chunks → trim_dangling_conjunctions → deduplicate_chunks
6. 결과 저장
```

### 구현 현황 / 추후 과제

| 항목 | 상태 | 설명 |
|------|------|------|
| 자동 티어 감지 + 파라미터 추천 | ✅ 완료 (v6) | `get_pdf_page_count()` + `recommend_params()` — 실행 시 자동 적용 |
| 문자 수 기반 배치 분할 | ✅ 완료 (v6) | `build_char_aware_batches(max_chars=4000)` — 표 블록 토큰 폭발 방지 |
| 체크포인트/재개 (`--resume`) | ⬜ 미구현 | 100p+ 문서 실패 시 재시작 비용 절감 |
| 스트리밍 저장 | ⬜ 미구현 | 배치 완료마다 append → 중간 결과 확인 |
| `split_long_chunks()` 후처리 | ⬜ 미구현 | 800자 초과 단일 블록(대형 표) 문장 경계에서 추가 분리 |

## ingestion_api.py LLM 청킹 통합 계획

### 목표

`test/chunk_with_llm.py`의 LLM 청킹 로직을 백엔드 파이프라인에 통합.
기존 rule-based 청킹은 하위 호환을 위해 유지하고, `chunking_method` 파라미터로 선택.

### 파일 구조

```
backend/ingestion/
├── __init__.py
├── parsers.py          # 기존 유지 (rule-based)
└── llm_chunker.py      # NEW — LLM 청킹 모듈
```

### `llm_chunker.py` 포함 내용

| 항목 | 설명 |
|------|------|
| `SYSTEM_PROMPT`, `USER_TEMPLATE` | 프롬프트 상수 (test 스크립트와 동일) |
| `recommend_params(page_count)` | 티어 기반 파라미터 자동 추천 |
| `build_char_aware_batches()` | 블록 수·문자 수 이중 기준 배치 분할 |
| `build_block_list()` | CTX/NEW 태깅 블록 목록 생성 |
| `_call_gemini(prompt, batch_size, client)` | gemini_client 외부 주입 방식으로 변경 |
| `_process_batch()` | 단일 배치 비동기 처리 |
| `merge_short_chunks()` | 100자 미만 청크 흡수 |
| `trim_dangling_conjunctions()` | 접속사 꼬리 후처리 |
| `deduplicate_chunks()` | Jaccard 중복 제거 |
| `run_llm_chunking(blocks, client, ...)` | **메인 진입점** |

**제거 항목**: Rich progress, `argparse`, CLI 전용 코드 (test 스크립트에만 유지)

**`run_llm_chunking` 인터페이스**:
```python
async def run_llm_chunking(
    blocks: list[dict],          # [{page, text, index}, ...]
    client,                      # google_genai.Client
    model: str = "gemini-2.5-flash",
    page_count: int = 0,         # recommend_params 자동 적용용
    batch_size: int | None = None,  # None이면 page_count로 자동
    overlap: int | None = None,
    parallel: int | None = None,
) -> list[dict]:                 # [{content, page, source_blocks}, ...]
```

### `ingestion_api.py` 변경 내용

**`/upload` 엔드포인트에 파라미터 추가**:
```python
chunking_method: str = Form("llm")   # "llm" | "rule"
```

**`process_and_ingest()` 분기**:

```
chunking_method == "llm"
  blocks → run_llm_chunking() → [{content, page}]
  → filter (toc / colophon / symbol / too_short / dedup)
  → embed → Supabase 저장
  (build_sections 불필요 — LLM이 의미 단위로 직접 분리)

chunking_method == "rule"  (기존 파이프라인 그대로)
  blocks → build_sections → chunk_blocks_aware → filter → embed → 저장
```

> section_title / section_path는 hierarchy tagging pass(Pass 3)에서 별도 부여하므로 LLM 청킹 경로에서는 빈 값으로 저장.

## domain_profiler + doc_metadata 통합 (2026-03-27)

### 배경

기존 domain_profiler는 QA 생성 job 시작 시마다 `analyze_domain()`을 LLM으로 호출.
같은 문서로 여러 job을 생성하면 동일 API 호출이 반복되는 비효율 존재.

### 변경 방향

`/analyze-hierarchy`(Pass 2) 실행 시 H1/H2/H3 master 생성과 동시에 `domain_profile`도 추출 → `doc_metadata` 테이블에 저장.
QA 생성 시 DB 조회로 재사용. LLM 호출 없이 즉시 사용 가능.

### 구현 내용

#### `backend/db/doc_metadata_repo.py` (신규)

| 함수 | 설명 |
|------|------|
| `upsert_doc_metadata(document_id, filename, domain_profile, h2_h3_master)` | ON CONFLICT (document_id) DO UPDATE — 재실행 시 덮어쓰기 |
| `get_doc_metadata(document_id)` | document_id 기반 조회 |
| `get_doc_metadata_by_filename(filename)` | filename 폴백 조회 (document_id 모를 때) |

#### `/analyze-hierarchy` 수정 (`ingestion_api.py`)

- LLM 프롬프트에 `domain_profile` 항목 추가 → hierarchy + domain 분석을 1회 LLM 호출로 처리
- `chunk_type_dist`는 실제 anchor 샘플 기반으로 덮어쓰기 (LLM 추정 대신)
- 분석 완료 후 `upsert_doc_metadata(document_id, filename, domain_profile, h2_h3_master)` 자동 저장
- `HierarchyAnalysisResponse`에 `domain_profile` 필드 추가 (프론트엔드에서 확인 가능)

```python
# domain_profile 구조 (LLM 반환, 5개 필드)
{
  "domain": "AI 데이터 구축 가이드라인",
  "domain_short": "AI 데이터",
  "target_audience": "데이터 구축 작업자",
  "key_terms": ["어노테이션", "품질관리", "데이터셋", "라벨링", "검수"],
  "tone": "기술 문서 격식체"
}
```

**제거된 필드 및 이유**

| 제거 필드 | 이유 |
|-----------|------|
| `main_topics` | `build_system_prompt` / `build_user_template` 어디서도 사용하지 않음 |
| `intent_hints` | chunk_type → intent 매핑은 문서마다 다르지 않음. LLM이 실제 intent 탁소노미(`fact\|purpose\|how\|condition\|comparison\|list`)와 다른 값을 생성하는 문제 있음. `build_user_template`의 기본값 폴백으로 대체 |
| `chunk_type_dist` | 실제 샘플 계산값을 LLM에 요청할 이유 없음. 필요 시 anchor 청크에서 직접 집계 가능 |

#### QA 생성 수정 (`generation_api.py`)

- job 시작 시 `doc_metadata`에서 `domain_profile` 먼저 조회 (document_id → filename 순)
- 캐시된 값 있으면 LLM 호출 없이 바로 사용
- `doc_metadata`에 없을 때만 `domain_profiler.analyze_domain()` 폴백 (LLM 호출)

```
/analyze-hierarchy 실행됨 → doc_metadata에 domain_profile 저장
  ↓
QA 생성 시 doc_metadata 조회 → 있으면 즉시 사용 (LLM 호출 0회)

/analyze-hierarchy 미실행 or doc_metadata 없음
  ↓
domain_profiler.analyze_domain() 호출 → LLM으로 직접 분석 (폴백)
```

**`domain_profiler.py` 역할 변경**: 정상 플로우에서는 호출되지 않는 폴백 전용 모듈로 유지. 완전 제거 시 `/analyze-hierarchy` 건너뛰는 케이스 대응 불가.

### doc_metadata 테이블 연결 현황

현재 Option A 적용 (논리적 연결, FK 없음):
- `document_id`: `doc_chunks.metadata->>'document_id'`와 일치하는 text 키
- DB 레벨 FK 불가 (doc_chunks의 document_id가 JSONB 내부에 있음)
- 장기 목표: Option B (DB 전면 재설계, doc_chunks에 전용 컬럼 추출 후 FK 연결) → 전체 파이프라인 안정화 이후 진행

> 상세 스키마 및 Option A/B 설계 비교: `docs/REF_DB.md` 참고

### 작업 현황

| 단계 | 상태 | 파일 |
|------|------|------|
| `db/doc_metadata_repo.py` 작성 | ✅ 완료 | `backend/db/doc_metadata_repo.py` |
