# DEV_260322 — Graph RAG 적용 설계

---

## 1. 배경 및 한계

현재 AutoEval은 청크 독립 가정 기반의 flat vector RAG 구조.
문서 간, 청크 간 의미적 관계가 없어 관계형 QA 생성 및 멀티홉 평가가 불가능한 구조적 한계 존재.

| 한계 | 영향 |
|------|------|
| 청크 각각이 독립적 — 청크 간 관계 없음 | 관계형 질문 생성 불가 |
| H1/H2/H3는 분류 체계, 의미적 관계 그래프 아님 | 멀티홉 컨텍스트 구성 불가 |
| Groundedness 평가가 단일 청크 기준 | 문서 내 모순 감지 전무 |

---

## 2. 핵심 결정 로그 (Decision Log)

설계 과정에서 내린 주요 판단과 근거.

| 결정 | 선택 | 근거 |
|------|------|------|
| 저장 방식 | `metadata.graph` JSONB 직접 확장 | POC 범위에서 Neo4j 등 별도 인프라 운영 비용이 이득보다 큼. `update_chunk_metadata()` 기존 경로 재활용으로 리스크 최소화 |
| 온톨로지 방식 | 문서 귀납형 동적 그래프 | 업로드 문서 도메인이 매번 다름. 사전 정의 온톨로지는 범용성 확보 불가 |
| Layer 4 적용 범위 | 조건부 레이어 | graph_claims 없는 QA에 중립값 부여 시 점수 왜곡. 기존 공식 하위 호환 유지가 더 안전 |
| `contradicts` 채우는 시점 | 3단계(평가 전 문서 단위 분석) | 청크 처리 시점(1단계)에는 다른 청크를 알 수 없음. 1단계 스키마에는 빈 배열로만 존재 |
| pairwise 모순 분석 | embedding 필터 후 LLM | 청크 N개 → N(N-1)/2 LLM 호출. 청크 50개 기준 1,225회. 고유사도 필터로 후보 쌍 좁힘 → 섹션 4 비용 절충 참고 |

## 3. 설계 상세

### 3-1. Graph RAG 삽입 위치

```
[현재]
Upload → Chunk → Embed → Hierarchy → Generate QA → Evaluate

[Graph RAG 삽입 후]
Upload → Chunk → Embed → Hierarchy
                              ↓
                    apply-granular-tagging
                    + entity/relation 추출   ← 프롬프트 확장
                              ↓
                    metadata.graph 저장
                              ↓
             Generate QA (그래프 컨텍스트 포함)
                              ↓
             Evaluate (+ Graph Consistency 조건부)
```

### 3-2. 저장 전략 — metadata JSONB 직접 확장

```python
# ingestion_api.py:602 — 기존 update_chunk_metadata() 패턴 재활용
meta = {
    **target.get("metadata", {}),
    "hierarchy_h1": ...,
    "hierarchy_h2": ...,
    "graph": {
        "entities": ["환불정책", "14일"],
        "relations": [["환불정책", "조건", "14일"]],
        "contradicts": []   # 1단계에서는 항상 빈 배열 — 3단계에서 채움
    }
}
```

`apply-granular-tagging` 응답 스키마 확장:
```json
// 현재
[{"idx": 0, "hierarchy": {"h1": "...", "h2": "...", "h3": "..."}}]

// 확장
[{
  "idx": 0,
  "hierarchy": {"h1": "...", "h2": "...", "h3": "..."},
  "graph": {
    "entities": ["환불정책", "14일"],
    "relations": [["환불정책", "조건", "14일"]],
    "contradicts": []
  }
}]
```

### 3-3. QA 생성 확장

신규 intent 유형:

| intent | 설명 | 생성 제약 |
|--------|------|-----------|
| `relational` | "A와 B의 관계는?" | `graph.relations`에 두 엔티티 간 경로 존재 시에만 |
| `comparative` | "X 방식과 Y 방식의 차이는?" | 동일 조건 |
| `causal_chain` | 연쇄 인과 관계 | 동일 조건 |

> 그래프 경로 없이 생성 시 평가 단계 근거 추적 불가 → Groundedness 점수 신뢰도 하락.

**이웃 청크 컨텍스트 토큰 상한 (prompts.py 설계 시 확정)**

depth=2 이웃 병합 시 컨텍스트 3~5배 증가. `claude-sonnet-4-6` workers=2로 가장 제약.

```
<context>{original_chunk}</context>
<graph_context max_tokens="{N}">     ← 원본 컨텍스트 토큰의 50% 이하 권고
  {neighbor_chunks_truncated}
</graph_context>
```

### 3-4. 평가 레이어 확장

```
[현재]
Layer 1-A: Syntax Validation      (0.10)
Layer 1-B: Dataset Statistics     (0.10)
Layer 2:   RAG Triad              (0.40)  ← Groundedness 조건부 분기 추가
Layer 3:   Quality                (0.40)

final_score = syntax×0.1 + stats×0.1 + rag×0.4 + quality×0.4

[추가]
Layer 4:   Graph Consistency (조건부)
  - 답변 내 엔티티가 그래프상 실제 연결되는가
  - 관계형 QA 답변이 그래프 경로와 일치하는가
  - 해당 청크에 contradicts 존재 여부 확인
```

**Layer 4 점수 공식 — 조건부 레이어**

```
graph_claims 있는 QA:
  final_score = syntax×0.1 + stats×0.1 + rag×0.35 + quality×0.35 + graph×0.10

graph_claims 없는 QA:
  final_score = syntax×0.1 + stats×0.1 + rag×0.4 + quality×0.4  (기존 유지)
```

`qa_eval_results.scores` JSONB 추가 필드:
```json
"graph": {"applicable": true, "score": 0.88}
```

## 4. 구현 로드맵

| 단계 | 내용 | 구현 위치 | 선행 조건 | 완료 기준 |
|------|------|-----------|-----------|-----------|
| **1단계** | `apply-granular-tagging` 프롬프트 확장 → `metadata.graph` 저장 (entities + relations, contradicts `[]` 고정) | `ingestion_api.py` | — | graph 블록 파싱 성공률(fallback 빈 배열 제외) ≥ 80% — 잠정치, 초기 테스트 실측 후 조정 |
| **1단계-B** | 기존 문서 전체 백필 (`graph` 없는 doc_chunks 재처리) | 일회성 마이그레이션 스크립트 | 1단계 배포 후 | 전체 doc_chunks 중 `metadata.graph` 존재 비율 = 100% |
| **2단계** | Groundedness 조건부 분기 — `graph_claims` 있으면 검증 전용 경로 | `rag_triad.py` | **1단계-B 완료 후 배포** | graph_claims 있는 QA에서 조건부 분기 진입 로그 확인 |
| **3단계** | 평가 전 문서 단위 모순 분석 → `metadata.graph.contradicts` 업데이트 | `evaluators/pipeline.py` 앞단 + `doc_chunk_repo.py` | 1단계-B 완료 | contradicts 비어있지 않은 청크 존재 시 평가 결과에 반영 확인 |
| **4단계** | 관계형 QA intent 추가 + 이웃 청크 컨텍스트 확장 | `generation_api.py` + `prompts.py` | 1단계-B 완료 | relational/comparative/causal_chain intent QA 생성 비율 > 0% |

> **2단계 주의**: `graph_claims 있으면` 분기는 1단계-B 완료 전까지 실제로 타지 않음. 배포 순서 반드시 준수.

### 3단계 쿼리 경로

```
평가 job 시작
  → qa_gen_results.id 로 source_doc(파일명) 조회
  → doc_chunk_repo.get_chunks_by_doc(doc_name)  ← 신규 함수 필요
      (또는 get_chunks_by_ids() + qa_list[*].docId 배열로 범위 한정)
  → pairwise 모순 분석 (아래 비용 절충안 적용)
  → metadata.graph.contradicts 업데이트
  → 평가 루프에서 각 QA context 청크 id로 contradicts 참조
```

**pairwise 분석 비용 절충**

```
청크 수 ≤ 30개  → 전체 pairwise LLM 호출 허용 (최대 435회)
청크 수 > 30개  → 고유사도 필터 선처리 필수
                   cosine similarity ≥ 0.75 인 쌍만 LLM 호출 대상
                   (의미적으로 유사한 청크 간에 모순 발생 가능성 높음)
```

## 6. 롤백 및 실패 처리

### 롤백 전략

**1단계-B 백필은 파괴적 작업** (`metadata` 직접 수정). 실행 전 확인 사항:

```
graph 키만 추가하는 방식이므로 기존 metadata 키는 보존됨.
원복 방법: UPDATE doc_chunks SET metadata = metadata - 'graph' WHERE ...
→ 별도 컬럼 백업 불필요, 키 삭제로 원복 가능.
```

백필 스크립트 필수 옵션:
- `--dry-run`: 실제 쓰기 없이 대상 청크 수 및 샘플 출력만
- `--doc-name`: 특정 문서만 선택 처리 (전체 일괄 전 검증용)

각 단계 배포 전 체크:
| 단계 | 롤백 방법 |
|------|-----------|
| 1단계 | 프롬프트 이전 버전 복구. 이미 저장된 `graph` 키는 무해함 |
| 1단계-B | `metadata - 'graph'` 쿼리로 전체 원복 |
| 2단계 | `rag_triad.py` 이전 커밋 복구. graph_claims 파라미터 제거 시 기존 동작 그대로 |
| 3단계 | `contradicts` 필드 무시하도록 pipeline 플래그 추가 권고 |
| 4단계 | `generation_api.py`·`prompts.py` 이전 커밋 복구. **단, 이미 생성된 relational/comparative/causal_chain intent QA는 DB에 잔존.** 롤백 후 해당 job의 관계형 intent QA는 평가에서 제외 또는 수동 삭제 처리 필요. 자동 처리 원할 경우 평가 파이프라인에 "graph_claims 없는 relational intent → 평가 skip" 방어 로직 사전 추가 권고. |

### 실패 처리

**1단계 — graph 블록 파싱 실패 시**

LLM이 graph 블록 누락 또는 잘못된 형식으로 반환할 경우:

```python
# fallback: 빈 구조로 저장 (재시도 없음)
# 데이터 불일치 방지를 위해 null 저장 금지
graph_fallback = {"entities": [], "relations": [], "contradicts": []}
```

판단 근거: graph 블록 파싱 실패 → `graph: null` 저장 시 2단계 조건부 분기에서
`graph_claims is None` 체크가 null과 []를 구분하지 못해 조용한 오류 발생 가능.
빈 배열 기본값이 가장 안전.

**2단계 — graph_claims 분기 오동작 시**

`graph_claims=[]`인 QA가 CoT Step 1 skip 경로로 잘못 진입하는 케이스.
`None`과 `[]` 모두 기존 CoT 경로로 처리되도록 방어 코드 필수:

```python
if not graph_claims:  # None과 [] 모두 처리
    # 기존 CoT 방식 그대로
```

배포 후 로그에서 `graph_claims=[]` QA가 skip 경로 진입 여부 확인.

**3단계 — pairwise 분석 중 LLM 오류 시**

해당 쌍 건너뜀, `contradicts`에 미추가. 평가 계속 진행.
오류율 ≥ 20% 시 해당 평가 job에 `graph_analysis_partial: true` 플래그 기록.

**4단계 — 경로 없는 relational intent 생성 시**

`graph.relations` 체크 없이 생성된 관계형 QA는 Layer 2 Groundedness에서
근거 추적 실패 → 낮은 점수로 집계. 데이터 오염은 아니나 점수 왜곡.
생성 시점 경로 체크 로직 누락 여부를 배포 전 단위 테스트로 검증 권고.
