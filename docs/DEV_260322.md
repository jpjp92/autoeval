# DEV_260322 — Graph RAG 적용 검토

## 논의 배경

현재 AutoEval은 청크 독립 가정 기반의 flat vector RAG 구조.
문서 간, 청크 간 의미적 관계가 없어 관계형 QA 생성 및 멀티홉 평가가 불가능한 구조적 한계 존재.

---

## 현재 구조의 한계

- 청크 각각이 독립적 → 청크 간 관계 없음
- H1/H2/H3는 분류 체계지 의미적 관계 그래프가 아님
- QA 생성이 단일 청크 컨텍스트에 한정 → 관계형 질문 생성 불가
- Groundedness 평가도 단일 청크 기준 → 멀티홉 사실 검증 불가

---

## Graph RAG 삽입 위치

```
[현재]
Upload → Chunk → Embed → Hierarchy → Generate QA → Evaluate

[Graph RAG 삽입 후]
Upload → Chunk → Embed → Hierarchy
                    ↓
              Entity/Relation 추출
                    ↓
              Knowledge Graph 구성
                    ↓
         Generate QA (그래프 컨텍스트 포함)
                    ↓
         Evaluate (+ Graph Consistency Layer)
```

## Ontology 방향

- 사전 정의 온톨로지는 AutoEval 목적에 과잉 (도메인이 문서마다 다름)
- **문서에서 귀납적으로 추출하는 경량 동적 그래프** 방식 채택 권고
- 현재 H1/H2/H3 계층이 묵시적 온톨로지 역할로 활용 가능

엔티티/관계 추출 예시:
```json
{
  "entities": ["환불 정책", "14일 이내", "영수증"],
  "relations": [
    ["환불 정책", "조건", "14일 이내"],
    ["환불 정책", "필요서류", "영수증"]
  ]
}
```

### metadata JSONB 직접 확장 (채택)

별도 테이블 없이 기존 `doc_chunks.metadata`에 `graph` 키 추가.
`update_chunk_metadata()`와 `apply-granular-tagging` 배치 패턴을 그대로 재활용 가능.

```python
# ingestion_api.py:602 — 기존 패턴 재활용
meta = {
    **target.get("metadata", {}),
    "hierarchy_h1": ...,
    "hierarchy_h2": ...,
    "graph": {          # ← 추가만 하면 됨
        "entities": ["환불정책", "14일"],
        "relations": [["환불정책", "조건", "14일"]],
        "contradicts": []   # 문서 내 모순 감지용
    }
}
update_chunk_metadata(target["id"], meta)
```

> 본격 운영 시 Neo4j / Apache AGE 전환 검토. POC는 JSONB로 충분.

## QA 생성 확장

- 그래프 이웃 청크(depth=2)를 컨텍스트로 병합 → 관계형 질문 생성

  - `relational` — "A와 B의 관계는?"

## 외부 피드백 검토 (2026-03-22)

### 동의 — 정확한 지적

**"H1/H2/H3 태깅이 이미 그래프 뼈대"**
3-Pass 계층 태깅 + metadata JSONB 자유 형식 → 별도 인프라 없이 graph 키 추가만으로 구현 가능. 초안의 별도 테이블 방식보다 훨씬 현실적.

**"`contradicts` 관계가 현재 4레이어에서 완전히 빠진 부분"**
현재 Groundedness는 답변이 컨텍스트에서 지지되는지만 평가. 컨텍스트 내부 모순 감지 로직은 전무. 평가 품질 갭이 가장 큰 지점.

**"Gap 1 — `apply-granular-tagging` 프롬프트 확장"**
현재 응답 스키마에 graph 블록 추가:
```json
// 현재
[{"idx": 0, "hierarchy": {"h1": "...", "h2": "...", "h3": "..."}}]

// 확장
[{
  "idx": 0,
  "hierarchy": {"h1": "...", "h2": "...", "h3": "..."},
  "graph": {
    "entities": ["환불정책", "14일"],
    "relations": [["환불정책", "조건", "14일"]],
    "contradicts": []
  }
}]
```

### 보완 — 실제 코드 기준 수정

**"Gap 2 — Groundedness에 graph.claims 커버리지 체크 추가"**

`rag_triad.py:179` Groundedness 프롬프트가 이미 CoT로 claims를 추론:
```
Step 1: IDENTIFY KEY CLAIMS
Step 2: FIND SUPPORTING EVIDENCE
```
graph.claims를 단순 추가하면 claims를 두 번 추출하는 구조가 됨.
올바른 방향은 **조건부 분기**:

```python
def evaluate_all_with_reasons(self, question, answer, context, graph_claims=None):
    # graph_claims 있으면 → Step 1 skip, 사전 추출 claims로 검증만
    # graph_claims 없으면 → 현재 CoT 방식 그대로 (하위 호환)
```

**"`contradicts` 감지의 구현 복잡도 과소평가"**

현재 평가 루프는 청크-QA 단위:
```
qa_list[i] → context(단일 청크) → RAG Triad
```
`contradicts` 감지는 청크A ↔ 청크B 간 비교가 필요 → **평가 시작 전 문서 단위 모순 분석 선행 단계** 필요. 청크별 루프 안에서 해결 불가.

## 단계별 구현 로드맵 (피드백 반영 수정)

| 단계 | 내용 | 구현 위치 |
|------|------|-----------|
| **1단계** | `apply-granular-tagging` 프롬프트 확장 → `metadata.graph` (entities + relations + contradicts) | `ingestion_api.py` 프롬프트 + 스키마 수정 |
| **2단계** | Groundedness 조건부 분기 — graph_claims 있으면 검증 전용 경로 | `rag_triad.py:evaluate_all_with_reasons()` |
| **3단계** | 평가 전 문서 단위 모순 분석 선행 단계 추가 | `evaluators/pipeline.py` 앞단 |
| **4단계** | 관계형 QA intent 추가 (`relational`, `comparative`, `causal_chain`) + 이웃 청크 컨텍스트 확장 | `generation_api.py` + `prompts.py` |

## 결론

- 온톨로지 선설계보다 **문서 귀납형 동적 그래프** 우선
- 저장: 별도 테이블 아닌 **`metadata.graph` JSONB 직접 확장** (기존 패턴 재활용)
- 1단계(`apply-granular-tagging` 프롬프트 확장)가 가장 낮은 비용으로 가장 큰 효과
- `contradicts` 감지는 청크 단위가 아닌 문서 단위 선행 분석 필요 → 3단계로 분리
