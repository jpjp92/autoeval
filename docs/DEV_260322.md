# DEV_260322 — Graph RAG 적용 검토

## 논의 배경

현재 AutoEval은 청크 독립 가정 기반의 flat vector RAG 구조.
문서 간, 청크 간 의미적 관계가 없어 관계형 QA 생성 및 멀티홉 평가가 불가능한 구조적 한계 존재.

---

## 현재 구조의 한계

- 청크 각각이 독립적 → 청크 간 관계 없음
- H1/H2/H3는 분류 체계지 의미적 관계 그래프가 아님
- QA 생성이 단일 청크 컨텍스트에 한정 → 관계형 질문 생성 불가
- Groundedness 평가도 단일 청크 기준 → 멀티홉 사실 검증 불가

---

## Graph RAG 삽입 위치

```
[현재]
Upload → Chunk → Embed → Hierarchy → Generate QA → Evaluate

[Graph RAG 삽입 후]
Upload → Chunk → Embed → Hierarchy
                    ↓
              Entity/Relation 추출
                    ↓
              Knowledge Graph 구성
                    ↓
         Generate QA (그래프 컨텍스트 포함)
                    ↓
         Evaluate (+ Graph Consistency Layer)
```

---

## Ontology 방향

- 사전 정의 온톨로지는 AutoEval 목적에 과잉 (도메인이 문서마다 다름)
- **문서에서 귀납적으로 추출하는 경량 동적 그래프** 방식 채택 권고
- 현재 H1/H2/H3 계층이 묵시적 온톨로지 역할로 활용 가능

엔티티/관계 추출 예시:
```json
{
  "entities": ["환불 정책", "14일 이내", "영수증"],
  "relations": [
    ["환불 정책", "조건", "14일 이내"],
    ["환불 정책", "필요서류", "영수증"]
  ]
}
```

## 저장 전략 (Supabase PostgreSQL)

```sql
-- 엔티티 노드
CREATE TABLE graph_entities (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  doc_name text,
  entity text,
  entity_type text,  -- concept / procedure / condition / value
  chunk_ids uuid[]
);

-- 관계 엣지
CREATE TABLE graph_relations (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  doc_name text,
  source_entity uuid REFERENCES graph_entities,
  relation text,
  target_entity uuid REFERENCES graph_relations,
  weight float  -- 등장 빈도 기반
);
```

> POC 수준: PostgreSQL 셀프조인으로 멀티홉 탐색 구현 가능.
> 본격 운영 시: Neo4j 또는 Apache AGE 전환 검토.

## QA 생성 확장

- 그래프 이웃 청크(depth=2)를 컨텍스트로 병합 → 관계형 질문 생성
- 신규 intent 유형 추가:
  - `relational` — "A와 B의 관계는?"
  - `comparative` — "X 방식과 Y 방식의 차이는?"
  - `causal_chain` — 연쇄 인과 관계

---

## 평가 레이어 확장안

```
[현재]
Layer 1-A: Syntax Validation
Layer 1-B: Dataset Statistics
Layer 2:   RAG Triad (relevance / groundedness / clarity)
Layer 3:   Quality (factuality / completeness / specificity / conciseness)

[추가 검토]
Layer 4:   Graph Consistency
  - 답변 내 엔티티가 그래프상 실제 연결되는가
  - 관계형 QA 답변이 그래프 경로와 일치하는가
```

---

## 단계별 구현 로드맵

| 단계 | 내용 | 구현 방식 |
|------|------|-----------|
| 1단계 | 엔티티 추출 → `doc_chunks.metadata.entities[]` 추가 | 현재 tagging 배치 패턴 재활용 |
| 2단계 | 관계 테이블 추가 + 이웃 청크 컨텍스트 확장 | graph_entities / graph_relations 테이블 |
| 3단계 | Graph Consistency 평가 레이어 추가 | evaluators/graph_consistency.py 신규 |
| 4단계 | Community Detection → 계층 요약 | H1/H2 분류 자동화 개선 |

1단계만 구현해도 `apply-granular-tagging`과 동일한 배치 패턴으로 바로 적용 가능.

## 결론

- 온톨로지 선설계보다 **문서 귀납형 동적 그래프** 우선
- 현재 LLM 배치 패턴(analyze-h2-h3 등) 재활용 → 아키텍처 변화 최소화
- 1단계(엔티티 추출) → 효과 검증 후 단계적 확장 권고
