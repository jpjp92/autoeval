## 1. 배경 및 한계

현재 AutoEval은 청크 독립 가정 기반의 flat vector RAG 구조.
문서 간, 청크 간 의미적 관계가 없어 관계형 QA 생성 및 멀티홉 평가가 불가능한 구조적 한계 존재.

| 한계 | 영향 |
|------|------|
| 청크 각각이 독립적 — 청크 간 관계 없음 | 관계형 질문 생성 불가 |
| H1/H2/H3는 분류 체계, 의미적 관계 그래프 아님 | 멀티홉 컨텍스트 구성 불가 |
| Groundedness 평가가 단일 청크 기준 | 문서 내 모순 감지 전무 |

## 3. 설계 상세

### 3-1. Graph RAG 삽입 위치

```
[현재]
Upload → Chunk → Embed → Hierarchy → Generate QA → Evaluate

[Graph RAG 삽입 후]
Upload → Chunk → Embed → Hierarchy
                              ↓
                    apply-granular-tagging
                    + entity/relation 추출   ← 프롬프트 확장
                              ↓
                    metadata.graph 저장
                              ↓
             Generate QA (그래프 컨텍스트 포함)
                              ↓
             Evaluate (+ Graph Consistency 조건부)
```

### 3-2. 저장 전략 — metadata JSONB 직접 확장

```python
# ingestion_api.py — 기존 update_chunk_metadata() 패턴 재활용
meta = {
    **target.get("metadata", {}),
    "hierarchy_h1": ...,
    "hierarchy_h2": ...,
    "graph": {
        "entities": ["환불정책", "14일"],
        "relations": [["환불정책", "조건", "14일"]],
        "contradicts": []   # 1단계에서는 항상 빈 배열 — 3단계에서 채움
    }
}
```

`apply-granular-tagging` 응답 스키마 확장:
```json
// 현재
[{"idx": 0, "hierarchy": {"h1": "...", "h2": "...", "h3": "..."}}]

// 확장
[{
  "idx": 0,
  "hierarchy": {"h1": "...", "h2": "...", "h3": "..."},
  "graph": {
    "entities": ["환불정책", "14일"],
    "relations": [["환불정책", "조건", "14일"]],
    "contradicts": []
  }
}]
```

### 3-3. QA 생성 확장

신규 intent 유형:

| intent | 설명 | 생성 제약 |
|--------|------|-----------|
| `relational` | "A와 B의 관계는?" | `graph.relations`에 두 엔티티 간 경로 존재 시에만 |
| `comparative` | "X 방식과 Y 방식의 차이는?" | 동일 조건 |
| `causal_chain` | 연쇄 인과 관계 | 동일 조건 |

> 그래프 경로 없이 생성 시 평가 단계 근거 추적 불가 → Groundedness 점수 신뢰도 하락.

**이웃 청크 컨텍스트 토큰 상한 (prompts.py 설계 시 확정)**

depth=2 이웃 병합 시 컨텍스트 3~5배 증가. `claude-sonnet-4-6` workers=2로 가장 제약.

```
<context>{original_chunk}</context>
<graph_context max_tokens="{N}">     ← 원본 컨텍스트 토큰의 50% 이하 권고
  {neighbor_chunks_truncated}
</graph_context>
```

### 3-4. 평가 레이어 확장

```
[현재]
Layer 1-A: Syntax Validation      (0.10)
Layer 1-B: Dataset Statistics     (0.10)
Layer 2:   RAG Triad              (0.40)  ← Groundedness 조건부 분기 추가
Layer 3:   Quality                (0.40)

final_score = syntax×0.1 + stats×0.1 + rag×0.4 + quality×0.4

[추가]
Layer 4:   Graph Consistency (조건부)
  - 답변 내 엔티티가 그래프상 실제 연결되는가
  - 관계형 QA 답변이 그래프 경로와 일치하는가
  - 해당 청크에 contradicts 존재 여부 확인
```

**Layer 4 점수 공식 — 조건부 레이어**

```
graph_claims 있는 QA:
  final_score = syntax×0.1 + stats×0.1 + rag×0.35 + quality×0.35 + graph×0.10

graph_claims 없는 QA:
  final_score = syntax×0.1 + stats×0.1 + rag×0.4 + quality×0.4  (기존 유지)
```

`qa_eval_results.scores` JSONB 추가 필드:
```json
"graph": {"applicable": true, "score": 0.88}
```

## 4. 구현 로드맵

| 단계 | 내용 | 구현 위치 | 선행 조건 | 실현가능성 | 완료 기준 |
|------|------|-----------|-----------|-----------|-----------|
| **1단계** | `apply-granular-tagging` 프롬프트 확장 → `metadata.graph` 저장 (entities + relations, contradicts `[]` 고정) | `ingestion_api.py` | — | ✅ 높음 | graph 블록 파싱 성공률(fallback 제외) ≥ 80% |
| **1단계-B** | 기존 문서 전체 백필 (`graph` 없는 doc_chunks 재처리) | 일회성 마이그레이션 스크립트 | 1단계 배포 후 | ✅ 높음 | 전체 doc_chunks 중 `metadata.graph` 존재 비율 = 100% |
| **2단계** | Groundedness 조건부 분기 — `graph_claims` 있으면 검증 전용 경로 | `rag_triad.py` | **1단계-B 완료 후 배포** | ✅ 높음 | graph_claims 있는 QA에서 조건부 분기 진입 로그 확인 |
| **3단계** | 평가 전 문서 단위 모순 분석 → `metadata.graph.contradicts` 업데이트 | `evaluators/pipeline.py` 앞단 | 1단계-B 완료 | ⚠️ 중간 | contradicts 비어있지 않은 청크 존재 시 평가 결과 반영 확인 |
| **4단계** | 관계형 QA intent 추가 + 이웃 청크 컨텍스트 확장 | `generation_api.py` + `prompts.py` | 1단계-B 완료 | ⚠️ 중간 | relational/comparative/causal_chain intent QA 생성 비율 > 0% |

> **2단계 주의**: `graph_claims 있으면` 분기는 1단계-B 완료 전까지 실제로 타지 않음. 배포 순서 반드시 준수.

### 3단계 쿼리 경로 (v2 업데이트)

```
평가 job 시작
  → qa_gen_results.id 로 source_doc(파일명) 조회
  → get_document_chunks(doc_name)   ← 기존 함수 재활용 (신규 함수 불필요)
  → 코사인 유사도 필터 (pgvector RPC)
      cosine_similarity ≥ 0.75 인 쌍만 후보 선별
  → 후보 쌍 유사도 내림차순 정렬 → 상위 50쌍만 LLM 호출 (절대 상한)
  → metadata.graph.contradicts 업데이트
  → 평가 루프에서 각 QA context 청크 id로 contradicts 참조
```

**pairwise 분석 비용 절충 (v2 — 상한 명시)**

```
청크 수 ≤ 30개  → 전체 pairwise 후보 생성 (최대 435쌍)
청크 수 > 30개  → 코사인 유사도 ≥ 0.75 필터 필수

공통 적용:
  후보 쌍 유사도 내림차순 정렬 → 상위 50쌍만 LLM 호출
  → Render free tier timeout/OOM 방지
  → 초과 쌍은 contradicts 미분석으로 처리 (평가 계속 진행)
```

## 6. 롤백 및 실패 처리

### 롤백 전략

**1단계-B 백필은 파괴적 작업** (`metadata` 직접 수정). 실행 전 확인 사항:

```
graph 키만 추가하는 방식이므로 기존 metadata 키는 보존됨.
원복 방법: UPDATE doc_chunks SET metadata = metadata - 'graph' WHERE ...
→ 별도 컬럼 백업 불필요, 키 삭제로 원복 가능.
```

백필 스크립트 필수 옵션:
- `--dry-run`: 실제 쓰기 없이 대상 청크 수 및 샘플 출력만
- `--doc-name`: 특정 문서만 선택 처리 (전체 일괄 전 검증용)

각 단계 롤백:

| 단계 | 롤백 방법 |
|------|-----------|
| 1단계 | 프롬프트 이전 버전 복구. 이미 저장된 `graph` 키는 무해함 |
| 1단계-B | `metadata - 'graph'` 쿼리로 전체 원복 |
| 2단계 | `rag_triad.py` 이전 커밋 복구. graph_claims 파라미터 제거 시 기존 동작 그대로 |
| 3단계 | `contradicts` 필드 무시하도록 pipeline 플래그 추가 권고 |
| 4단계 | `generation_api.py`·`prompts.py` 이전 커밋 복구. 단, 이미 생성된 relational/comparative/causal_chain QA는 DB 잔존 → 해당 job 관계형 QA 수동 삭제 또는 평가 skip 처리 |

### 실패 처리

**1단계 — graph 블록 파싱 실패 시**

```python
# fallback: 빈 구조로 저장 (재시도 없음, null 저장 금지)
graph_fallback = {"entities": [], "relations": [], "contradicts": []}
```

`graph: null` 저장 시 2단계 조건부 분기에서 `None`과 `[]` 구분 불가 → 조용한 오류 발생 가능. 빈 배열 기본값이 가장 안전.

**2단계 — graph_claims 분기 오동작 시**

```python
if not graph_claims:  # None과 [] 모두 기존 CoT 경로로 처리
    # 기존 CoT 방식 그대로
```

**3단계 — pairwise 분석 중 LLM 오류 시**

해당 쌍 건너뜀, `contradicts`에 미추가. 평가 계속 진행.
오류율 ≥ 20% 시 해당 평가 job에 `graph_analysis_partial: true` 플래그 기록.
50쌍 cap 초과분은 분석 없이 통과 (contradicts 미갱신).

**4단계 — 경로 없는 relational intent 생성 시**

`graph.relations` 체크 없이 생성된 관계형 QA는 Layer 2 Groundedness에서
근거 추적 실패 → 낮은 점수로 집계. 생성 시점 경로 체크 로직 배포 전 단위 테스트 필수.

## 7. 단계별 실현가능성 요약

| 단계 | 실현가능성 | 핵심 리스크 |
|------|-----------|------------|
| 1단계 프롬프트 확장 | ✅ 높음 | LLM graph JSON 파싱 실패율 (빈 배열 fallback으로 방어) |
| 1단계-B 백필 스크립트 | ✅ 높음 | LLM 호출 비용 (dry-run 사전 확인 가능) |
| 2단계 Groundedness 분기 | ✅ 높음 | 없음 |
| **3단계 모순 분석** | ⚠️ 중간 | 평가 지연 — 50회 cap + pgvector RPC 필터로 완화. RPC 설계가 선행 조건 |
| **4단계 관계형 QA** | ⚠️ 중간 | 토큰 비용 + Render 메모리 — free tier 환경 부담. 컨텍스트 토큰 상한 실측 후 결정 |
| Layer 4 조건부 점수 | ✅ 높음 | 가중치 확정만 필요 |
