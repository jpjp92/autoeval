# DEV_260324 — opendataloader-pdf 기반 청킹 & Hierarchy 구조화 도입 계획

> **배경**: 기존 PyMuPDF heuristic 파서의 heading 감지 한계를 보완하고,
> bounding box 기반 citation metadata 및 XY-Cut++ reading order를 활용해
> 청크 품질과 구조화 정확도를 높이기 위한 파이프라인 개선 계획.
>
> 설치 완료: `opendataloader-pdf==2.0.2` (2026-03-24, `uv add` 완료)
> Java 환경: OpenJDK 21.0.10 (2026-03-24, WSL2 `default-jre-headless` 설치 완료)

## 0. 실측 벤치마크 결과 (2026-03-24, 테스트데이터_1.pdf)

> 테스트 스크립트: `test/test_opendataloader_pdf.py`, `test/test_odl_noise.py`

### 0-1. PyMuPDF vs opendataloader-pdf 직접 비교

| 항목 | PyMuPDF | opendataloader-pdf |
|------|---------|-------------------|
| **파싱 속도** | **1.96s** | 3.75s (JVM cold start 포함) |
| **총 텍스트 추출량** | 10,861 chars | **11,430 chars** (+5%) |
| **Heading 감지** | **0개** (is_heading 플래그 없음) | **15개** (노이즈 필터 후) |
| **Table 감지** | 수동 `\|` 패턴 의존 | **7개 자동 분리** |
| **Header/Footer 분리** | heuristic 반복 탐지 | **9/9개 자동 분리** |
| **Caption 분리** | 수동 `[표]\n` 태깅 | **7개 자동 분리** |
| **Bounding box** | x0/y0/x1/y1 (부분적) | **116/116 100% 전 요소** |

> **중요**: PyMuPDF heading 0개 확인 — `is_heading` 플래그를 세팅하지 않는 구조.
> `build_sections()`의 섹션 분류(7개)는 폰트 크기 추정 heuristic에만 의존하고 있었음.

> **텍스트 추출량 오해 주의**: opendataloader는 `list`, `table`, `header`의 텍스트가
> 중첩 필드(`list items`, `rows` 등)에 저장됨. top-level `content`만 합산하면 2,461 chars로
> 보이지만 재귀 추출 시 11,430 chars — PyMuPDF와 동등한 수준.

### 0-3. 노이즈 이슈 실측 (test_odl_noise.py)

heading 32개 중 **17개(53%)가 노이즈** → 필터 후 유효 heading 15개

| 노이즈 유형 | 수 | 패턴 | 필터 정규식 |
|-----------|-----|------|-----------|
| **NOISE 1** TOC 리더 | 4개 | `·····` 포함 | `r'[·.…]{4,}'` |
| **NOISE 2** 불릿 오분류 | 9개 | `l ` 접두사 (Level 8~9) | `r'^l\s'` |
| **NOISE 3** 캡션 오분류 | 2개 | `<표 / <그림` 접두사 (Level 11) | `r'^<(표\|그림)'` |
| **NOISE 4** 태그 레이블 | 2개 | `[LLM]` 패턴 (Level 10) | `r'^\[[A-Z가-힣]{1,10}\]$'` |

필터 후 heading level 분포: L7~L11 전부 제거 → L1~L6 유효 heading만 잔존

### 0-4. 페이지 번호 추출 비교

| 항목 | PyMuPDF | opendataloader |
|------|---------|---------------|
| 인식 페이지 수 | **12페이지** | **15페이지 전부** |
| `page number` 필드 | 블록마다 존재, 불일치 없음 | 116/116 100% 모든 요소 |
| 이미지 전용 페이지 (p1, p2, p5) | ❌ 텍스트 없으면 건너뜀 | ✅ bounding box + source로 정확히 기록 |

> **PyMuPDF 누락 페이지**: p1(표지), p2(목차 이미지), p5(구분 페이지) — 텍스트가 없는 이미지 전용 페이지를 건너뜀.
> 현재 청크 metadata의 `page` 값이 실제 PDF 페이지와 어긋날 수 있음.
> opendataloader 교체 시 모든 페이지의 정확한 `page number` 보장 → citation URL 정확도 향상.

### 0-5. 페이지별 텍스트 커버리지 이상 페이지

| 페이지 | PyMuPDF | opendataloader | 비율 | 비고 |
|-------|---------|---------------|------|------|
| p1, p2, p5 | 0 chars (누락) | 요소 있음 | ∞ | 이미지 전용 페이지 — PyMuPDF 건너뜀 |
| p11 | 1,626 chars | 590 chars | 36% | ⚠️ 전체 목차 표 페이지 — 추가 확인 필요 |
| p12 | 515 chars | 980 chars | 190% | opendataloader가 더 많이 추출 |

### 0-6. 핵심 확인 사항

- **텍스트 커버리지 동등** — 재귀 추출 기준 PyMuPDF 대비 105%, 실질적 손실 없음
- **페이지 인식 우위** — 15/15 전 페이지 인식 (PyMuPDF는 이미지 전용 페이지 12/15만 인식)
- **구조 인식 압도적 우위** — heading/table/caption 자동 분리로 수동 처리 대부분 대체 가능
- **bounding box 100%** — citation metadata 구현 토대 확보
- **p11 이상 현상** 추가 확인 필요 (목차 전체가 표로 인식되어 `table`에 흡수된 것으로 추정)

## 1. 현황 및 문제 인식

### 현재 파이프라인 요약

```
PDF bytes
  └─ extract_text_by_page (PyMuPDF fitz)
       └─ block 추출 — x0/y0 좌표 있으나 heading_level 감지 없음
  └─ merge_adjacent_short_blocks
  └─ build_sections (heuristic: 폰트 크기 추정 기반 heading 분류)
  └─ RecursiveCharacterTextSplitter (고정 크기 분할)
  └─ 필터링 (TOC/colophon/symbol/중복)
  └─ LLM 3-pass Hierarchy Tagging
       1. analyze-hierarchy  → H1 master 후보 도출 (Gemini)
       2. analyze-h2-h3      → H2/H3 master 생성
       3. apply-granular-tagging → 청크별 H1/H2/H3 일괄 적용 (5-concurrency)
  └─ Gemini Embedding → Supabase doc_chunks
```

### 핵심 문제

| 문제 | 영향 |
|------|------|
| `build_sections`의 heading 감지가 heuristic (폰트 크기 추정) | 멀티컬럼/표 많은 PDF에서 section 경계 오탐 |
| 고정 크기 RecursiveCharacterTextSplitter | 의미 단위를 무시한 중간 절단 발생 |
| bounding box가 청크 metadata에 미저장 | 출처 추적(citation) 기능 구현 불가 |
| 멀티컬럼 레이아웃 reading order 미보장 | 컬럼 경계 오인식 → 청크 품질 저하 |

## 2. opendataloader-pdf 핵심 차별점

| 기능 | 설명 | 벤치마크 |
|------|------|---------|
| **XY-Cut++ reading order** | 멀티컬럼 레이아웃 정확 인식 | — |
| **MHS (Hierarchical Reasoning Model)** | heading level 자동 감지 | 76~83% |
| **TEDS (Table Extraction)** | 표 추출 정확도 | heuristic 49% / hybrid 93% |
| **JSON 구조화 출력** | type, heading_level, bounding_box, page, id 포함 | — |
| **전체 벤치마크** | layout 분석 종합 | heuristic 72% / hybrid 90% |
| **GPU 불필요** | heuristic 모드는 CPU only | Render free tier 호환 |

### 두 Hierarchy 개념 구분 (중요)

| 구분 | 현재 시스템 H1/H2/H3 | opendataloader-pdf heading_level |
|------|----------------------|----------------------------------|
| **종류** | Semantic (내용 카테고리) | Structural (문서 레이아웃 수준) |
| **예시** | H1: "서비스소개", H2: "요금정책" | H1: "1장 개요", H2: "1.2 배경" |
| **용도** | QA stratification 필터 | 청크 경계 결정 |
| **부여 주체** | LLM (Gemini) | opendataloader 자동 감지 |

→ **대체 관계 아님 — 보완 관계**: structural heading으로 청크를 자르고, semantic H1/H2/H3로 분류

### 1단계 — 파서 교체 (즉시 적용 가능)

**목표**: PyMuPDF → opendataloader-pdf 교체. 기존 LLM 3-pass·DB 스키마·임베딩 파이프라인은 전혀 건드리지 않음.

#### 변경 파일

| 파일 | 작업 |
|------|------|
| `backend/Dockerfile` | `default-jre-headless` 추가 |
| `backend/ingestion/parsers.py` | `extract_text_by_page()` 교체 + 어댑터 구현 |
| `backend/api/ingestion_api.py` | chunk metadata에 `bounding_box`, `element_id`, `odl_element_type` 추가 |

#### parsers.py 어댑터 설계

```
opendataloader.convert(pdf, format="json,markdown") 호출
  └─ JSON: data["kids"] 116개 요소 플랫 리스트
  └─ Markdown: 표 처리용

요소 순회 → 타입별 분기:

  ✅ 임베딩 대상
    heading (유효)  → 청크 경계 마커 + content 추출
    paragraph       → content 직접 추출
    list            → list items 재귀 추출
    table           → Markdown 출력 파싱 + [표]\n 접두사
    caption         → content 추출
    text block      → content 추출

  ❌ 스킵 (임베딩 제외)
    image           → content 없음 (좌표+파일경로만)
    header          → 페이지 헤더 반복 노이즈
    footer          → 페이지 푸터 반복 노이즈
    heading (노이즈 4종) → 아래 필터 적용

노이즈 필터 적용 순서 (test_odl_noise.py 확정 패턴):
  전제: el["type"] == "heading" 인 요소만 대상

  순서 1: NOISE 1 r'[·.…]{4,}'           → TOC 리더 제거 (우선 순위 최고 — 콘텐츠 오염 방지)
  순서 2: NOISE 2 r'^l\s'                → 불릿 오분류 제거 (Level 8~9 집중)
  순서 3: NOISE 3 r'^<(표|그림)'          → 캡션 오분류 제거 (Level 11 집중)
  순서 4: NOISE 4 r'^\[[A-Z가-힣]{1,10}\]$' → 태그 레이블 제거 (Level 10 집중)

  → 4개 필터 모두 OR 조건으로 병렬 적용 가능 (상호 배타적 패턴)
```

#### 표 처리 변경

```python
# 현재: format_table_as_text() 400줄 rowspan/colspan 수동 처리
# 변경: opendataloader Markdown 출력 파싱 (자동 처리됨)

def parse_table_from_markdown(md_text: str) -> str:
    cleaned = re.sub(r'<br\s*/?>', ' ', md_text)  # <br> 제거
    return "[표]\n" + cleaned
```

#### heading level 필드 선택 기준

opendataloader JSON 요소에는 heading 관련 필드가 두 개 존재:

| 필드 | 타입 | 예시 | 용도 |
|------|------|------|------|
| `heading level` | int | `1`, `3`, `8` | 수치 비교·정렬·필터에 사용 → **우선 사용** |
| `level` | str | `"Doctitle"`, `"Subtitle"` | 명칭 레이블 — 보조 정보로만 활용 |

```python
# 1단계 어댑터 구현 시
section_level = el.get("heading level")  # int, 없으면 None
level_label   = el.get("level", "")     # str, 노이즈 필터 후 로깅용
```

### 2단계 — Structural Heading 기반 section_path 교체 (1단계 검증 후)

**목표**: `build_sections()` heuristic 제거. opendataloader heading tree를 section 경계로 직접 활용.

#### 변경 내용

- `parsers.py` `build_sections()` → opendataloader `heading level` 기반 tree 빌드로 교체
- `ingestion_api.py` section_path 생성 로직 교체
- metadata에 `structural_path` 신규 필드 추가

```json
{
  "structural_path": "제1장 > 작성 배경 및 목적 > 1 작성 배경",
  "section_path": "제1장 > 작성 배경 및 목적",
  "hierarchy_h1": "서비스소개",
  "bounding_box": [86.8, 487.0, 530.5, 500.2],
  "page": 6
}
```

#### 기대 효과

- section 경계가 문서 실제 구조와 일치 → 청크 의미 단위 품질 향상
- LLM 3-pass tagging 시 `structural_path` 컨텍스트 활용 → H1/H2/H3 분류 정확도 향상
- RAG retrieval 시 structural_path 기반 필터링 가능

### 3단계 — Semantic Chunking 전환 (장기)

**목표**: 고정 크기 `RecursiveCharacterTextSplitter` → heading-aware semantic chunking.

- heading 만나면 이전 청크 저장, 새 청크 시작
- max_tokens 초과 시에만 추가 분할
- 청크마다 `heading_path` (상위 heading 경로) 자동 포함

**조건**: 2단계 structural heading 정확도 실측 후 결정.

### 4단계 — LLM 3-pass 프롬프트 최적화 (장기)

**목표**: `analyze-hierarchy`, `apply-granular-tagging` 프롬프트에 heading tree 컨텍스트 추가.

- 현재: 청크 샘플 텍스트만 LLM 입력 → H1 master 후보가 텍스트 내용에만 의존
- 개선: `structural_path` (heading tree) + 청크 텍스트 함께 입력 → **H1 master 생성 품질 향상**
- 주의: 호출 수는 동일하거나 증가 가능 (청크당 토큰 증가). **"호출 감소" 아님 — 품질 향상이 목표**
- 입력 토큰 증가분(structural_path 추가)이 H1 분류 오류 감소로 충분히 상쇄되는지 실측 필요

**조건**: 3단계 완료 후 H1 master 분류 정확도 지표로 효과 검증 (비용 증가 허용 여부 선결정).

## 3-B. 표(Table) 처리 방식 비교

### 현재 PyMuPDF 방식

```
page.find_tables()
  └─ format_table_as_text()
       └─ rowspan/colspan forward-fill 수동 처리
       └─ "[표]\n| col1 | col2 |\n|---|---|\n| ... |" 형태로 변환
       └─ chunk_type: "table", is_table: True 블록으로 저장
```

핵심 처리 로직: `parsers.py` `format_table_as_text()` — 400줄 규모의 rowspan/colspan 처리, 구조 기반 제목 테이블 감지 포함.

### opendataloader-pdf 방식

**JSON 구조**:
```json
{
  "type": "table",
  "number of rows": 6, "number of columns": 4,
  "rows": [
    { "cells": [
        { "row span": 1, "column span": 1,
          "kids": [{ "type": "paragraph", "content": "연번" }]
        }
    ]}
  ]
}
```
- `row_span`, `column_span` 자동 감지
- 셀마다 `bounding box` + `page number` 포함
- 셀 내용은 `kids` 하위 paragraph/list에 저장 (재귀 추출 필요)

**Markdown 출력** (`format="markdown"` 사용 시):
```markdown
|연번|버전|발간일|주요 내용|
|---|---|---|---|
|1|v1.0|'21.02|o 최초 발간<br><br>- 인공지능 학습용 ...|
```
- 멀티라인 셀 내용은 `<br>` 태그로 처리
- rowspan/colspan 자동 처리됨
- 이미 사용 가능한 Markdown 형태

### JSON 재귀 추출 vs Markdown 출력 비교

| 항목 | JSON 재귀 추출 | Markdown 출력 |
|------|--------------|--------------|
| 셀 내 다중 항목 | 단순 공백 join → **내용 손실 가능** | `<br>` 보존 → **완전 추출** |
| rowspan/colspan | 수동 처리 필요 | 자동 처리됨 |
| 코드 복잡도 | 재귀 추출 함수 직접 구현 | 파일 읽기만 |
| `[표]` 접두사 | 수동 추가 | 수동 추가 |

> **실측 확인**: JSON 재귀 추출 시 셀 내 `list` type 하위 항목이 누락됨 (표 2, 3의 `세부목차` 컬럼 공백).
> Markdown 출력은 전체 내용 보존.

### 결론: Markdown 출력 사용

1단계 구현에서 표 처리는 **`format="markdown"` 출력을 직접 파싱** 하는 방식 채택.
- `<br>` 태그 → 공백 또는 `\n`으로 변환
- `[표]\n` 접두사 추가 (기존 `chunk_type: table` 관례 유지)
- `format_table_as_text()` 400줄 로직 제거 가능

```python
# 1단계 어댑터 pseudo-code
def parse_markdown_table(md_table_text: str) -> str:
    """opendataloader Markdown 표 → 기존 [표]\n 형식으로 변환."""
    cleaned = re.sub(r'<br\s*/?>', ' ', md_table_text)   # <br> 제거
    return "[표]\n" + cleaned
```

## 3-A. DB 저장 구조 변화 검토

### 결론: DB 스키마(DDL) 변경 없음

`doc_chunks` 테이블의 컬럼 구조(`content`, `embedding vector(1536)`, `metadata jsonb`, `created_at`)는 그대로 유지.
신규 필드는 모두 기존 `metadata JSONB`에 추가하면 되므로 마이그레이션 불필요.

### 현재 chunk_metadata 키 vs 1단계 후 변화

| metadata 키 | 현재 값 (PyMuPDF) | 1단계 후 (opendataloader) | 변화 |
|------------|-----------------|--------------------------|------|
| `page` | PyMuPDF 기준 12페이지만 | 15페이지 전부 정확한 번호 | **값 정확도 향상** |
| `section_title` | heuristic 폰트 추정 제목 | opendataloader heading content | **값 품질 향상** |
| `section_path` | heuristic 경로 | opendataloader heading tree 기반 | **값 품질 향상** |
| `section_level` | heuristic 추정 레벨 | opendataloader `heading level` int (int 타입 필드 우선, `level` str 필드는 보조) | **값 품질 향상** |
| `chunk_type` | `detect_chunk_type()` 텍스트 패턴 | opendataloader type 직접 매핑 | **값 품질 향상** |
| `content_hash` | SHA-1 | 동일 | 변경 없음 |
| `keywords` | `extract_keywords()` | 동일 (텍스트 기반) | 변경 없음 |
| `hierarchy_h1/h2/h3` | LLM 3-pass 태깅 | 동일 | 변경 없음 |
| `embedding_model` | `gemini-embedding-2-preview` | 동일 | 변경 없음 |
| `ingested_at`, `filename` 등 | — | 동일 | 변경 없음 |
| **`bounding_box`** | ❌ 없음 | **✅ 신규 추가** `[x1, y1, x2, y2]` | **신규** |
| **`element_id`** | ❌ 없음 | **✅ 신규 추가** opendataloader 요소 ID | **신규** |
| **`odl_element_type`** | ❌ 없음 | **✅ 신규 추가** `paragraph`/`list`/`table` 등 원본 type | **신규** |

### chunk_type 매핑표

현재 `detect_chunk_type()`의 텍스트 패턴 → opendataloader type 직접 매핑으로 교체.

| opendataloader type | → chunk_type 값 | 비고 |
|--------------------|----------------|------|
| `heading` (유효) | `heading` | 노이즈 4종 제외 후 |
| `paragraph` | `body` | |
| `list` | `list` | |
| `table` | `table` | |
| `caption` | `body` | 표/그림 설명은 본문으로 분류 |
| `text block` | `body` | |

### 기존 청크와의 호환성

- 기존 저장된 청크: `bounding_box`, `element_id`, `odl_element_type` 키 없음 → JSONB 특성상 쿼리 시 `NULL`로 처리되므로 **기존 QA 생성·평가 파이프라인에 영향 없음**

## 6. Docker / 배포 환경 구성

### 6-1. backend/Dockerfile 변경 사항

opendataloader-pdf는 내부적으로 JVM 프로세스를 생성하므로 컨테이너에 JRE가 필요.
현재 베이스 이미지 `python:3.12-slim`에는 JVM 미포함.

```dockerfile
# 현재
RUN apt-get update && apt-get install -y --no-install-recommends curl tzdata \
    && rm -rf /var/lib/apt/lists/*

# 변경 후 — default-jre-headless 추가
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl tzdata default-jre-headless \
    && rm -rf /var/lib/apt/lists/*
```

변경 범위: `backend/Dockerfile` 6번째 줄 `apt-get install` 목록에 `default-jre-headless` 추가만으로 완료.

### 6-2. 이미지 크기 영향

| 구성 | 추가 용량 | 비고 |
|------|---------|------|
| `default-jre-headless` (OpenJDK 21) | ~200MB | Debian 기본 JRE |
| `temurin-17-jre-headless` (Eclipse Temurin) | ~150MB | 경량 대안, 별도 apt 저장소 등록 필요 |

→ **1단계에서는 `default-jre-headless` 사용** (설정 단순, 추가 저장소 불필요).
이미지 크기 민감해지면 Temurin 17로 전환 검토.

> Render free tier 이미지 크기 제한 없음. 단 빌드 시간 증가(~1~2분) 및 첫 배포 cold start 영향 있음.

### 6-3. Render 배포 시 주의사항

| 항목 | 내용 |
|------|------|
| **JVM cold start** | 컨테이너 기동 후 첫 `convert()` 호출 시 ~3.75s 소요 (이후 배치 호출은 파일당 ~0.05s) |
| **Render cold start** | 서비스 슬립 후 재기동 ~30s — JVM cold start와 별개 |
| **메모리** | JVM 기본 힙 ~256MB + Python 프로세스 ~150MB = 총 ~400MB 예상. Render free tier 512MB 제한 — **1단계 배포 직후 Render dashboard → Metrics 탭에서 실측값 확인 필수** (완료 조건에 포함) |
| **로컬 개발 (WSL2)** | `sudo apt-get install -y default-jre-headless` 완료 (2026-03-24) |

### 6-4. JVM 재사용 전략 요약

```
단건 업로드 (현재 패턴)
  └─ 파일 1개 → convert([file]) → JVM 기동 ~3.75s → 결과 반환
     허용 가능: 업로드 빈도 낮음, background task로 사용자 대기 없음

다수 파일 동시 업로드 시 (향후 개선)
  └─ 파일 N개 큐 → convert([f1, f2, ..., fN]) → JVM 1회 기동 → 파일당 ~0.05s
     옵션 B (큐 배치) 전환 시점: 동시 업로드 3건 이상 빈번해질 때
```
