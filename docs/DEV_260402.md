## 5. LLM 청킹 품질 강화 (`llm_chunker.py`)

### 5.1 무한루프 버그 수정 (`merge_short_chunks`)

**원인**: `else: i += 1` (단독 짧은 청크) 이후에도 `changed = True` 실행 → 앞뒤가 없는 짧은 청크 하나가 남으면 while 루프 무한 반복.

**수정**: `else` 브랜치에서 `continue`로 `changed = True` 우회.

```python
else:
    i += 1
    continue  # changed 변경 없이 루프 유지
changed = True
```

### 5.2 프롬프트 — 의미 기반 청킹 규칙 추가 (PDF + DOCX 동일 적용)

`<chunking_rules>`에 **"주제 전환 기준"** 섹션 추가. 크기 기준보다 의미적 완결성 우선.

- 하나의 청크 = 하나의 완결된 주제·논점·사례
- 새로운 개념/주장/사례 시작 시 새 청크
- 접속사 없이 주제 전환되는 지점을 경계로 사용
- 맥락상 마무리(결론·요약·효과) 후 새 시작 지점을 경계로 사용
- 의미 완결성 우선, 800자 상한은 반드시 준수

### 5.3 중복 감지 텍스트 기반으로 강화 (`deduplicate_chunks`)

**기존**: `source_blocks` 인덱스 Jaccard만 사용 → 배치 경계를 넘는 텍스트 내용 중복 미감지.

**수정**: `_text_jaccard()` 추가, source_blocks OR 텍스트 유사도 중 하나라도 threshold(0.5) 초과 시 중복 처리.

```python
def _text_jaccard(a: str, b: str) -> float:
    sa, sb = set(a.split()), set(b.split())
    return len(sa & sb) / len(sa | sb)
```

### 5.4 `trim_dangling_conjunctions` — 마지막 청크 내용 유실 방지

**기존**: 마지막 청크 끝 접속사 감지 시 다음 청크가 없어 그냥 잘라버림 → 내용 손실.

**수정**: 마지막 청크는 `continue`로 건너뛰어 원본 유지.

### 5.5 `merge_short_chunks` min_chars 기준 통일

**기존**: 기본값 100자 — 프롬프트 기준 200자와 불일치.

**수정**: `min_chars=200` — 프롬프트 최소 청크 크기와 동일하게 통일.

## 6. DB 저장 안정성 강화 (`doc_chunk_repo.py`)

### 6.1 `save_doc_chunks_batch` — 세마포어 + 지수 백오프 재시도

**원인**: 대용량 문서 처리 시 5개 embed 배치가 동시에 Supabase INSERT 시도 → HTTP/2 커넥션 풀 초과 → `Server disconnected` (4/5 배치 실패). `update_chunk_metadata`는 `_write_sem()`을 쓰는데 이 함수만 누락되어 있었음.

**수정**:

- `_write_sem()` (max 2 동시) 세마포어 적용
- 실패 시 지수 백오프 재시도 (최대 3회: 1s → 2s → 4s)

```python
async with _write_sem():
    for attempt in range(retries):
        try:
            ...
            return result
        except Exception as e:
            if attempt < retries - 1:
                await asyncio.sleep(2 ** attempt)
            else:
                logger.error(...)
                return []
```

## 24. 빈 QA DB 저장 문제 — 원인 분석 및 개선 방향

### 24.1 현상

H2/H3 hierarchy를 선택해 QA 생성을 시작했을 때, 해당 노드의 컨텍스트가 너무 적으면 LLM이 빈 `qa_list`를 반환한다. 그런데도 `total_qa=0`인 빈 레코드가 DB에 그대로 저장되는 문제 발생.

### 24.2 Root Cause 분석

#### Layer 1 — `hierarchy_repo.py` 필터링 불충분

```python
MIN_CHUNKS_FOR_QA = 2  # 청크 개수만 검증
```

- `get_hierarchy_list()`에서 청크 수가 2 미만인 노드만 드롭다운에서 제외
- **문제**: 청크 수는 충족해도 각 청크의 실제 텍스트가 매우 짧을 수 있음
  - 예) 청크 2개 × 각 50자 = 총 100자 → 필터 통과하지만 의미 있는 QA 생성 불가
- 콘텐츠 길이·품질에 대한 검증 없음

#### Layer 2 — `generation_api.py` 저장 가드 없음

```python
total_qa = sum(len(r.get("qa_list", [])) for r in results)
# ↓ total_qa == 0 여도 저장 로직 진입
supabase_id = await save_qa_generation_to_supabase(...)
```

- `total_qa`가 0이어도 `save_qa_generation_to_supabase()` 가 무조건 호출됨
- 빈 `qa_set` 레코드가 Supabase에 저장되어 히스토리에 노출

### 24.3 개선 방향

#### Fix 1 — `hierarchy_repo.py`: 최소 콘텐츠 길이 조건 추가

```python
MIN_CHUNKS_FOR_QA   = 2
MIN_CONTENT_CHARS   = 300   # 노드 내 총 텍스트 300자 미만이면 드롭다운 제외
```

- 청크를 불러올 때 `content` 길이 합산
- `total_chars < MIN_CONTENT_CHARS` 이면 해당 H2/H3 노드 제외
- 두 조건 모두 충족해야 드롭다운 노출: `chunk_count >= 2 AND total_chars >= 300`

#### Fix 2 — `generation_api.py`: 빈 QA 저장 방지 가드

```python
if total_qa == 0:
    await update_job_status(job_id, "FAILED",
        message="선택한 계층의 컨텍스트가 부족하여 QA를 생성할 수 없습니다.")
    return  # DB 저장 건너뜀
```

- `total_qa == 0` 시 job 상태를 `FAILED`로 업데이트하고 저장 로직 건너뜀
- 프론트에서 FAILED 상태를 수신하면 "컨텍스트 부족" 안내 메시지 표시

### 24.4 적용 파일

| 파일 | 수정 내용 |
| ---- | --------- |
| `backend/db/hierarchy_repo.py` | `MIN_CONTENT_CHARS` 상수 추가, `get_hierarchy_list()` 에서 콘텐츠 길이 합산 필터 적용 |
| `backend/api/generation_api.py` | `total_qa == 0` 가드 추가, FAILED 상태 설정 후 early return |

### 24.5 구현 내용

#### `backend/db/hierarchy_repo.py`

```python
MIN_CHUNKS_FOR_QA = 2
MIN_CONTENT_CHARS = 300  # 노드 내 총 텍스트 300자 미만이면 드롭다운 제외
```

- 청크 순회 시 `h3_chars`, `h2_chars` 딕셔너리에 `len(content)` 누산
- 필터 조건을 이중화: `cnt < MIN_CHUNKS_FOR_QA OR chars < MIN_CONTENT_CHARS` → H2/H3 제외
- 로그 메시지에 chars 값 포함하여 디버깅 용이성 확보

#### `backend/api/generation_api.py`

```python
if total_qa == 0:
    msg = "선택한 계층의 컨텍스트가 부족하여 QA를 생성할 수 없습니다."
    job_manager.update_job(job_id, status=JobStatus.FAILED, progress=0, error=msg, message=msg)
    logger.warning(f"QA generation produced 0 results — skipping DB save (job_id={job_id})")
    return
```

- `save_qa_generation_to_supabase()` 호출 직전에 `total_qa == 0` 체크
- 조건 충족 시 job을 `FAILED` 상태로 업데이트하고 early return (DB 저장 건너뜀)
- 프론트에서 FAILED 상태 수신 시 "컨텍스트 부족" 메시지 노출
