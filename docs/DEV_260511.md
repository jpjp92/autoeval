# L1-B Statistical Pre-screening 재정의 최종안 — 2026-05-11

`DEV_260508.md`의 L1-B 메트릭 재설계 논의가 길어져, 현재 프로젝트에 적용할 최종 결론만 분리 정리한다.

---

## 배경

기존 2026-05-08 설계는 reference-free 통계 지표 후보군으로는 타당했지만, 현재 프로젝트의 QA 생성 방식에 그대로 적용하면 운영 의도와 어긋나는 부분이 있었다.

현재 QA 생성은 일반 생성 텍스트가 아니라 다음 특성을 가진다.

- intent가 6종 고정 스키마로 통제됨
- 질문은 단일 차원(single-scope)으로 생성됨
- 답변은 context의 복수 항목을 빠짐없이 포함하도록 유도됨
- QA마다 `docId`가 연결된 문서 근거형 데이터임
- 개별 batch가 단일 문서 또는 특정 hierarchy 범위로 생성될 수 있음

따라서 L1-B는 "균등할수록 좋다" 또는 "context와 많이 겹칠수록 좋다"가 아니라, "현재 생성 정책을 잘 따르고 있는가"와 "dataset-level 반복/편중 문제가 없는가"를 보는 방향으로 재정의한다.

---

## LLM Judge 4개 지표와의 분리

현행 LLM Judge는 개별 QA 의미 품질을 본다.

| LLM Judge | 측정 단위 | 실제 측정 내용 |
|-----------|----------|---------------|
| Answer Relevance | 개별 QA | 질문의 모든 요구를 답변이 다루는가 |
| Groundedness | 개별 QA | 답변의 claim이 context에 의미적으로 지지되는가 |
| Context Relevance | 개별 QA | context가 질문에 답할 정보를 담고 있는가 |
| Completeness | 개별 QA | 원자 하위 질문 분해 시 빠진 답변이 없는가 |

L1-B는 위 의미 품질을 다시 측정하지 않는다. L1-B는 dataset-level 통계, 정책 준수, 반복, 소스 편중만 본다.

---

## 폐기/교체 결정

### 폐기: EFC + Density / Answer Extractiveness Style

`Answer Extractiveness Style`은 `answer ↔ context` lexical overlap을 본다.

문제:

- L2 Groundedness와 같은 축처럼 해석될 위험이 남음
- 좋은 paraphrase 답변은 Groundedness가 높고 lexical overlap은 낮을 수 있음
- integrated score에서 제외해도 운영자가 "context와 많이 겹치면 좋은 답변"으로 오해할 수 있음
- 현재 프로젝트는 복수 근거를 재구성하는 답변이 많아 신호가 약함

결론:

- EFC + Density는 L1-B 최종 지표에서 제외
- `efc_score`, `avg_coverage * 10`, `density` 기반 점수화 계획 폐기
- 필요하면 향후 별도 실험용 diagnostic으로만 재검토

### 채택: Answer Near-Duplicate Rate

대체 지표는 `answer ↔ answer` cross-QA 유사도다.

이 지표는 서로 다른 질문에 같은 답변이 재사용되는 문제를 잡는다. LLM Judge는 각 QA를 독립 평가하므로 이 dataset-level 반복 문제를 구조적으로 탐지하기 어렵다.

## 최종 L1-B 지표 구성

| 구분 | 지표 | 역할 | LLM Judge 중복 |
|------|------|------|:-------------:|
| Core score | Intent Policy Adherence | intent 생성 정책 준수 여부 | 없음 |
| Core score | Source Balance by Scope | 다문서/corpus batch의 소스 편중 여부 | 없음 |
| Light score / Monitor | Query Surface Diversity | 질문 표현 반복 진단 | 없음 |
| Light score / Monitor | Answer Near-Duplicate Rate | 답변 재사용 반복 진단 | 없음 |

## 1. Intent Policy Adherence

### 목적

TypeEntropy를 단순 균등성 지표로 쓰지 않고, 프롬프트 정책을 얼마나 따랐는지 보는 핵심 score로 변경한다.

### 대상 intent

```text
T = {fact, purpose, how, condition, comparison, list}
```

### 1차 구현 기준

현재 `DatasetStats`에는 generation config나 domain profile이 전달되지 않으므로, 1차 구현은 고정 정책 기반으로 시작한다.

```text
fact + list <= 40%
comparison <= 40%
fact <= 2개 또는 batch 규모 기반 상한
list <= 1개 또는 batch 규모 기반 상한
condition/comparison/how 최소 포함 조건은 context evidence를 알아야 하므로 1차에서는 warning만 검토
```

### 권장 출력

```python
{
    "intent_distribution": {...},
    "intent_ratios": {...},
    "type_entropy": float,
    "type_coverage": float,
    "policy_violations": [...],
    "intent_policy_score": float,  # 0~10
}
```

## 2. Source Balance by Scope

### 목적

기존 `doc_max_ratio` 구간 점수를 normalized entropy 기반으로 바꾸되, batch scope를 반영한다.

### 적용 원칙

| 범위 | 처리 |
|------|------|
| 단일 문서 batch | N/A 또는 score 제외 |
| 소수 문서 batch | monitor 또는 낮은 비중 |
| corpus/multi-doc batch | core score |

### 권장 출력

```python
{
    "source_distribution": {...},
    "source_entropy": float,
    "normalized_source_entropy": float,
    "source_scope": "single_doc" | "multi_doc" | "corpus",
    "source_balance_score": float | None,
}
```

## 3. Query Surface Diversity

### 목적

질문 품질 자체가 아니라 질문 표현 반복 여부를 보는 보조 지표다.

### 정의

```text
Distinct-1 = unique question unigrams / total question unigrams
Distinct-2 = unique question bigrams  / total question bigrams
SurfaceDiversity = (Distinct-1 + Distinct-2) / 2
```

### 주의

- Distinct-N은 여전히 corpus size에 민감함
- 소규모 batch에서는 절대 점수보다 추세/경고 용도로 사용
- LLM Judge가 보지 않는 question-set-level 표면 반복을 진단

### 권장 출력

```python
{
    "distinct_1": float,
    "distinct_2": float,
    "surface_diversity_score": float,
}
```

## 4. Answer Near-Duplicate Rate

### 목적

서로 다른 QA가 유사하거나 동일한 답변을 반복 재사용하는지 탐지한다.

질문 중복은 Query Surface Diversity가 일부 감지하지만, 답변 중복은 별도 측정이 필요하다. LLM Judge는 개별 QA를 독립적으로 평가하므로 "다른 질문에 같은 답변이 반복됐는가"를 잡지 못한다.

### 정의

현재 `_analyze_duplication()`의 `SequenceMatcher` 기반 near-duplicate 로직을 답변 필드에 적용한다.

```python
ratio = SequenceMatcher(None, answer_i, answer_j).ratio()
duplicate if ratio >= 0.7
```

pair 개수를 그대로 rate로 쓰면 pair 수 증가로 100%를 넘을 수 있으므로, 기존 구현처럼 중복으로 판정된 후속 QA item 수를 센다.

```text
answer_duplicate_count = near-duplicate로 묶인 후속 answer item 수
answer_dup_rate = answer_duplicate_count / total_qa * 100
answer_dup_score = max(0, 10 - answer_dup_rate / 2)
```

### 권장 출력

```python
{
    "answer_duplicate_count": int,
    "answer_near_duplicate_rate": float,
    "answer_dup_score": float,
    "answer_duplicate_pairs": [
        {"i": int, "j": int, "ratio": float}
    ],
    "method": "answer_sequence_similarity_0.7",
}
```

`answer_duplicate_pairs`는 UI/로그 확인용으로 최대 20개 정도만 유지한다.

## Integrated Score

단일 score가 필요하면 monitor 지표를 과도하게 반영하지 않는다.

### 단일 문서 또는 단일 hierarchy batch

```text
Integrated_L1B =
  0.8 * IntentPolicyScore
  + 0.2 * AnswerDupScore
```

Source Balance는 N/A 처리한다.

### multi-doc / corpus batch

```text
Integrated_L1B =
  0.5 * IntentPolicyScore
  + 0.3 * SourceBalanceScore
  + 0.2 * AnswerDupScore
```

Query Surface Diversity는 우선 monitor로 둔다. 운영상 질문 표현 반복이 실제 문제로 확인되면 낮은 비중으로 추가 검토한다.

## 최종 결론

L1-B는 LLM Judge와 겹치지 않는 통계 레이어로 재정의한다.

- EFC + Density / Answer Extractiveness Style은 폐기
- Answer Near-Duplicate Rate를 채택
- Intent Policy Adherence를 핵심 score로 승격
- Source Balance는 scope-aware 조건부 score로 유지
- Query Surface Diversity는 monitor로 유지
- integrated score는 의미 품질이 아니라 dataset-level 정책/반복/편중 신호만 반영

### 결정 2 — Intent Policy Adherence 점수 변환 방식

**✅ 확정: Coverage 방식** (`met_conditions / total_conditions * 10`)

```python
met_conditions = total_conditions - len(policy_violations)
intent_policy_score = round((met_conditions / total_conditions) * 10, 2)
```

- 정책 조건 항목이 늘어나도 수식 변경 불필요 (자동 스케일)
- "총 N개 조건 중 M개 충족 → X점"으로 누구에게나 설명 가능
- `policy_violations` 리스트와 점수가 직접 연결됨
- 레이더 차트에서 "높을수록 정책 준수"로 직관적 해석 가능

폐기 후보:
- ~~A. 위반 건수 차감~~ (`10 - violations * 2.5`): 조건 수 고정 전제, 계수 임의적
- ~~B. TypeEntropy 유지~~: 균등성 측정이지 정책 준수 측정이 아님
- ~~C. 혼합~~: 설명 복잡

**✅ 확정: 단순 2분류**

| scope | 조건 | Source Balance |
|-------|------|---------------|
| `single_doc` | unique docId == 1 | N/A (점수 제외) |
| `multi_doc` | unique docId >= 2 | 활성화 |

근거: 현재 QA 생성 단위가 "단일 문서 → 카테고리 선택 → 해당 청크로 QA 생성"이므로 단일 docId batch가 일반적. corpus tier 별도 구분은 현재 사용 패턴에 불필요. docId 수가 많을수록 entropy 값에 자연스럽게 반영됨.
