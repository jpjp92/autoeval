## 1. 개요 및 배경

2026-04-09 세션 문서(`docs/DEV_260409.md`)를 확인하고 적용 상태를 점검한 결과, 파싱 관련 버그 조치(FIX-A, FIX-B) 및 샘플링 쿼리 전략 등은 반영되어 있으나 **파이프라인 최적화 및 에러 대응을 위한 핵심 아키텍처**가 백엔드 코드에 미반영 상태임을 확인했다.

본 세션에서는 안정적인 대용량 인제스션(Ingestion) 및 QA 품질 개선 루프 달성을 위해 아래 4대 핵심 과제를 순차적으로 구현한다.

---

## 2. 작업 목표 및 상세 계획

### 2.1 임베딩 파이프라인 안정화 (우선순위: 매우 높음)
동시 임베딩 요청 모델 제한에 대응하고 네트워크 불안정으로 인한 파이프라인 중단을 방지한다.

- **대상 파일**: `backend/ingestion/pipeline.py`
- **적용 상세**:
  - `_embed_and_save` 내부 `embed_content` 호출부에 3회 지수 백오프 기반 재시도(Retry) 로직 추가 (tenacity 대신 자체 루프 구현 유지).
  - API 동시 폭주 방지를 위한 `asyncio.Semaphore` (`EMBED_SEM`) 계층 신설.

### 2.2 비동기 인제스션(Ingestion) 아키텍처 전환
대용량 파일 처리 시 타임아웃 문제를 해결하고, 태스크별(EXTRACTING, CHUNKING, EMBEDDING) 구체적 진행 상황을 트래킹한다.

- **대상 파일**: `backend/ingestion/job_manager.py` (신규), `backend/api/ingestion_api.py`, `backend/ingestion/llm_chunker.py`
- **적용 상세**:
  - `IngestionJobManager` 및 `IngestionStatus` Enum 신설.
  - `POST /upload` 엔드포인트를 FastAPI `BackgroundTasks`를 통한 비동기 패턴으로 전환.
  - `llm_chunker.py`의 배치 루프에 상태 업데이트 보고 기능 주입.

### 2.3 동일 문서 안전 재처리 체계 구축 (Orphan Cleanup & Retry)
임베딩이나 DB 저장 도중 실패했을 때, 지저분하게 남은 잔여 데이터(orphan chunk)를 자동 정리하고 지정된 문서 ID로 클린 재시작을 지원한다.

- **대상 파일**: `backend/db/doc_chunk_repo.py`, `backend/ingestion/pipeline.py`, `backend/api/ingestion_api.py`
- **적용 상세**:
  - 지정 문서 단위 청크 일괄 삭제 함수 `delete_chunks_by_document_id(doc_id: str)` 개발.
  - `process_and_ingest` 함수의 `doc_id` 커스텀 허용을 통한 멱등성(Idempotency) 보장.
  - `POST /api/ingestion/{job_id}/retry` 재처리 엔드포인트 마련 (실패 job 조회 -> cleanup -> background task 재시작 로직).

### 2.4 QA 품질 기반 재처리 시스템 (QA Recovery)
저품질 혹은 실패한 특정 QA를 추적하고, 이전 실패 사유를 프롬프트에 주입하여 재생성하는 파이프라인을 구축한다.

- **대상 파일**: `backend/generators/qa_generator.py`, `backend/generators/prompts.py`, 데이터베이스 스키마
- **적용 상세**:
  - QA 응답 파싱 과정에서 고유 식별자(`qa_id`, UUID) 보장 로직 추가.
  - 평가 피드백(실패 사유)를 프롬프트에 주입할 수 있는 `build_recovery_user_template` 프롬프트 빌더 개발.
  - `qa_gen_results` 테이블 스키마 확장 (`retry_count`, `parent_job_id`, `is_recovery` 속성 추가).

## 3. 프론트엔드 연동 계획
백엔드 상의 API들이 비동기 및 상태 기반으로 리팩토링된 이후 적용한다.

1. **상태 폴링(Polling) UI**: 업로드 직후 `job_id`를 기준으로 백그라운드 진행 상태(IngestionStatus)를 폴링하여 UI에 퍼센티지 및 단계 표시.
2. **리트라이 UI**: 업로드 실패 시 '재시도' 버튼 활성화.
3. **Zustand 리팩토링**: 이전에 남은 과제인 `DataStandardizationPanel` ErrorBoundary 분리 및 Prop drilling 해소 동시 반영 검토.
