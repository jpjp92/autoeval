# L1-B Statistical Pre-screening 메트릭 재설계 — 2026-05-08

LLM Judge(L2 RAG Triad, L3 Completeness) 이전에 실행하는 통계 기반 사전 검증 레이어 재설계 논의 정리.

---

## 설계 원칙

| 조건 | 내용 |
|------|------|
| Reference-free | 정답 QA 없이 계산 가능 |
| LLM 호출 없음 | 통계·수식 기반 |
| LLM Judge와 차원 분리 | L2/L3과 측정 대상 겹치지 않음 |
| 논문 인용 가능 | 명확한 출처 있음 |
| 4개 메트릭이 서로 다른 차원 | 중복 없음 |

## 최종 후보 4개

### 1. Distinct-1/2 — 질문 어휘 다양성 (Lexical)

**논문**: Li, J., Galley, M., Brockett, C., Gao, J., & Dolan, B., *"A Diversity-Promoting Objective Function for Neural Conversation Models"*, NAACL 2016. ACL Anthology: N16-1014.

**수식**:
```
Distinct-1 = |unique unigrams| / |total unigrams|   ← 어휘 다양성
Distinct-2 = |unique bigrams|  / |total bigrams|    ← 구문 다양성

Score = (Distinct-1 + Distinct-2) / 2
```

**측정 차원**: 질문 집합의 어휘·구문 수준 표면 다양성. Distinct-1은 TTR과 동일하나 Distinct-2를 추가해 구문 반복 패턴까지 감지하고 코퍼스 크기 민감성을 완화.

**논문 적용 범위**: Li et al.의 원 논문은 신경 대화 모델의 응답 다양성 측정에 Distinct-N을 적용. 이 시스템에서는 동일 수식을 생성 QA 데이터셋의 질문 다양성 측정에 적용. 질문 역시 생성 텍스트에 해당하므로 적용 범위 동일.

**Vendi Score 관계**: Vendi Score(arXiv:2210.02410)는 embedding 기반 semantic 다양성으로 더 강한 신호이나, 평가 단계 embedding 조달 문제로 보류. 조달 방법 확보 시 Distinct-1/2를 대체할 예정.

**LLM Judge 겹침**: 없음. L2/L3은 질문 다양성을 보지 않음.

### 2. Extractive Fragment Coverage + Density — 답변 추출성

**논문**: Grusky, Naaman & Artzi, *"Newsroom: A Dataset of 1.3 Million Summaries with Diverse Extractive Strategies"*, NAACL-HLT 2018

**수식**:
```
F(A, C) = answer A와 context C의 공통 연속 fragment 집합

Coverage(A, C) = (1/|A|) × Σ_{f ∈ F} |f|
  → 답변 토큰 중 context에서 직접 추출된 비율

Density(A, C)  = (1/|A|) × Σ_{f ∈ F} |f|²
  → 추출 fragment의 평균 길이 (길수록 덩어리째 복사)
```

**해석**:
```
Coverage 높 + Density 높  → 긴 문장을 통째로 복사 (extractive)
Coverage 높 + Density 낮  → 흩어진 단어 조합 (semi-extractive)
Coverage 낮               → 추론·재구성 중심 (abstractive)
```

**논문 적용 범위**: Grusky et al.의 원 논문은 뉴스 요약(summarization)을 대상으로 Coverage/Density 수식을 제안. 이 시스템에서는 동일 수식을 QA 답변 추출성 측정에 전용(轉用).

**측정 차원**: 개별 QA 쌍의 답변이 컨텍스트에서 얼마나 직접 도출됐는지. Distinct-1/2가 질문 집합 횡적 다양성을 보는 것과 달리, 종적 answer-context 연결을 봄.

**LLM Judge 겹침**: 없음. L2 Groundedness는 semantic entailment(flexible matching 포함), Coverage는 lexical fragment 추출 비율. 좋은 paraphrase 답변은 L2 높고 Coverage 낮을 수 있음 → 다른 신호.

### 3. TypeEntropy — 의도 커버리지

**논문 근거**:
- Xu et al., *"Fantastic Questions and Where to Find Them: FairytaleQA – An Authentic Dataset for Narrative Comprehension"*, ACL 2022
  → question type distribution을 명시적 데이터셋 품질 기준으로 사용
- Lai et al., *"RACE: Large-scale ReAding Comprehension Dataset From Examinations"*, EMNLP 2017
  → reasoning type(detail/inference/main idea/attitude) 분포 보고를 표준 관행으로 확립

**수식**:
```
T = {fact, purpose, how, condition, comparison, list}  (6종 intent)

TypeEntropy = -Σ_{t ∈ T} p(t) log p(t)
  → 균등 분포일수록 높음 (최대값 = log 6 ≈ 1.79)

TypeCoverage = |{t ∈ T : count(t) ≥ 1}| / |T|
  → 6종 중 실제로 등장한 종류 비율
```

**수식 출처**: TypeEntropy 수식 자체는 표준 Shannon entropy(정보이론)이며 별도 논문 출처 없음. 위 두 논문은 QA 데이터셋 평가에서 question type 분포를 품질 기준으로 사용한 선례로 인용.

**Macro F1 대신 TypeEntropy를 쓰는 이유**: Macro F1은 reference 분포(이상적 비율)가 필요. TypeEntropy는 reference 없이 분포 균형도만 측정.

**측정 차원**: 생성 프롬프트가 6종 intent를 요구하는데 실제 생성 결과가 이를 지키는지 직접 검증. Vendi Score가 표현 다양성을 보는 것과 달리, 의도 유형 커버리지를 봄.

**LLM Judge 겹침**: 없음. L2/L3은 intent 분포를 보지 않음.

### 4. Normalized Source Entropy — 소스 균형도

**논문 근거**:
- Dodge et al., *"Documenting Large Webtext Corpora: A Case Study on the Colossal Clean Crawled Corpus"*, EMNLP 2021
  → 소스 도메인 분포를 entropy로 분석, 특정 도메인 과잉 집중 문제를 entropy 감소로 진단
- Dasigi et al., *"QASPER: A Dataset of Information-Seeking Questions over Scientific Research Papers"*, NAACL 2021
  → 논문(소스)별 QA 수 분포를 통계로 보고

**수식**:
```
p(d) = 문서 d에서 생성된 QA 수 / 전체 QA 수

H = -Σ_{d ∈ D} p(d) log₂ p(d)
H_max = log₂ |D|   (완전 균등 시 최대값)

Normalized_H = H / H_max   → 0(완전 집중) ~ 1(완전 균등)
```

**측정 차원**: QA가 특정 문서에 쏠리지 않고 전체 소스에 균등하게 분포하는지. 현재 L1-B의 doc_max_ratio 구간 점수를 수식 기반으로 대체.

**LLM Judge 겹침**: 없음. L2/L3은 소스 분포를 보지 않음.

## L2/L3 LLM Judge 메트릭 레퍼런스

### RAG Triad 프레임워크 — L2 전체

**프레임워크 출처**: "RAG Triad" 명칭은 TruLens/TruEra 실무 프레임워크에서 유래 — 단일 학술 논문 없음.

**가장 근접한 학술 논문**: Es, S., James, J., Espinosa-Anke, L., & Schockaert, S. (2023). *RAGAS: Automated Evaluation of Retrieval Augmented Generation*. arXiv preprint arXiv:2309.15217.
→ answer_relevancy / faithfulness(=groundedness) / context_precision 3종을 동일하게 정의·측정.

---

### L2-1. Answer Relevance — 답변 관련성

**측정 차원**: 답변이 질문의 모든 측면을 완전히 다루는지

**논문 근거**:
- Es et al. (2023), RAGAS — `answer_relevancy`: 질문과 답변의 의미적 일치도를 LLM Judge로 평가하는 방법론의 직접적 선례.

**LLM Judge 겹침**: L3 Completeness와 관련 있으나 측정 방향이 다름. Relevance는 질문→답변 적합성(지엽적 답변 패널티), Completeness는 질문 분해 기반 커버리지(누락 항목 패널티).

---

### L2-2. Groundedness — 근거성

**측정 차원**: 답변의 모든 주장이 컨텍스트에 직접 근거하는지 (환각 감지)

**논문 근거**:
- Maynez, J., Narayan, S., Bohnet, B., & McDonald, R. (2020). *On Faithfulness and Factuality in Abstractive Summarization*. ACL 2020. ACL Anthology: 2020.acl-main.173.
  → faithfulness(충실성) 개념의 기초 논문. 생성 텍스트의 주장이 소스에 직접 지지되는지 측정하는 구조가 이 시스템의 groundedness와 동일.
- Es et al. (2023), RAGAS — `faithfulness`: claim 단위 분해 후 context 지지 여부 판정.

**논문 적용 범위**: Maynez et al.의 원 논문은 뉴스 요약(summarization)의 faithfulness를 대상으로 함. 이 시스템에서는 동일 개념을 RAG QA 답변의 근거성 측정에 전용(轉用).

---

### L2-3. Context Relevance — 컨텍스트 관련성

**측정 차원**: 검색된 컨텍스트가 질문에 답하는 데 필요한 정보를 담고 있는지

**논문 근거**:
- Es et al. (2023), RAGAS — `context_precision` / `context_relevance`: 검색된 컨텍스트 중 질문과 관련된 비율을 LLM Judge로 평가.
- Saad-Falcon, J., Khattab, O., Potts, C., & Zaharia, M. (2023). *ARES: An Automated Evaluation Framework for Retrieval-Augmented Generation Systems*. arXiv preprint arXiv:2311.09476. (NAACL 2024)
  → Context Relevance를 RAG 평가의 독립 차원으로 정의하고 LLM Judge 방식으로 측정하는 방법론 선례.

---

### L3-4. Completeness — 완전성

**측정 차원**: 질문을 원자 단위 하위 질문으로 분해 → 각 하위 질문의 답변·컨텍스트 지지 여부 → 커버리지 비율

**논문 근거**:
- Min, S., Krishna, K., Lyu, X., et al. (2023). *FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text Generation*. EMNLP 2023. ACL Anthology: 2023.emnlp-main.741.
  → 생성 텍스트를 atomic fact 단위로 분해 후 각 fact의 지지 여부를 판정하는 방법론. 이 시스템의 sub-question 분해·커버리지 계산 구조의 직접적 선례.
- Wolfson, T., Geva, M., Gupta, A., et al. (2020). *Break It Down: A Question Understanding Benchmark*. TACL 2020. ACL Anthology: 2020.tacl-1.13.
  → 복합 질문을 원자 단위로 분해하는 question decomposition 방법론의 기초 벤치마크.

**논문 적용 범위**: FActScore는 원 논문에서 long-form 생성 텍스트의 factual precision을 claim 레벨에서 평가. 이 시스템에서는 동일한 atomic decomposition 구조를 QA 답변의 완전성(sub-question 커버리지) 측정에 전용(轉用).

**수식 출처**: Completeness = (컨텍스트 지지를 받으며 답변된 하위 질문 수) / (전체 하위 질문 수). 분해·커버리지 구조는 FActScore에서, 비율 계산은 표준 coverage 공식.

#### 각 메서드 구현 요약

**`_calc_distinct()`** — Distinct-1/2 평균
```python
unigrams = [w for q in questions for w in q.split()]
d1 = len(set(unigrams)) / max(len(unigrams), 1)

bigrams = [(w, questions[i].split()[j+1])
           for i, q in enumerate(questions)
           for j, w in enumerate(q.split()[:-1])]
d2 = len(set(bigrams)) / max(len(bigrams), 1)

score = ((d1 + d2) / 2) * 10
stats = {"distinct_1": d1, "distinct_2": d2}
```

**`_calc_efc_density()`** — Extractive Fragment Coverage + Density
```python
# SequenceMatcher 이미 import됨 — get_matching_blocks() 재활용
for qa in self.qa_list:
    a_tokens = qa.get("a", "").split()
    c_tokens = qa.get("context", "").split()
    matcher  = SequenceMatcher(None, a_tokens, c_tokens, autojunk=False)
    fragments = [b.size for b in matcher.get_matching_blocks() if b.size > 0]
    n = max(len(a_tokens), 1)
    coverage = sum(fragments) / n
    density  = sum(f**2 for f in fragments) / n

score = avg_coverage * 10   # coverage 기반 점수, density는 참고 stat
stats = {"avg_coverage": avg_coverage, "avg_density": avg_density}
```

**`_calc_type_entropy()`** — TypeEntropy + TypeCoverage
```python
INTENT_TYPES = {"fact", "purpose", "how", "condition", "comparison", "list"}

counts = Counter([qa.get("intent", "").lower() for qa in self.qa_list])
total  = max(sum(counts.values()), 1)

entropy = -sum(
    (counts.get(t, 0) / total) * math.log(counts.get(t, 0) / total)
    for t in INTENT_TYPES if counts.get(t, 0) > 0
)
max_entropy = math.log(6)   # log(6) ≈ 1.791 — 6종 고정 (동적 max 아님)
coverage    = len([t for t in INTENT_TYPES if counts.get(t, 0) >= 1]) / 6

score = (entropy / max_entropy) * 10
stats = {"type_entropy": entropy, "type_coverage": coverage,
         "intent_distribution": {t: counts.get(t, 0) for t in INTENT_TYPES}}
```

> 현재 `_analyze_diversity()`의 intent entropy는 `log2(실제 unique 수)` 동적 max → TypeEntropy는 `log(6)` 고정 max로 변경. 미출현 intent type 패널티 부여.

**`_calc_source_entropy()`** — Normalized Source Entropy
```python
doc_dist = Counter([qa.get("docId", "unknown") for qa in self.qa_list])

H     = self._calculate_entropy(doc_dist)          # 기존 메서드 재활용
H_max = math.log2(len(doc_dist)) if len(doc_dist) > 1 else 1.0
normalized_H = H / H_max if H_max > 0 else 1.0

score = normalized_H * 10
stats = {"doc_distribution": dict(doc_dist),
         "source_entropy": H, "normalized_entropy": normalized_H}
```

> 현재 `_analyze_skewness()`는 `doc_entropy`를 이미 계산하지만 score에 미반영 (구간 점수 사용). `_calculate_entropy()` 재활용 + 정규화만 추가.

## 프로젝트 적합형 4개 지표

### 1. Query Surface Diversity — 질문 표면 다양성

**기존 후보 대응**: Distinct-1/2 유지, 해석만 수정

**정의**:
```
Distinct-1 = |unique unigrams in questions| / |total unigrams|
Distinct-2 = |unique bigrams in questions|  / |total bigrams|

SurfaceDiversity = (Distinct-1 + Distinct-2) / 2
```

**프로젝트 맞춤 해석**:
- 이 값은 질문 품질 자체가 아니라 질문 표현 반복 여부를 보는 보조 지표다.
- intent 템플릿이 고정되어 있으므로 낮다고 해서 곧바로 나쁜 셋이라고 볼 수 없다.
- 동일 intent 안에서 질문 골격이 과도하게 복제되는지 감지하는 용도로 사용한다.

**권장 용도**:
- `monitor` 또는 `warning`
- integrated score 반영 비중은 낮게 유지
- 소규모 배치에서는 점수 비교보다 추세 비교에 사용

**권장 출력**:
- `distinct_1`
- `distinct_2`
- `surface_diversity_score`

**주의**:
- 문서당 QA 수가 적으면 값이 크게 흔들린다.
- 가능하면 동일 개수 샘플 기준 비교 또는 최소 QA 수 조건을 두는 것이 바람직하다.

---

### 2. Answer Extractiveness Style — 답변 추출성 스타일

> **명칭 변경 사유**: 이전 명칭 "Answer Grounding Style"은 LLM Judge의 Groundedness(의미적 근거성)와 같은 축처럼 읽힐 수 있어 혼동 위험이 있다. 이 지표는 lexical copy 비율을 보는 통계 진단이며 semantic grounding과 별개다.

**기존 후보 대응**: EFC + Density 유지, 이름과 역할 수정

**정의**:
```
Coverage(A, C) = answer 토큰 중 context와 lexical fragment로 직접 겹치는 비율
Density(A, C)  = 겹친 fragment가 얼마나 긴 덩어리인지 나타내는 값
```

**LLM Judge 비중복 확인**:
- L2 Groundedness는 답변의 claim이 context에 **의미적으로** 지지되는지를 본다 (semantic entailment 기반).
- 이 지표는 답변 토큰이 context 토큰과 **문자 레벨**로 얼마나 겹치는지를 본다 (lexical copy 비율).
- 좋은 paraphrase 답변은 L2 Groundedness 높고 Coverage 낮을 수 있다 → 서로 다른 신호.
- 이 지표를 integrated score에 강하게 반영하거나 "근거성" 명칭으로 쓰면 Groundedness와 충돌한다.

**프로젝트 맞춤 해석**:
- 이 값은 "답변이 얼마나 잘 맞는가"가 아니라 "답변이 얼마나 복사형인가"를 본다.
- 현재 프로젝트 답변은 비교, 목적, 조건, 열거형에서 여러 근거를 합쳐 서술하는 구조가 많다.
- 따라서 낮은 Coverage가 곧바로 품질 저하를 뜻하지 않는다.

**권장 용도**:
- `monitor only`
- integrated score에서 제외하거나 비중을 매우 낮게 유지
- intent별 분포 확인용 진단 지표로 사용

**권장 출력**:
- `avg_coverage`
- `avg_density`
- `coverage_by_intent`
- `density_by_intent`

**권장 해석 예시**:
- `comparison`, `purpose`: paraphrase 허용 폭이 크므로 Coverage 낮아도 정상 가능
- `list`, `fact`: context 항목 직접 반영이 많아 Coverage 상대적으로 높을 수 있음

**주의**:
- 이 지표를 groundedness 대체 지표로 쓰면 안 된다.
- lexical overlap이 높은 답변이 semantic grounding도 높다는 보장은 없다.

### 3. Intent Policy Adherence — 의도 분포 준수도

**기존 후보 대응**: TypeEntropy 대체 해석 및 점수식 수정

현재 프로젝트는 intent가 단순 태그가 아니라 생성 정책이다. 따라서 균등 분포 최대화보다 프롬프트 목표 분포를 얼마나 잘 따랐는지를 보는 것이 더 적합하다.

**입력 전제**:
```
T = {fact, purpose, how, condition, comparison, list}
```

**권장 측정 방식**:
```
Observed(t) = 생성 결과에서 intent t의 비율
Target(t)   = 프롬프트 정책 또는 운영 설정에서 기대하는 비율

PolicyDeviation = Σ_t |Observed(t) - Target(t)|
PolicyScore = 1 - PolicyDeviation / MaxDeviation
```

**Target 분포 설정 예시**:
- 균등 분포를 기본값으로 두지 않음
- 프롬프트 규칙 반영:
  - fact + list 과다 비중 제한
  - comparison 상한 반영
  - condition/comparison 최소 포함 조건 반영
  - how 포함 권장 반영

**보조 통계**:
```
TypeEntropy = -Σ p(t) log p(t)
TypeCoverage = 등장한 intent 종류 수 / 6
```

**프로젝트 맞춤 해석**:
- TypeEntropy는 참고 통계로 남길 수 있다.
- 하지만 실제 운영 점수는 `정책 이탈도` 중심으로 계산하는 편이 맞다.

**권장 용도**:
- `hard gate + score`
- L1-B 핵심 지표 중 하나

**권장 출력**:
- `intent_distribution`
- `type_entropy`
- `type_coverage`
- `policy_deviation`
- `intent_policy_score`

---

### 4. Source Balance by Scope — 범위 인지형 소스 균형도

**기존 후보 대응**: Normalized Source Entropy 유지, 적용 범위 분리

**정의**:
```
p(d) = 문서 d에서 생성된 QA 수 / 전체 QA 수
H = -Σ p(d) log₂ p(d)
H_max = log₂ |D|
Normalized_H = H / H_max
```

**프로젝트 맞춤 해석**:
- 이 값은 코퍼스 수준 분포 진단에는 유효하다.
- 하지만 단일 문서 생성 job이나 특정 hierarchy만 평가하는 배치에서는 낮게 나오는 것이 정상일 수 있다.

**권장 적용 범위 구분**:

| 범위 | 사용 방식 |
|------|----------|
| 단일 문서 batch | 비활성 또는 N/A 처리 |
| 소수 문서 batch | 참고 지표만 표시 |
| 코퍼스 집계 batch | 핵심 score로 사용 가능 |

**권장 용도**:
- `scope-aware score`
- batch metadata를 보고 score 계산 여부를 결정

**권장 출력**:
- `source_distribution`
- `source_entropy`
- `normalized_source_entropy`
- `source_scope` (`single_doc` / `multi_doc` / `corpus`)

**주의**:
- 현재 프로젝트처럼 문서 단위 생성이 흔한 경우, 모든 batch에 일괄 점수화하면 false alarm이 많아진다.

## 권장 레이어 재정의

### L1-B를 2개 score + 2개 monitor로 분리

| 구분 | 지표 | 역할 |
|------|------|------|
| Core score | Intent Policy Adherence | 생성 정책 준수 여부 |
| Core score | Source Balance by Scope | 다문서 집계 시 소스 편중 여부 |
| Monitor | Query Surface Diversity | 질문 표현 반복 진단 |
| Monitor | Answer Extractiveness Style | 복사형/재구성형 답변 추출성 진단 |

이 구성이 현재 프로젝트에 더 맞는 이유는 다음과 같다.

- 현재 프로젝트는 intent 통제가 강하므로 intent policy adherence가 실제 품질 리스크를 가장 잘 드러냄
- source balance는 코퍼스 운영 품질에는 중요하지만 문서 단위 batch에는 비적합할 수 있음
- distinct와 coverage는 스타일 진단에는 유용하지만 hard gate로 쓰기엔 오탐 가능성이 큼

## 권장 integrated score 구성

단일 score가 꼭 필요하면 아래처럼 범위별로 다르게 계산하는 것이 안전하다.

### A. 단일 문서 또는 단일 hierarchy batch

```
Integrated_L1B =
  0.7 * IntentPolicyScore
  + 0.3 * SurfaceDiversityScore
```

- Source Balance는 제외 또는 N/A
- Grounding Style은 참고 통계만 제공

### B. 다문서 corpus batch

```
Integrated_L1B =
  0.5 * IntentPolicyScore
  + 0.3 * SourceBalanceScore
  + 0.2 * SurfaceDiversityScore
```

- Grounding Style은 여전히 monitor 유지

기존 4개 후보를 모두 버릴 필요는 없다. 다만 metric의 이름과 점수 역할을 아래처럼 바꾸는 것이 현재 프로젝트에 더 적합하다.

| 현재 후보 | 프로젝트 적합형 이름 | 권장 역할 |
|----------|--------------------|----------|
| Distinct-1/2 | Query Surface Diversity | monitor / low-weight score |
| EFC + Density | Answer Extractiveness Style | monitor only |

## 결론

현재 프로젝트에 맞는 L1-B는 "질문이 얼마나 다양하냐"보다 "의도 정책을 지켰는가", "다문서 집계에서 특정 소스에 과도하게 쏠렸는가"를 더 우선해서 봐야 한다.

즉, 기존 4개 후보는 완전히 폐기할 대상이 아니라 다음처럼 재배치하는 것이 적절하다.

- Distinct-1/2: 유지하되 보조 진단 지표로 격하
- EFC + Density: 유지하되 스타일 모니터로 한정
- TypeEntropy: 정책 준수도 중심 지표로 승격
- Source Entropy: batch scope 인지형 지표로 조건부 사용

이 재정의는 현재 프로젝트의 intent-controlled, doc-linked, context-grounded QA 생성 구조와 가장 잘 맞는다.

## L1-B ↔ LLM Judge 비중복 검토 (2026-05-11)

`rag_triad.py`, `qa_quality.py` 기반으로 LLM Judge 4개 축과 L1-B 재정의안 4개 지표의 중복 여부를 직접 확인.

### LLM Judge 4개 축 (현행)

| LLM Judge | 측정 단위 | 실제 측정 내용 |
|-----------|----------|---------------|
| Answer Relevance | 개별 QA | 질문의 모든 요구를 답변이 다루는가 |
| Groundedness | 개별 QA | 답변의 claim이 context에 **의미적으로** 지지되는가 |
| Context Relevance | 개별 QA | context가 질문에 답할 정보를 담고 있는가 |
| Completeness | 개별 QA | 원자 하위 질문 분해 시 빠진 답변이 없는가 |

### L1-B 4개 지표 vs LLM Judge 겹침 여부

| L1-B 지표 | 측정 단위 | LLM Judge와 중복 | 판단 근거 |
|-----------|----------|:----------------:|---------|
| Query Surface Diversity | 질문 집합 | 없음 | LLM Judge는 개별 QA 의미 품질을 보며 질문 집합의 표현 다양성은 측정 대상 밖 |
| Answer Extractiveness Style | 개별 QA (집계) | **부분 위험** | lexical copy 비율 진단은 비중복이나, 이름에 "근거" 관련 단어가 들어가거나 integrated score에 강하게 반영하면 Groundedness와 혼동 |
| Intent Policy Adherence | 데이터셋 | 없음 | intent 분포가 생성 정책을 따르는지 보는 dataset-level 통계. LLM Judge는 개별 QA가 질문/답변으로 성립하는지만 봄 |
| Source Balance by Scope | 데이터셋 | 없음 | docId 분포 진단. Context Relevance는 개별 QA context 충분성이라 측정 단위와 대상이 다름 |

### 핵심 위험 지점

**Answer Extractiveness Style — Groundedness 혼동 위험**

이전 명칭 "Answer Grounding Style"은 LLM Judge Groundedness와 같은 축처럼 읽힌다. 이를 방지하기 위해 두 가지를 지킨다.

1. 명칭을 "Answer Extractiveness Style"로 확정 — Grounding 계열 단어 사용 금지
2. integrated score 반영 금지 — 이 지표를 점수화해 aggregation에 포함하면 Groundedness와 사실상 이중 측정이 된다

### 최종 권장 구조 (LLM Judge 비중복 반영)

```
L1-B Core score
  ├── Intent Policy Adherence       → dataset-level, 정책 준수 여부
  └── Source Balance by Scope       → dataset-level, 소스 편중 여부 (multi-doc 한정)

L1-B Monitor (score 미반영)
  ├── Query Surface Diversity        → 질문 표현 반복 진단
  └── Answer Extractiveness Style   → lexical copy 비율 진단 (LLM Groundedness와 별개)

L2 LLM Judge
  ├── Answer Relevance
  ├── Groundedness                  ← Extractiveness Style과 측정 축 다름
  └── Context Relevance

L3 LLM Judge
  └── Completeness
```
