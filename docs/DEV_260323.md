### 1-1. QADetailView 레이아웃 재구성

**문제**

| 문제 | 위치 |
|------|------|
| 점수(avg)와 근거(reason)가 분리 — 같은 정보를 두 섹션에 나눠 표현 | 점수 2열 ↔ 평가 근거 |
| 중첩 컨테이너 3단계 — 평가 근거 div > RAG 박스 > row | 평가 근거 섹션 |
| 주요 실패 유형이 맨 마지막 — 실패 QA 검토 시 가장 먼저 봐야 할 정보 | 전체 구조 |

**개선 후 섹션 순서**

```
[헤더] 목록으로 | #번호 · intent · 상태
[실패 callout]  ← primary_failure 있을 때만, 헤더 직후
[질문]
[답변]
[컨텍스트]
[RAG Triad 섹션]  헤더에 avg 포함 → 차원별 reason 나열
[품질 평가 섹션]  헤더에 avg 포함 → 차원별 reason 나열
```

**구조 변경 요점**

| 현재 | 개선 |
|------|------|
| 점수 2열 카드 + 평가 근거 2박스 = 4개 컨테이너 | RAG 섹션 + 품질 섹션 = 2개 컨테이너 |
| avg 카드 → 스크롤 → reason 박스 (분리) | avg를 섹션 헤더에 인라인 (통합) |
| 주요 실패 유형 맨 마지막 | 헤더 직후로 이동 |
| 중첩 3단계 | 섹션 > row 2단계 |

**수정 파일**: `QAEvaluationDashboard.tsx` — `QADetailView` 함수 내부만 변경

### 1-2. 대시보드 로그 클릭 → Evaluation 히스토리 연동 버그

**원인**

```
Dashboard onEvalSelect(eval_job_id)
  → App.tsx: setLastEvalJobId(eval_job_id)
  → QAEvaluationDashboard: evalJobId prop 수신
  → useEffect: getEvalStatus(evalJobId)   ← in-memory 세션 조회만 함
```

`getEvalStatus()`는 현재 세션 in-memory job만 조회. Dashboard에서 넘어오는 `eval_job_id`는 DB 저장 과거 평가라 in-memory에 없음 → 조회 실패 → 마지막 세션 상태 유지.

히스토리 선택 경로(`selectHistory` → `getEvalExportById(item.id)`)가 Dashboard 진입 시 연결되지 않은 것.

추가 문제: `recent_jobs`에서 반환하는 `eval_job_id`는 `qa_eval_results.job_id`(UUID)이지만, `selectHistory`는 `qa_eval_results.id`(Supabase PK)를 필요로 함 → DB에서 애초에 잘못된 식별자를 반환.

**수정 범위**

| 파일 | 변경 내용 |
|------|----------|
| `backend/db/dashboard_repo.py` | `recent_jobs`에 `eval_id` 필드 추가 (`linked_eval.get("id")`) |
| `frontend/.../DashboardOverview.tsx` | 인터페이스 `eval_id` 추가, `onEvalSelect` 시 `eval_id` 전달 |
| `frontend/src/App.tsx` | `lastEvalDbId` state 추가, `initialEvalDbId` prop 연결 (기존 `lastEvalJobId`는 Generation 세션용으로 유지) |
| `frontend/.../QAEvaluationDashboard.tsx` | `initialEvalDbId` prop 추가. historyList 로드 후 id 매칭 → `selectHistory` 자동 호출 |

**effect 분리 설계**

```
// historyList 로드 (마운트 시 1회)
useEffect(() => { getEvalHistory() → setHistoryList }, []);

// initialEvalDbId 변경 시 자동 선택 (historyList 준비 대기)
useEffect(() => {
  if (!initialEvalDbId || !historyList.length) return;
  const target = historyList.find(h => h.id === initialEvalDbId);
  if (target) selectHistory(target);
}, [initialEvalDbId, historyList]);
```

`historyList` 로드와 `initialEvalDbId` 설정 타이밍을 effect 분리로 해결.

## 5. 백엔드 성능 개선

### 5-1. Dashboard doc_chunks 쿼리 최적화 — 구현 완료

**문제**

`dashboard_repo.py`에서 doc_chunks 전체 행의 metadata를 Python으로 가져와 distinct 처리 중.
청크 수가 많을수록 전송 데이터가 선형 증가.

```python
# 기존: 전체 metadata 전송 후 Python에서 distinct
chunks_res = supabase.table("doc_chunks").select("metadata").execute()
doc_names = set()
for c in chunks_res.data or []:
    fn = (c.get("metadata") or {}).get("filename")
    if fn:
        doc_names.add(fn)
```

**구현**

Supabase SQL Editor에 RPC 함수 등록 완료:

```sql
CREATE OR REPLACE FUNCTION get_distinct_doc_count()
RETURNS integer AS $$
  SELECT COUNT(DISTINCT metadata->>'filename')::integer FROM doc_chunks;
$$ LANGUAGE sql STABLE;
```

`dashboard_repo.py` 변경:

```python
# 개선: DB에서 integer 1개만 반환
count_res = supabase.rpc("get_distinct_doc_count").execute()
total_documents = count_res.data or 0
```

**효과**

| 항목 | 내용 |
|------|------|
| doc_chunks 쿼리 | warm 상태에서 청크 수 무관하게 정수 1개 반환 → 데이터 전송 최소화 |
| Cold start (Render sleep) | 개선 없음 — 외부 keep-alive ping으로 별도 대응 |
| 3개 쿼리 직렬화 | supabase-py 동기 클라이언트 한계로 병렬화 미적용 |

**참고**: `docs/plan/SUPABASE_SCHEMA.sql`은 기존 기획 단계 파일로 실제 테이블명(`qa_gen_results`, `qa_eval_results`)과 불일치. `docs/plan/`은 `.gitignore`에 이미 포함되어 있음.

## 6. 백엔드 성능 개선 계획 (미구현)

### 6-1. 파이프라인 단계별 병목 분석

| 단계 | 실제 원인 | 개선 가능 여부 |
|------|-----------|----------------|
| 업로드 → 벡터화 | 청크별 Supabase write 순차 (배치 64개 × 2 호출 = 128 네트워크 요청) | **개선 가능** |
| 계층구조 분석 | LLM 응답 대기 1회 (문서 크기 무관) | LLM 한계, 어려움 |
| 계층 태깅 | 청크 수 비례, 이미 `asyncio.gather + Semaphore(5)` 병렬 처리 | 현재 구조 양호 |
| QA 평가 | 모델별 worker 수 제한 (Anthropic 2, Google 10, OpenAI 8) | 모델 선택으로 조절 가능 |
| Cold start | Render 무료 플랜 15분 비활성 시 sleep | UptimeRobot 등 외부 ping으로 완화 |

### 6-2. 벡터화 배치 INSERT 최적화 — 구현 완료

**문제**

`save_doc_chunk()`는 `async def`로 선언되어 있으나 내부는 동기 supabase 클라이언트 사용 (blocking).
청크 1개당 SELECT(중복체크) + INSERT = 네트워크 요청 2회 → 64청크 배치 기준 **128회** 순차 호출.
`asyncio.gather`만으로는 효과 없음 (event loop blocking).

**구현**

```python
# 1) 중복 체크: hash 목록 1회 SELECT
existing_res = supabase.table("doc_chunks")
    .select("metadata")
    .in_("metadata->>content_hash", hash_list)
    .execute()

# 2) 신규 청크만 1회 배치 INSERT
supabase.table("doc_chunks").insert([row1, row2, ...]).execute()
```

**수정 파일**

| 파일 | 변경 내용 |
|------|-----------|
| `backend/db/doc_chunk_repo.py` | `save_doc_chunks_batch()` 신규 추가 (hash 목록 1회 SELECT → 신규만 배치 INSERT) |
| `backend/config/supabase_client.py` | `save_doc_chunks_batch` re-export 추가 |
| `backend/api/ingestion_api.py` | 배치 루프 내 청크별 `await save_doc_chunk()` → `batch_rows` 수집 후 `await save_doc_chunks_batch(batch_rows)` 1회 호출 |

**효과**

배치 1회(64청크) 기준 128 네트워크 호출 → 2 네트워크 호출. 청크가 많을수록 효과 선형 증가.

**참고**

`metadata.chunk_index` 필드(`i + idx`)로 논리적 순서 보존. `.in_("metadata->>content_hash", ...)` PostgREST JSONB 필터 지원 여부는 실제 업로드 테스트로 확인 필요.

### 8-2. QA 생성 "문서 청크를 찾을 수 없습니다" 오류 수정

**원인**

`ingestion_api.py`의 `process_and_ingest()`에서 `chunk_metadata`를 구성한 직후 `hierarchy_h1/h2/h3`를 `pop`으로 제거하고 있었음.
→ DB에 저장된 청크에 해당 필드가 없으므로, 생성 시 `get_doc_chunks_by_filter(hierarchy_h1=h1, ...)`가 항상 0건 반환 → `ValueError` 발생.

**수정 범위**

| 파일 | 변경 내용 |
|------|-----------|
| `backend/api/ingestion_api.py` | `hierarchy_h1/h2/h3` `pop` 3줄 제거 → 신규 인제스션부터 metadata에 정상 저장 |
| `backend/api/generation_api.py` | hierarchy 필터 결과 0건 + filename 있으면 filename-only로 자동 재쿼리 (기존 청크 fallback) |

**generation_api.py fallback 로직**

```python
# hierarchy 필터 결과가 없고 filename이 있으면 filename만으로 재쿼리
# (이전에 hierarchy_h1/h2/h3가 metadata에 미저장된 청크 대응)
if not chunks and doc_filename and (h1 or h2 or h3):
    logger.warning(
        f"[{job_id}] hierarchy filter returned 0 chunks — "
        f"retrying filename-only (h1={h1!r}, h2={h2!r}, h3={h3!r})"
    )
    chunks = await get_doc_chunks_by_filter(
        hierarchy_h1=None,
        hierarchy_h2=None,
        hierarchy_h3=None,
        filename=doc_filename,
        limit=max(samples * 3, 30),
    )
```

기존 DB 청크는 재인제스션 없이 즉시 동작.

### 9-2. 해결 방안 — UptimeRobot (추천)

**설정 절차 (3분)**

1. https://uptimerobot.com → 무료 가입
2. Dashboard → `Create Monitor` 클릭
3. Monitor 설정:
   - **Monitor Type**: `HTTP(s)`
   - **URL**: `https://autoeval-uccr.onrender.com/health`
   - **Check Interval**: `5분` (콜드스타트 15분 이전)
   - **Alert contacts**: 비활성화 (모니터링만 목적)
4. 저장

**효과**: 5분마다 자동으로 /health ping → sleep 상태 방지 → 항상 warm 상태

**비용**: 무료

### 9-3. 해결 방안 — GitHub Actions (선택사항)

`.github/workflows/keep-alive.yml` 신규 생성:

```yaml
name: Keep Render Warm

on:
  schedule:
    - cron: '*/5 * * * *'  # 5분마다 실행 (UTC)

jobs:
  ping:
    runs-on: ubuntu-latest
    steps:
      - name: Ping Health Endpoint
        run: curl -f https://autoeval-uccr.onrender.com/health || true
```

**설정 절차**:

1. GitHub 저장소 → `.github/workflows/keep-alive.yml` 파일 생성
2. 위 내용 복사 붙여넣기
3. Commit & Push → 자동 활성화

**효과**: GitHub Actions에서 5분마다 ping 실행

**장점**: 
- 무료 (GitHub Actions 포함)
- 자동화
- Render 외부 모니터링

**단점**:
- GitHub Actions 월별 사용 제한 있음 (private: 2,000분, public: 무제한)
