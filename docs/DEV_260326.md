## 1. 배경 및 목적

### 현재 문제점

현재 Groundedness 평가는 LLM에게 (질문, 답변, 컨텍스트)를 통째로 넘겨 **holistic 판단**을 요청합니다.
이로 인해 reasoning이 generic해지는 경우가 발생합니다.

**현재 reasoning 예시 (문제):**
```
"답변의 모든 내용은 컨텍스트의 '카메라 어플' 설명에서 직접적으로 찾아볼 수 있습니다."
```
→ 섹션 제목만 언급, 실제 컨텍스트 문장 인용 없음

**기대 reasoning 예시 (개선 후):**
```
"두 가지 주장을 검증한 결과, '등록한 얼굴 제외'는 '촬영자가 등록한 얼굴을 제외한
얼굴에 대한 자동 비식별 후 촬영'에 근거하며, '나머지 얼굴 비식별' 역시 동일 문장에서
확인됩니다."
```

> **핵심 설계 원칙**: `claims` 배열은 LLM 내부 CoT 계산용(점수 산출 근거)으로만 사용하고,
> `groundedness_reason`은 claim 검증 결과를 **1~2문장 한국어로 자연스럽게 합성**하여 출력.
> bullet-point 나열이 아닌 prose 형태의 문장으로 재구성하도록 프롬프트에 명시.

## 3. 개선 설계

### 3-1. 아키텍처

```
현재:
(Q, A, Context) → LLM → { groundedness: 0-10, groundedness_reason: "..." }

개선 후:
(Q, A, Context) → LLM → {
    claims: [
        { claim: "...", supported: true,  evidence: "컨텍스트 원문 인용" },
        { claim: "...", supported: false, evidence: null }
    ],
    groundedness: 0-10,
    groundedness_reason: "N개 claim 중 M개 컨텍스트 근거 확인 (Korean)"
}
```

### 3-2. Claim 검증 규칙

| Verdict | 조건 |
|---|---|
| `supported` | 컨텍스트에서 직접 인용 또는 명확히 추론 가능 |
| `unsupported` | 컨텍스트에 없거나 모순되는 내용 |

- Score 반영: `groundedness = supported_count / total_claims × 10`
- 단, LLM 최종 점수가 우선 (claim 비율은 reasoning 근거로 활용)

### 3-3. 기존 출력 구조 호환

```python
# 반환 구조 변경 없음 — 기존 키 유지
{
    "groundedness":        float,   # 0.0~1.0 (기존 동일)
    "groundedness_reason": str,     # 한국어 1문장 (기존 동일)
    # 내부적으로 claims 파싱하여 reason 생성, 외부 인터페이스 변경 없음
}
```

프론트엔드, export, DB 스키마 **변경 없음**.

## 7. relevance / clarity 동일 방식 적용 여부 검토

### 7-1. 지표별 RAGAS 방식 적합성

| 지표 | 측정 대상 | Claim 분해 적합성 | 판단 |
|---|---|---|---|
| **Groundedness** | 답변의 모든 주장이 컨텍스트에 근거하는가 | ✅ 높음 | **적용** |
| **Relevance** | 답변이 질문의 의도를 충족하는가 | ⚠️ 부분 적합 | **검토 필요** |
| **Clarity** | 질문·답변이 명확하고 모호함이 없는가 | ❌ 낮음 | **현행 유지** |

### 7-2. Relevance — 부분 적용 가능

RAGAS의 Answer Relevance는 "question 측에서 역방향 검증"하는 방식 사용:
```
Answer → LLM으로 가상 질문 N개 생성 → 원래 질문과 cosine similarity 측정
```
이는 임베딩 모델이 필요하여 현재 구조에서 비용이 큼.

대신 **Question Aspect 분해** 방식 적용 가능:
```
Question의 핵심 요구사항(aspect) 추출 → 답변이 각 aspect를 충족하는지 검증
→ relevance_reason에 "N개 요구사항 중 M개 충족" prose 합성
```
- Groundedness와 유사한 구조, 방향만 반대 (Context→Answer 대신 Question→Answer)
- **단, 현재 relevance reasoning이 이미 충분히 구체적인 경우가 많으므로 우선순위 낮음**

### 7-3. Clarity — 현행 유지

Clarity는 언어적 품질(모호함, 중의성, 문장 구조)을 평가하는 지표로
claim 분해보다 **holistic 언어 판단**이 더 적합함.
RAGAS/TruLens/DeepEval 모두 별도 clarity 지표를 두지 않는 이유와 동일.
→ 프롬프트 개선 방향이 있다면 "모호한 표현 구체적 지적" 정도로 제한.

### 7-4. 적용 우선순위

```
1순위: Groundedness  →  이번 세션 구현 (claim CoT 하이브리드)
2순위: Relevance     →  추후 검토 (aspect 분해 방식, 필요시)
3순위: Clarity       →  현행 유지
```
