# DEV_260326 — Groundedness 평가 강화 (LLM Judge + RAGAS 하이브리드)

> 작성일: 2026-03-26
> 대상 파일: `backend/evaluators/rag_triad.py`

---

## 1. 배경 및 목적

### 현재 문제점

현재 Groundedness 평가는 LLM에게 (질문, 답변, 컨텍스트)를 통째로 넘겨 **holistic 판단**을 요청합니다.
이로 인해 reasoning이 generic해지는 경우가 발생합니다.

**현재 reasoning 예시 (문제):**
```
"답변의 모든 내용은 컨텍스트의 '카메라 어플' 설명에서 직접적으로 찾아볼 수 있습니다."
```
→ 섹션 제목만 언급, 실제 컨텍스트 문장 인용 없음

**기대 reasoning 예시 (개선 후):**
```
"두 가지 주장을 검증한 결과, '등록한 얼굴 제외'는 '촬영자가 등록한 얼굴을 제외한
얼굴에 대한 자동 비식별 후 촬영'에 근거하며, '나머지 얼굴 비식별' 역시 동일 문장에서
확인됩니다."
```

> **핵심 설계 원칙**: `claims` 배열은 LLM 내부 CoT 계산용(점수 산출 근거)으로만 사용하고,
> `groundedness_reason`은 claim 검증 결과를 **1~2문장 한국어로 자연스럽게 합성**하여 출력.
> bullet-point 나열이 아닌 prose 형태의 문장으로 재구성하도록 프롬프트에 명시.

## 3. 개선 설계

### 3-1. 아키텍처

```
현재:
(Q, A, Context) → LLM → { groundedness: 0-10, groundedness_reason: "..." }

개선 후:
(Q, A, Context) → LLM → {
    claims: [
        { claim: "...", supported: true,  evidence: "컨텍스트 원문 인용" },
        { claim: "...", supported: false, evidence: null }
    ],
    groundedness: 0-10,
    groundedness_reason: "N개 claim 중 M개 컨텍스트 근거 확인 (Korean)"
}
```

### 3-2. Claim 검증 규칙

| Verdict | 조건 |
|---|---|
| `supported` | 컨텍스트에서 직접 인용 또는 명확히 추론 가능 |
| `unsupported` | 컨텍스트에 없거나 모순되는 내용 |

- Score 반영: `groundedness = supported_count / total_claims × 10`
- 단, LLM 최종 점수가 우선 (claim 비율은 reasoning 근거로 활용)

### 3-3. 기존 출력 구조 호환

```python
# 반환 구조 변경 없음 — 기존 키 유지
{
    "groundedness":        float,   # 0.0~1.0 (기존 동일)
    "groundedness_reason": str,     # 한국어 1문장 (기존 동일)
    # 내부적으로 claims 파싱하여 reason 생성, 외부 인터페이스 변경 없음
}
```

## 7. relevance / clarity 동일 방식 적용 여부 검토

### 7-1. 지표별 RAGAS 방식 적합성

| 지표 | 측정 대상 | Claim 분해 적합성 | 판단 |
|---|---|---|---|
| **Groundedness** | 답변의 모든 주장이 컨텍스트에 근거하는가 | ✅ 높음 | **적용** |
| **Relevance** | 답변이 질문의 의도를 충족하는가 | ⚠️ 부분 적합 | **검토 필요** |
| **Clarity** | 질문·답변이 명확하고 모호함이 없는가 | ❌ 낮음 | **현행 유지** |

### 7-2. Relevance — 부분 적용 가능

RAGAS의 Answer Relevance는 "question 측에서 역방향 검증"하는 방식 사용:
```
Answer → LLM으로 가상 질문 N개 생성 → 원래 질문과 cosine similarity 측정
```
이는 임베딩 모델이 필요하여 현재 구조에서 비용이 큼.

대신 **Question Aspect 분해** 방식 적용 가능:
```
Question의 핵심 요구사항(aspect) 추출 → 답변이 각 aspect를 충족하는지 검증
→ relevance_reason에 "N개 요구사항 중 M개 충족" prose 합성
```
- Groundedness와 유사한 구조, 방향만 반대 (Context→Answer 대신 Question→Answer)
- **단, 현재 relevance reasoning이 이미 충분히 구체적인 경우가 많으므로 우선순위 낮음**

### 7-3. Clarity — 현행 유지

Clarity는 언어적 품질(모호함, 중의성, 문장 구조)을 평가하는 지표로
claim 분해보다 **holistic 언어 판단**이 더 적합함.
RAGAS/TruLens/DeepEval 모두 별도 clarity 지표를 두지 않는 이유와 동일.
→ 프롬프트 개선 방향이 있다면 "모호한 표현 구체적 지적" 정도로 제한.

### 7-4. 적용 우선순위

```
1순위: Groundedness  →  이번 세션 구현 (claim CoT 하이브리드)
2순위: Relevance     →  추후 검토 (aspect 분해 방식, 필요시)
3순위: Clarity       →  현행 유지
```

## 1. 문제 요약

| 문제 | 원인 | 영향 |
|------|------|------|
| **텍스트 중복** | `mc:AlternateContent` — `Choice`(현대 Word)와 `Fallback`(구버전 호환) 두 벌이 있는데 현재 `child.iter()`가 둘 다 순회 | `"Ⅰ. 인공지능(AI) 개념 이해Ⅰ. 인공지능(AI) 개념 이해"` 형태로 중복 삽입 |

## 2. 실측 (테스트데이터_2.docx)

- **중복 단락**: 21개 수정 (Fallback 제거 후 0개)
- **테이블 행 절단**: v4 (`\n\n` 구분자) 적용 후 모든 청크 경계가 행 단위로 정렬됨
- `테스트데이터.docx` (DOCX v1): `AlternateContent` 없음 → 영향 없음 (호환성 확인)

## 3. 수정 내용

### 3-1. `_iter_t()` 헬퍼 추가

```python
MC_NS        = "http://schemas.openxmlformats.org/markup-compatibility/2006"
_FALLBACK_TAG = f"{{{MC_NS}}}Fallback"
_WNS_T        = f"{{{WNS}}}t"

def _iter_t(elm):
    """w:t 노드 수집 — mc:Fallback 하위 트리 제외 (AlternateContent 중복 방지)"""
    for child in elm:
        if child.tag == _FALLBACK_TAG:
            continue          # Fallback 전체 스킵
        if child.tag == _WNS_T:
            yield child
        yield from _iter_t(child)
```

적용 위치:
- DOCX 본문 `<w:p>` 텍스트 수집: `child.iter()` → `_iter_t(child)`
- `_cell_text()` 내 `<w:p>` 순회: `p.iter()` → `_iter_t(p)`

### 3-2. `_cell_text()` — 단락별 정규화 + 짧은 단락 병합

```python
def _cell_text(cell_elm) -> str:
    paragraphs = []
    for p in cell_elm.findall(f"{{{WNS}}}p"):
        raw = "".join(node.text or "" for node in _iter_t(p))
        normed = normalize_text(raw.strip())
        if normed:
            paragraphs.append(normed)
    merged = []
    for para in paragraphs:
        if merged and len(merged[-1]) <= 20 and len(para) <= 30:
            merged[-1] = merged[-1] + " " + para
        else:
            merged.append(para)
    return " ".join(merged) if merged else ""
```

### 3-3. `_extract_table()` — 행 구분자 `\n` → `\n\n`

```python
return "\n\n".join(rows_text)   # 기존: "\n".join(rows_text)
```

`\n\n`이 `RecursiveCharacterTextSplitter`의 최우선 구분자이므로 행 경계에서만 분할됨.

# DEV_260326-C — Anchor 선택 개선: Stride → Embedding Clustering 하이브리드

> 대상 파일: `backend/db/doc_chunk_repo.py`, `backend/api/ingestion_api.py`

---

## 1. 배경 및 문제

현재 `sample_doc_chunks` RPC는 **chunk_index 기준 stride 샘플링** (step = total / n).

### stride 샘플링의 한계

| 문제 | 설명 |
|------|------|
| 위치 기반만 | 의미적 유사도 무시 — 반복 구조(표, 조문) 과대 대표 |
| 반복 섹션 낭비 | 유사한 내용 청크를 n 슬롯 여러 개 소비 |
| 밀도 불균등 | 중요 섹션이 짧으면 anchor 미포함 가능 |

### Anchor가 downstream에 미치는 영향

```
anchor chunks
    ├─ H1/H2/H3 taxonomy 생성   ← 다양성 부족 → H1 편향/과잉
    ├─ 전체 청크 태깅 기준       ← taxonomy 품질에 종속
    └─ QA 생성 컨텍스트          ← anchor 편향 → QA 범위 편향
```

---

## 2. stride vs clustering 효율성 비교

### 500청크 문서 예시 (반복 구조 40% 가정)

| | stride n=30 | stride n=150 | clustering (150→30) |
|--|-------------|--------------|---------------------|
| 다양 섹션 | 18개 | 90개 | 27개 (대표) |
| **반복 조문/표** | **12개** | **60개** | **3개** ← 핵심 차이 |
| LLM 입력 토큰 | 30개 분량 | **150개 분량** | 30개 분량 |
| 실질 의미 다양성 | 낮음 | 낮음(비용만 증가) | **높음** |

**stride n 증가의 문제**: 반복 콘텐츠도 비례 증가 → 토큰 비용만 늘고 다양성은 비슷.
**clustering**: 동일 30개 슬롯을 의미적으로 최대한 다르게 채움 → 비용 동일, 품질 향상.

---

## 3. 설계: Stride + Embedding Clustering 하이브리드

### 3-1. 흐름

```
sample_doc_chunks (n=150, embedding 포함)   ← DB 1회 RPC
        │
        ▼
k-means clustering (k=30, sklearn)           ← Python in-memory
        │  각 클러스터에서 중심에 가장 가까운 청크 1개 선택
        ▼
anchor_ids (30개, 의미 다양성 최대화)
        │
        ├─ H1/H2/H3 taxonomy 생성
        └─ QA 생성 컨텍스트
```

### 3-2. 처리 속도 영향

| 단계 | 현재 | 변경 후 |
|------|------|---------|
| sample_doc_chunks RPC | ~300ms (n=30, embedding 없음) | ~800ms (n=150, embedding 포함) |
| k-means (150×1536, k=30) | — | ~300ms |
| H1/H2/H3 LLM 생성 | ~15-25s | ~15-25s (동일) |
| **총합** | **~16-26s** | **~17-27s (+1s, 약 5%)** |

→ 체감 차이 없음.

### 3-3. taxonomy용 vs QA용 샘플 분리

taxonomy와 QA 생성이 **anchor를 같이 쓰는 것**은 구조적 문제:

| 용도 | 필요 특성 |
|------|----------|
| taxonomy 생성 | 넓은 커버리지 (n=100+, 짧은 preview 가능) |
| QA 생성 | 깊은 콘텐츠 (400-800자 필요, n=30-40) |

**분리 방안**:
```python
taxonomy_chunks = cluster_sample(n_fetch=150, k=50)   # 커버리지 우선
qa_anchor_ids   = cluster_sample(n_fetch=150, k=30)   # 콘텐츠 깊이 우선
# → analyze-hierarchy에서 taxonomy_chunks로 H1/H2/H3 생성
# → anchor_ids를 generation_api에 전달 (QA 생성용)
```

```python
    """
    embedding clustering으로 의미적으로 다양한 k개 청크 선택.
    chunks: [{"id": ..., "content": ..., "embedding": [...], ...}]
    """
    if len(chunks) <= k:
        return chunks

    embeddings = np.array([c["embedding"] for c in chunks], dtype=np.float32)
    kmeans = MiniBatchKMeans(n_clusters=k, random_state=42, n_init=3)
    kmeans.fit(embeddings)

    # 각 클러스터에서 중심에 가장 가까운 청크 선택
    selected = []
    for cluster_id in range(k):
        mask = kmeans.labels_ == cluster_id
        cluster_chunks = [c for c, m in zip(chunks, mask) if m]
        if not cluster_chunks:
            continue
        center = kmeans.cluster_centers_[cluster_id]
        cluster_embs = embeddings[mask]
        dists = np.linalg.norm(cluster_embs - center, axis=1)
        best_idx = np.argmin(dists)
        selected.append(cluster_chunks[best_idx])

    return selected
```

# DEV_260326-E — Block-Aware Chunking 도입 (RecursiveCharacterTextSplitter 대체)

> 대상 파일: `backend/ingestion/parsers.py`, `backend/api/ingestion_api.py`

---

## 1. 문제 요약

`RecursiveCharacterTextSplitter(chunk_size=1000)`는 한국 법령 문서 구조를 인식하지 못하여 **문장 중간 하드컷** 발생.

**기존 방식 문제:**
```
# 마지막 부분 잘림 예시
제4조(정보주체의 권리) ... 다음 각 호의 권리를 가진다. <개정 2023. 3.  ← 여기서 1000자 컷
14.>  ← 다음 청크 시작
```

---

## 2. 설계: Block-Aware Chunking

PyMuPDF가 이미 블록(논리 텍스트 유닛) 단위로 추출하므로, 블록 경계에서만 청크를 나눔.

```
기존: blocks → section_text (전체 이어붙임) → RecursiveCharacterTextSplitter (1000자 하드컷)
신규: blocks → chunk_blocks_aware() → 제X조/①②③ 경계 분리 (블록 경계에서만 컷)
```

**분리 우선순위:**
1. `제X조` 시작 → 항상 새 청크 (조 단위 보장)
2. `①②③` 시작 + 현재 청크 ≥ 150자 → 새 청크 (항 단위 분리)
3. `max_chars(900)` 초과 → 다음 블록 경계에서 분리 (문장 중간 컷 없음)

---

## 3. 구현 (`parsers.py`)

```python
_LAW_ARTICLE_RE = re.compile(r'^제\s*\d+조')
_LAW_CLAUSE_RE  = re.compile(r'^[\u2460-\u2473\u24EA\u3251-\u325F]')  # ①-⑳

def chunk_blocks_aware(blocks, max_chars=900, min_chars=150) -> List[str]:
    chunks, buf, buf_chars = [], [], 0

    for block in blocks:
        text = block.get("text", "").strip()
        if not text: continue
        is_article = bool(_LAW_ARTICLE_RE.match(text))
        is_clause  = bool(_LAW_CLAUSE_RE.match(text))

        if is_article and buf:           _flush()
        elif is_clause and buf_chars >= min_chars: _flush()
        elif buf_chars + len(text) > max_chars and buf: _flush()

        buf.append(text); buf_chars += len(text) + 1
    _flush()
    return chunks
```

`ingestion_api.py`에서 `splitter.split_text(section_text)` → `chunk_blocks_aware(sec["blocks"])`로 교체.
`section_text` + `block_page_offsets`는 page 역추적용으로 유지.

## 6. RecursiveCharacterTextSplitter vs Block-Aware Chunking 비교

### 6-1. 법령 문서 (개인정보 보호법)

| 항목 | RecursiveChar | Block-Aware |
|------|--------------|-------------|
| 문장 중간 컷 | ❌ 발생 (`<개정 2023. 3.` / `14.>` 분리) | ✅ 없음 (블록 경계에서만 분리) |
| 조항 경계 인식 | ❌ 없음 (문자 수만 봄) | ✅ 제X조/①②③ 패턴으로 자연 분리 |
| overlap 잔여물 | ❌ 이전 청크 서술어 조각 포함 가능 | ✅ overlap 없음 |

### 6-2. 보고서 문서 (인공지능 영향평가 가이드라인)

| 항목 | RecursiveChar | Block-Aware |
|------|--------------|-------------|
| 소형 섹션 (≤800자) | 동일 | 동일 |
| 표 경계 보존 | ❌ 구분선 행(`\| --- \|`)에서 컷 | ✅ `[표]` 블록 단위 완전 유지 |
| 표 이어지는 청크 시작 | ❌ `\| ① ...` (표 중간 진입, 레이블 없음) | ✅ `[표]\n\| ...` (표 레이블 포함) |
| 대형 표 (2000자+) | 양쪽 모두 큰 청크 (표 분할은 의미 손실이므로 불가피) | |

### 6-3. 핵심 차이 요약

```
RecursiveCharacterTextSplitter
  ├─ 동작: section_text(문자열) → 1000자 단위 컷 → separator 우선 탐색
  ├─ 문제: separator 없으면 문자 단위 강제 컷 → 문장/표 중간 절단
  └─ 한계: 문서 구조(조항번호·표 경계) 무인식

Block-Aware Chunking
  ├─ 동작: blocks(PDF 논리 단위) → 제X조/①/표 경계 감지 → 블록 경계에서만 분리
  ├─ 장점: 문장 항상 완전, 표 경계 보존, 조항 단위 자연 분리
  └─ 한계: 2000자+ 대형 표는 단일 청크 (의도된 동작)
```

### 6-4. DOCX 적용 가능 여부

**결론: DOCX도 `chunk_blocks_aware` 사용.**

```
PDF 블록  = 시각적 한 줄 (레이아웃 단위)  → 제X조/①②③ 감지로 의미 경계 추가
DOCX 블록 = XML <w:p> 단락 하나 (의미 단위) → 이미 의미 경계, 블록 병합만으로 충분
```

DOCX 테스트 (`테스트데이터_2.docx`, `Ⅴ. 참고자료` 섹션 67블록):

| | Block-Aware | RecursiveChar |
|--|-------------|--------------|
| 청크 수 | 8개 | 6개 |
| 1000자 초과 청크 | 없음 ✅ | C3(1091자), C5(1047자) ❌ |
| ①②③ 경계 인식 | ✅ (DOCX에도 원문자 사용 시 자동 감지) | ❌ |

- `ingestion_api.py`는 PDF/DOCX/TXT 구분 없이 `sec["blocks"]`를 넘기므로 별도 분기 불필요
- DOCX에서 ①②③ 패턴이 등장하면 `_LAW_CLAUSE_RE`가 동일하게 처리

### 6-5. 테스트 환경

| | 법령 PDF | 보고서 PDF | 일반 DOCX |
|--|---------|-----------|----------|
| 파일 | 개인정보 보호법(제21445호).pdf | 테스트데이터_3.pdf (25p) | 테스트데이터_2.docx |

## 7. detect_heading / section_path 개선 (2026-03-26 오후)

### 7-1. 문제 요약

`section_path` / `section_title` metadata가 본문 내용을 포함하는 문제 발생:


### 7-2. detect_heading 수정 내역

| 수정 | 내용 |
|------|------|
| `r'^\d+\.'` 패턴 제거 | `1. "개인정보"란...` 등 번호 정의항목 heading 오인 방지 |
| `r'^제\s*\d+\s*조'` 추가 | `제2조(정의)`, `제3조` 등 법률 조문 heading 정상 인식 |
| `r'^\d+\.\d+(\.\d+)?'` 유지 | 보고서 `1.1`, `1.1.1` 섹션 번호는 그대로 heading |
| `^\([가-힣a-zA-Z]` 제외 | `(정확성과 신뢰성)` 등 괄호 주석 소항목 heading 오인 방지 |

예: `"전력계통의효율적운영및"` (공백 없음)

**원인 분석**:
- `get_text("dict")` 모드: span 단위로 텍스트 반환, 폰트 인코딩에 공백 글리프 없으면 문자 연속 출력
- `get_text("rawdict")` 모드: 문자(char) 단위 bbox 포함 → gap 측정으로 공백 복원 가능

**수정 내용**:

1. **`get_text("dict")` → `get_text("rawdict")` 변경**

2. **`_span_text_with_spaces()` 헬퍼 추가**
   - `chars` 배열에서 문자 간 gap 분석
   - gap > 평균 문자 폭 × 0.4 이면 공백 삽입
   - 한글 없는 span은 chars 단순 재구성 (공백 감지 불필요)
   - `chars` 없으면 `span["text"]` fallback (dict 모드 하위 호환)

3. **블록 병합 패턴 개선**

   | 패턴 | 이전 | 이후 |
   |------|------|------|
   | `_KOR_CONT` | `^[가-힣]{1,4}[\s.,。;!?]` | `^[가-힣]{1,4}(?:[\s.,。;!?]\|$)` — 단독 음절도 포함 |
   | `_KOR_TAIL` | `[가-힣]$` | `[가-힣][\s①-⑳]*$` — 원형 번호 뒤에 오는 경우 허용 |

   **원형 번호 삽입 처리**: `"알고리즘개 ③"` + `"발"` → `"알고리즘개발 ③"` (번호 앞에 음절 삽입)

4. **테스트 스크립트 추가**: `test/test_spacing_fix.py`
   - 단위 테스트: chars gap 감지, fallback 동작
   - 통합 테스트: `테스트데이터_8.pdf` 대상 공백 없는 블록 잔존율 측정

### 8-2. 목차 청크 QA 생성 제외 (`backend/ingestion/parsers.py`, `backend/config/prompts.py`)

**문제**: 점선 없는 번호 목차(예: `"목 차 서론 1. 개요 1.1 배경 1.2 ..."`)가 `is_toc_chunk()` 감지를 통과해 DB에 저장되고, QA 생성 시 구조 오해 기반 환각오류 발생.

**수정 내용**:

1. **`is_toc_chunk()` Layer 4, 5 추가** (`parsers.py`)

   | Layer | 감지 조건 |
   |-------|-----------|
   | Layer 4 (신규) | `목\s*차` 키워드 + 섹션 번호(`\d+\.\d+`) 5개 이상 |
   | Layer 5 (신규) | 섹션 번호 밀도 ≥ 5% + 완성 문장(주어+서술어) 2개 이하 |

   검증 결과:
   ```
   목차 텍스트 감지: True   ✓
   정상 본문 감지:   False  ✓ (false positive 없음)
   번호 있는 본문:   False  ✓ (false positive 없음)
   ```

2. **`prompts.py` context_screening 규칙 구체화** (SYSTEM_PROMPT_KO_V1, _CORE_PRINCIPLES_KO 2곳 동일 적용)
   - "목차·제목만 있고 본문 내용이 없는 경우" 항목에 예시 및 판별 기준 추가
   - `"목차 서론 1. 개요 1.1 배경 ..."` 형태를 명시적 예시로 포함
   - LLM 판별 기준 추가: "완성된 서술 문장(주어+서술어 구조)이 없고 항목 나열만 있으면 빈 목록 반환"

> **참고**: 기존에 DB에 저장된 목차 청크는 `backend/scripts/cleanup_toc_chunks.py`로 정리 가능 (업데이트된 `is_toc_chunk` import하여 사용).

### 8-3. 평가 상세 페이지 맥락성(context_relevance) 누락 수정

**문제**: HTML 리포트 및 히스토리 평가 상세 페이지에서 맥락성 항목이 표시되지 않음. (관련성·근거성·완전성만 표시)

**원인**: `context_relevance_reason`이 파이프라인에서는 정상 생성되나 두 경로에서 전달이 끊김.

**데이터 흐름 및 누락 위치**:

```
[pipeline.py]  qa_preview에 context_relevance_reason 포함 ✓
    ↓ 실시간 job status 경로
[React QADetailView]  정상 표시 ✓

[pipeline.py]  평가 결과 Supabase 저장 ✓
    ↓ 히스토리 조회 경로
[evaluation_api.py] _build_export_detail() 누락 ✗ → 수정
    ↓
[React QADetailView]  historyQaPreview 경로 맥락성 없음 → 해결

[QAEvaluationDashboard.tsx] detailedQA 매핑 누락 ✗ → 수정
    ↓
[exportUtils.ts → HTML export]  맥락성 렌더링 로직은 OK, 데이터 없음 → 해결
```

**수정 파일 및 내용**:

| 파일 | 수정 내용 |
|------|-----------|
| `backend/api/evaluation_api.py` | `_build_export_detail()` 내 `"context_relevance_reason": r.get("context_relevance_reason", "")` 추가 |
| `frontend/src/components/evaluation/QAEvaluationDashboard.tsx` | `detailedQA` 매핑에 `context_relevance_reason: q.context_relevance_reason` 추가 |

> `exportUtils.ts`의 렌더링 로직(`clarity_reason` → 명확성 / `context_relevance_reason` → 맥락성 분기)은 원래 정상이었음.
