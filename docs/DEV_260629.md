# 개발 세션 노트 (2026-06-29)

> Render 인제스션 실패 로그 기준 Gemini Embedding 2 quota/routing 진단 및 보안 로그 정리

---

## 1. 배경

Render 로그에서 PDF 인제스션은 LLM 청킹까지 성공했지만, 임베딩 단계에서 429로 중단되었다.

```text
Quota exceeded for aiplatform.googleapis.com/online_prediction_requests_per_base_model
with base model: gemini-embedding-2
```

해당 파일은 청크 수가 32개였고, 현재 임베딩 배치 크기 기본값은 64라서 임베딩 API 요청은 1회만 발생하는 상황이었다.

---

## 2. 판단

이번 429는 단순한 API key 불일치나 요청 폭주보다는 **Vertex AI base model별 online prediction quota** 문제로 보는 것이 타당하다.

확인한 조건:

- Render에 등록된 `GOOGLE_API_KEY`와 `backend/.env`의 `GOOGLE_API_KEY`는 동일하다고 확인됨
- 코드 표준 환경변수는 `GOOGLE_API_KEY`로 유지
- 에러 메시지에 `aiplatform.googleapis.com`와 `online_prediction_requests_per_base_model`가 명시됨
- 32 chunks / batch size 64 조건이라 앱 내부 동시성으로 대량 호출한 케이스가 아님

우선 확인할 운영 환경변수:

```text
GOOGLE_API_KEY
GOOGLE_GENAI_USE_VERTEXAI
GOOGLE_CLOUD_PROJECT
GOOGLE_CLOUD_LOCATION
GOOGLE_CLOUD_REGION
EMBED_MODEL
EMBED_BATCH_SIZE
EMBED_CONCURRENCY
```

`GOOGLE_GENAI_USE_VERTEXAI=true`인 경우 Google Cloud 콘솔에서 Gemini API quota가 아니라 Vertex AI quota를 확인해야 한다. 특히 프로젝트, 리전, base model `gemini-embedding-2`가 Render 실행 환경과 일치해야 한다.

---

## 3. 코드 변경

### 3.1 Google API key 변수명 통일

파일: `backend/api/ingestion_api.py`

- `GOOGLE_API_KEY`만 읽도록 정리
- 기존 내부 변수명 `GEMINI_API_KEY` 제거
- 키 미설정 시 `GOOGLE_API_KEY not set for Google Gemini client.` 경고 로그 출력

### 3.2 키 동일성 확인용 key_id 로그

파일: `backend/api/ingestion_api.py`

API key 원문 노출 없이 동일성만 확인하도록 SHA-256 기반 비가역 ID를 로그에 남긴다.

```python
def _secret_id(secret: str) -> str:
    return hashlib.sha256(secret.encode("utf-8")).hexdigest()[:10]
```

로그 예시:

```text
Google Gemini client initialized (GOOGLE_API_KEY key_id=xxxxxxxxxx)
```

주의:

- 원문 key는 로그에 출력하지 않는다
- Render와 로컬의 `key_id`가 같으면 동일 key 주입 여부만 확인할 수 있다

### 3.3 Vertex 라우팅 진단 로그

파일: `backend/api/ingestion_api.py`

`GOOGLE_GENAI_USE_VERTEXAI=true`일 때만 Vertex 라우팅 로그를 남기도록 했다.

```text
Google GenAI Vertex routing enabled (project=..., location=...)
```

`"false"` 문자열이 설정된 경우를 enabled로 오인하지 않도록 정확히 `"true"`일 때만 출력한다.

### 3.4 임베딩 quota 에러 메시지 개선

파일: `backend/ingestion/pipeline.py`

- 기존 "비용 소진" 표현을 "quota/rate limit"으로 수정
- `aiplatform.googleapis.com` 또는 `online_prediction_requests_per_base_model`가 포함되면 Vertex AI base-model quota 확인 힌트를 추가
- `EMBED_MODEL`, `EMBED_BATCH_SIZE`를 환경변수로 조절 가능하게 변경

기본값:

```text
EMBED_MODEL=gemini-embedding-2
EMBED_BATCH_SIZE=64
EMBED_CONCURRENCY=5
```

2026-06-29 추가 확인:

- 공식 Gemini API 문서 기준 최신 Embedding 2 모델명은 `gemini-embedding-2`
- `task_type` 설정 대신 입력 문자열에 `task`, `query`, `title`, `text` prefix를 포함
- `gemini-embedding-2`는 여러 contents 입력 시 개별 청크 벡터가 아니라 aggregated embedding 동작이 될 수 있으므로 청크별로 호출

### 3.5 프론트 번들 키 주입 제거

파일: `frontend/vite.config.ts`

프론트에서 `process.env.GOOGLE_API_KEY`를 define으로 주입하던 설정을 제거했다.

이유:

- Google API key는 백엔드 서버 환경변수로만 사용해야 함
- Vite `define`은 빌드 결과물에 값이 들어갈 수 있어 브라우저 노출 위험이 있음
- 현재 프론트 `src`에서 해당 값을 직접 참조하지 않음

## 4. 검증

실행한 검증:


결과:

- 컴파일 성공
- `frontend/src`에서 `process.env.GOOGLE_API_KEY` / `process.env.GEMINI_API_KEY` 직접 참조 없음

## 5. 다음 확인 사항

Render 재배포 후 로그에서 아래를 확인한다.

```text
Google Gemini client initialized (GOOGLE_API_KEY key_id=...)
Google GenAI Vertex routing enabled (project=..., location=...)
```

판단 기준:

- `key_id`가 로컬과 같으면 Render에 동일 key가 주입된 것
- Vertex routing 로그가 뜨면 Google Cloud 콘솔에서 해당 project/location의 Vertex AI quota를 확인
- Vertex routing 로그가 없는데도 `aiplatform.googleapis.com` quota가 뜨면 SDK 또는 배포 환경의 추가 설정을 재점검
