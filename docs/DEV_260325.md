## 배경 — 발견된 핵심 문제

실제 생성된 QA 사례:

```
컨텍스트:
[시행 2025.10.1.] [대통령령 제35811호, 2025.10.1., 타법개정]
식품의약품안전처 (식품안전정책과) 043-719-2032, 2014 ...

생성된 Q: "대통령령 제35811호의 구체적인 시행 일자는 언제입니까?"
생성된 A: "해당 법령의 시행일은 2025년 10월 1일입니다."

RAG Triad: 1.000 / QA Quality: 1.000
```

**점수는 완벽하지만 교육적 가치가 없는 trivial QA** — 시스템 전반적 설계 문제.

#### 문제 P1 — 행정 메타 청크가 anchor에 포함 → H1 오염

```python
# 현재
anchor_chunks = await get_doc_chunks_sampled(request.filename, n=30)
concatenated_text = "\n\n---\n\n".join(c["content"][:600] for c in anchor_chunks)
```

균등 샘플링이면 문서 앞부분(표지, 시행일, 연락처)이 필연적으로 포함됨.
이 청크들이 anchor에 들어가면 LLM이 H1을 "시행일관리", "담당기관" 같은 메타 카테고리로 생성.

**해결방안**
```python
# anchor_chunks 수신 후, 생성 전에 행정 메타 패턴 필터
_ADMIN_PATTERN = re.compile(
    r'^\s*(\[시행|\[대통령령|시행일|고시번호|\d{2,4}-\d{3,4}|\d{2,3}-\d{3,4}-\d{4})',
    re.MULTILINE
)
def _is_admin_chunk(content: str) -> bool:
    short = len(content.strip()) < 300
    has_admin = bool(_ADMIN_PATTERN.search(content[:200]))
    return short and has_admin

anchor_chunks = [c for c in anchor_chunks if not _is_admin_chunk(c["content"])]
# 필터 후 부족하면 get_doc_chunks_sampled 재호출 (n=40)로 보충
```

#### 문제 P2 — 600자 절단 → LLM 내용 오판

```python
c["content"][:600]  # 법령 조문 1개가 평균 400~800자 → 중간에 잘림
```

조문이 잘리면 LLM이 주제를 부분적으로만 보고 H1을 결정.
특히 조문 도입부(제목)와 내용이 분리되면 오판 가능성 높음.

**해결방안**
```python
# 600 → 900자로 상향 (18,000 → 27,000자 한도 내 처리 가능)
# 단, 총 입력이 20,000자 초과하면 anchor를 20개로 줄임
MAX_ANCHOR_CHARS = 20000
per_chunk_limit = max(400, MAX_ANCHOR_CHARS // len(anchor_chunks))
concatenated_text = "\n\n---\n\n".join(c["content"][:per_chunk_limit] for c in anchor_chunks)
```

#### 문제 P3 — H2 개수 고정 → 문서 크기와 무관

```
현재 constraints:
"For each H1, create 2~5 H2 sub-categories"
```

법령/시행령은 조문 구조상 H2 6~8개가 자연스럽고,
단순 매뉴얼은 2~3개면 충분한데 억지로 5개 생성됨.

**해결방안**
```
constraints 동적 조정:
- 전체 청크 수 ≤ 50:  "H1당 H2 2~3개"
- 전체 청크 수 51~150: "H1당 H2 2~5개" (현재)
- 전체 청크 수 > 150:  "H1당 H2 3~7개"

구현:
chunk_count = len(anchor_chunks)  # 전체 청크 수를 별도 조회 또는 추정
h2_guide = "2~3" if chunk_count <= 50 else ("3~7" if chunk_count > 150 else "2~5")
```

#### 문제 P4 — H1 생성 제약 불충분

```
현재:
"H1 must represent content themes or domains — NOT section titles or headings."
```

"content themes"의 범위가 모호 → 행정 식별자가 H1이 되는 걸 막지 못함.

**해결방안**
```
constraints에 추가:
"H1 must NOT be: dates, ordinance numbers, contact info, department names, page headers."
"H1 must be learnable — a reader should be able to study or apply it."
"Bad H1 examples: '시행일 관리', '담당부서', '고시번호'"
"Good H1 examples: '데이터 연계 기준', '정보시스템 운영', '보안 정책'"
```

#### 문제 T1 — 행정 메타 청크 강제 분류 (탈출구 없음)

```python
# 현재 프롬프트:
"Select H1, H2, H3 values EXCLUSIVELY from master_hierarchy. Do NOT generate new values."
```

표지/연락처/시행일 청크도 master의 어딘가에 반드시 태깅됨.
→ "전자정부법 > 데이터연계 > 연계절차"에 "시행 2025.10.1." 청크가 붙는 구조.

**해결방안**
```
constraints에 추가:
"If chunk contains ONLY: dates, phone numbers, ordinance numbers, department names,
 or page headers — set h1='__admin__', h2='__admin__', h3='__admin__'."
"Use __admin__ sparingly — only when chunk has NO learnable content."

이후 generation_api._should_skip():
meta = chunk.get("metadata", {})
if meta.get("hierarchy_h1") == "__admin__":
    return True  # QA 생성 제외
```

```python
# 현재 (h2_h3_master 없을 때)
"H2/H3: Korean, under 15 characters each"
# → 배치마다 다른 H2/H3 자유 생성
```

```
② 불가피하면 fallback에서도 H2/H3 고정 목록 강제
   - h2_h3_master 없을 때 → apply-granular-tagging 호출 자체를 프론트에서 차단
```

#### 문제 T3 — 5개 배치 동시 태깅 → 상호 영향

```python
batch_size = 5
# 비슷한 청크 5개를 한 프롬프트에 → LLM이 상호 참조해 임의 분류
```

특히 유사한 조문 내용이 같은 배치에 들어가면 LLM이 구분을 포기하고 같은 H2/H3 배정.

**해결방안**
```
① 배치 내 청크 독립성 강화 — 프롬프트에 명시
   "Classify each chunk INDEPENDENTLY. Do not compare chunks within this batch."

② 배치 크기 조정 검토
   - 현재 5 → 배치 크기 줄이면 정밀도 상승하나 LLM 호출 수 증가
   - 단기: 프롬프트 명시로 해결 시도
   - 중기: 청크 내용 유사도 기반 배치 분리 (같은 섹션 청크는 다른 배치)
```

#### 문제 T4 — section_path/section_title 선택적 전달 → 문맥 부족

```python
# 현재
if sp:
    entry["section_path"] = sp
if st:
    entry["section_title"] = st
# → 없으면 content[:800]만 전달
```

법령 조문은 조항 번호(제X조 ①②③) 없이 내용만 보면 분류 어려움.

**해결방안**
```python
# section_path가 없어도 chunk_type과 char_length는 항상 포함
entry = {
    "idx": i,
    "content": c["content"][:800],
    "chunk_type": meta.get("chunk_type", "unknown"),
    "char_length": meta.get("char_length", len(c["content"])),
}
if sp: entry["section_path"] = sp
if st: entry["section_title"] = st
# chunk_type으로 LLM이 "이건 표/목록/본문"을 파악해 H3 선택에 참고
```

### 1-4. document_id 기반 청크 버전 관리

> 2026-03-25 테스트 중 발견 — 같은 파일 재업로드 시 이전 청크와 현재 청크가 혼재되는 문제

#### 문제

```
1차 업로드 → document_id=AAA, 청크 3개 저장
2차 업로드 → document_id=BBB, 청크 2개 (hash 동일 → INSERT skip)
DB 상태   → AAA 청크 3개만 존재

apply-granular-tagging → filename 전체 조회 → AAA 청크 3개 태깅
QA 생성                → filename 조회 → 3개 대상 (2개 예상인데)
```

#### 잘못된 접근 (롤백)

```
delete_chunks_by_filename() 호출 → 이전 청크 전체 삭제 후 새로 삽입
문제: qa_gen_results / qa_eval_results가 chunk_id 참조 → 히스토리 단절
```

#### 채택한 접근 — document_id 추적

```
매 업로드: doc_id = uuid4() → metadata.document_id 에 저장 (기존)

analyze-hierarchy:
  anchor_chunks[0].metadata.document_id 추출 → 응답에 포함

apply-granular-tagging:
  request.document_id 수신
  get_document_chunks(filename, document_id=...) → 해당 버전만 태깅

결과:
  이전 버전(AAA) 청크 + QA + 평가 → DB 보존 (히스토리 유지)
  현재 버전(BBB) 청크만 태깅·생성·평가 대상
```

### 1-5. trivial 청크 차단 — 3단계 방어 구조

> 특정 패턴 매칭만으로는 문서 유형이 달라지면 동작하지 않음.
> 코드·프롬프트·LLM 판단을 계층적으로 조합해 어느 단계가 놓쳐도 다음이 포획.

```
[1단계] 앵커 필터 — _is_admin_anchor() (내용 밀도 기반, 코드 레벨)
  - 한글 비율 < 30% 또는 완성 문장 0개인 400자 미만 청크 제거
  - H1 생성 LLM에 전달되는 anchor에서 선제 차단
  - 특정 키워드 패턴 미사용 → 법령/매뉴얼/보고서 등 문서 유형 무관

[2단계] H1 제약 강화 — analyze-hierarchy 프롬프트 (프롬프트 레벨)
  - "H1 must be LEARNABLE — NOT administrative metadata"
  - Bad/Good H1 예시 명시
  - 1단계를 통과한 메타 청크가 있어도 LLM이 H1 생성 시 무시

[3단계] 태깅 __admin__ — apply-granular-tagging 프롬프트 (LLM 판단)
  - 모든 청크에 대해 LLM이 직접 "learnable content 없으면 __admin__ 배정"
  - 1·2단계를 통과한 trivial 청크 최종 포획
  - __admin__ 태깅 청크 → generation_api._should_skip() 에서 QA 생성 제외 (예정)
```

| 단계 | 위치 | 놓칠 수 있는 경우 | 다음 단계가 커버 |
|------|------|-----------------|----------------|
| 1 앵커 필터 | `_is_admin_anchor()` | 한글 많은 표지 청크 | 2단계 H1 제약 |
| 2 H1 제약 | `analyze-hierarchy` prompt | LLM이 메타 H1 생성 | 3단계 __admin__ |
| 3 태깅 LLM | `_build_prompt()` | LLM 분류 오류 | generation_api skip (예정) |

## 2. QA 평가 프롬프트 문제점 및 개선 방향

### 2-1. Layer 2 — RAG Triad

**파일**: `rag_triad.py` — `evaluate_all_with_reasons()`

#### 현재 문제점

| # | 문제 | 증거 |
|---|------|------|
| R1 | `relevance` 기준: "질문에 직접 답하는가" — 질문 자체가 trivial해도 답변이 맞으면 10점 | trivial QA relevance=10 |
| R2 | `groundedness` "flexible matching" — 단순 날짜 답변도 근거 추적 가능하면 10점 | trivial QA groundedness=10 |
| R3 | `clarity` 기준: "명확하고 모호하지 않은가" — 단문 답변이 오히려 명확해서 10점 | trivial QA clarity=10 |
| R4 | 세 지표 모두 **답변이 컨텍스트에 맞는가**만 체크 — 질문 자체의 가치 평가 없음 | 전반적 설계 한계 |

#### 개선 방향

```
① relevance 루브릭에 질문 가치 기준 추가
   - 현재: "질문에 직접 답하는가"
   - 개선: "질문 자체가 해당 문서의 핵심 내용을 다루는가?
           단순 식별자(날짜·번호·연락처) 확인만 하는 질문은 relevance 5 이하"

② groundedness에 "답변 깊이" 기준 추가
   - 현재: "모든 주장이 컨텍스트에 근거"
   - 개선 추가: "답변이 단순 값 반환(날짜 1개, 번호 1개)에 그치면 groundedness 6 이하
               (사실은 맞지만 컨텍스트의 핵심 내용을 충분히 반영하지 않음)"

③ 또는: relevance를 두 가지로 분리
   - answer_relevance: 답변이 질문에 맞는가 (현재)
   - question_value: 질문이 문서의 핵심을 다루는가 (신규)
   - 이 경우 가중치 재조정 필요
```

### 2-2. Layer 3 — QA Quality

**파일**: `qa_quality.py` — `evaluate_all()`

#### 현재 문제점

| # | 문제 | 증거 |
|---|------|------|
| Q1 | `specificity`: "0-3: Generic filler language" — 단순 날짜 답변은 generic이 아니어서 10점 | "2025년 10월 1일" → specificity=10 |
| Q2 | `completeness`: factoid 루브릭 "포괄적 답변 = 10" — 단문 답변도 질문에 완전히 답하면 10점 | trivial factoid completeness=10 |
| Q3 | `conciseness`: "5문장 이내 반복 없음" — 단문 답변이 오히려 최고점 | trivial QA conciseness=10 |
| Q4 | 4개 지표 전부 **답변 품질**만 평가 — 질문-답변 쌍의 교육적 가치 평가 없음 | 전반적 설계 한계 |

#### 개선 방향

```
① specificity 루브릭 강화 — "교육적 구체성" 개념 추가
   - 현재: "구체적인 수치/절차/항목 사용"
   - 개선:
     - 10: 요건·메커니즘·효과를 구체적 근거와 함께 설명
     - 6-7: 핵심 사실 포함, 부분적 설명
     - 3-4: 단순 값 하나만 반환 (날짜·번호 등) — 사실은 맞지만 맥락 이해 기여 낮음
     - 0-2: 모호하거나 무의미

② completeness에 "맥락 기여도" 추가
   - 현재: "intent 유형별 루브릭 (열거 = 10 등)"
   - 개선 추가 (factoid/definition 유형):
     "단순 식별자(날짜·번호·기관명)만 답변하면 completeness 5 이하.
      조문의 의미·적용 방법·제한 조건 중 하나 이상 포함해야 10점 가능."

③ 신규 지표 검토: educational_value (교육적 가치)
   - 0-10: 이 QA를 학습하면 문서의 어떤 개념/요건을 이해할 수 있는가?
   - 1-3: 단순 사실 확인 (날짜·번호)
   - 4-6: 하나의 개념/요건 이해
   - 7-10: 복수 개념 연결, 적용·판단 능력 향상
   - 단, 추가 시 점수 공식 재조정 필요 (현재 가중치 합 = 1.0 고정)
```

## 5. Intent 분류 체계 검토

### 5-1. 현재 구조의 문제

현재 8개 intent가 만들어내는 실제 답변 패턴:

| intent | 전형적 답변 패턴 | 인지 수준 |
|--------|----------------|----------|
| factoid | "X는 Y입니다." | Level 1 — 단순 회상 |
| definition | "X란 Y를 말합니다." | Level 1-2 |
| how | "X는 Y 방법으로 합니다." | Level 2 |
| numeric | "기준은 N개입니다." | Level 1 — 수치 조회 |
| procedure | "1. A 2. B 3. C" | Level 2 |
| why | "X를 위해 필요합니다." | Level 2 |
| list | "A, B, C입니다." | Level 1 — 열거 |
| boolean | "네/아니오, ~합니다." | Level 2 |

**공통 문제**: 모든 intent가 단일 사실 추출 → 단문 답변에 최적화되어 있음.

### 5-2. 현재 intent로 커버 안 되는 패턴

법령/규정 문서에서 핵심적인 질문 패턴이 누락되어 있음:

#### 누락 패턴 ①: 조건/예외 (condition / exception)
```
컨텍스트:
"전자적 시스템을 연계하여야 한다. 다만, 공동이용센터를 활용하는 것이
연계·통합보다 효율적인 경우에는 공동이용센터를 활용하여야 한다."

현재 생성 가능한 질문:
- list: "전자적 시스템 연계 시 정해야 할 사항을 나열하세요"
- boolean: "전자적 시스템 연계는 의무인가?"

커버 안 되는 질문:
- "공동이용센터를 대신 활용할 수 있는 조건은 무엇인가?"  ← condition
- "원칙적으로 연계·통합을 해야 하지만 예외가 허용되는 경우는?" ← exception
```

#### 누락 패턴 ②: 비교 (comparison)
```
컨텍스트:
"중앙행정기관은 30일 이내, 지방자치단체는 60일 이내에 처리하여야 한다."

현재 생성 가능한 질문:
- numeric: "중앙행정기관의 처리 기한은?"  (각각 따로 생성)
- numeric: "지방자치단체의 처리 기한은?"

커버 안 되는 질문:
- "중앙행정기관과 지방자치단체의 처리 기한이 다른 이유는?" ← comparison + why
- "두 기관의 처리 기한을 비교하면?" ← comparison
```

#### 누락 패턴 ③: 적용/판단 (application)
```
컨텍스트:
"행정기관등의 장은 정보시스템을 폐기하려는 경우 행정안전부장관에게
사전에 통보하여야 한다."

현재 생성 가능한 질문:
- factoid: "정보시스템 폐기 시 통보 대상은?"
- boolean: "정보시스템 폐기 전 통보는 의무인가?"

커버 안 되는 질문:
- "A 기관이 정보시스템을 폐기하려 할 때 취해야 할 조치는?" ← application (특정 상황 적용)
```

### 5-3. 실제 분포 문제

이전 평가 결과에서 확인된 intent 분포:
```
list    20.5%  ─────────────
boolean 19.3%  ────────────
factoid  ~15%  ─────────
how      ~12%  ────────
definition~10% ──────
why       ~8%  ─────
numeric   ~8%  ─────
procedure ~7%  ────
```

`list + boolean = 39.8%` 지배 원인:
1. diversity_rules가 "우선 선택 50% 초과 금지"를 강제 → 조건부 선택 채워야 함
2. `why`, `condition`, `exception`은 컨텍스트 근거 부족 시 skip
3. `list`, `boolean`은 거의 모든 컨텍스트에서 근거 찾기 쉬움
4. → 결과적으로 diversity_rules가 list/boolean 과잉 유발

### 5-4. 확정 Intent 체계 — 6개

**통합 내역**
- `factoid` + `definition` + `numeric` → `fact` (사실형) — 답변 형태 동일, LLM 구분 불필요
- `how` + `procedure` → `how` (방법형) — 순서 있으면 방법형 안에서 단계 제시
- `boolean` + `exception` → `condition` (조건형) — 단서·예외 조항 포함
- `comparison` (비교형) — 신규 추가
- `why` → `purpose` (원인형) — 목적·배경까지 확장
- `list` (열거형) — 유지, 상한 1개

---

#### ① 사실형 `fact`

```
정의: 조문의 적용대상·요건·효과·범위에 관한 사실 확인
금지: 시행일·고시번호·전화번호 등 식별자 단순 조회
reasoning:
  1) 근거확인 — 해당 조항이 무엇을 규정하는가 (조문 내용 직접 추출)
  2) 범위해석 — 적용 대상·조건·범위가 구체적으로 어디까지인가
예시 Q: "전자적 시스템 연계를 요청할 수 있는 기관의 범위는?"
```

#### ② 원인형 `purpose`

```
정의: 조문의 목적·이유·입법 취지 (명시된 목적 또는 효과로부터 유추 가능한 경우)
reasoning:
  1) 근거확인 — 해당 규정의 내용
  2) 목적해석 — 어떤 필요/문제를 해결하기 위한 규정인가
예시 Q: "행정안전부장관이 연계 사업에 재정 지원을 할 수 있도록 한 이유는?"
```

#### ③ 방법형 `how`

```
정의: 특정 행위의 구체적 방법·기준·절차 (순서 있으면 단계 포함)
reasoning:
  1) 근거확인 — 무엇을 어떻게 해야 하는가 (조문 직접 추출)
  2) 핵심분석 — 방법·기준의 핵심 조건 또는 단계 해석
예시 Q: "전자적 시스템 연계 요청 시 포함해야 할 구체적 사항은?"
       "정보시스템 폐기 절차는 어떻게 되는가?" (순서 있는 경우)
```

#### ④ 조건형 `condition`

```
정의: 조건 분기·단서 조항·예외 처리
      ("다만", "~의 경우에는", "~에도 불구하고", 금지/제한 포함)
reasoning:
  1) 원칙확인 — 기본 규정은 무엇인가
  2) 조건분석 — 어떤 조건에서 예외·분기·제한이 발생하며, 판단 기준은 무엇인가
예시 Q: "전자적 시스템 연계 대신 공동이용센터를 활용할 수 있는 조건은?"
```

#### ⑤ 비교형 `comparison`

```
정의: 두 개 이상의 대상·기관·조건·의무를 비교
조건: 컨텍스트에 비교 가능한 두 대상이 명시된 경우
reasoning:
  1) 대상확인 — 비교 대상 A와 B 각각의 내용 확인
  2) 차이분석 — 어떤 점이 다른가 (의무 수준·기간·조건 차이)
예시 Q: "중앙행정기관의 장과 행정안전부장관의 역할은 어떻게 구분되는가?"
```

#### ⑥ 열거형 `list`

```
정의: 복수의 항목·유형·요건을 열거 (순서 없음)
상한: 청크당 최대 1개
reasoning:
  1) 항목확인 — 열거된 항목이 몇 개이며 각각 무엇인가
  2) 분류해석 — 항목들의 공통 기준 또는 누락 없이 완전한지 확인
예시 Q: "전자적 시스템 연계 요청 시 정해야 하는 사항을 모두 나열하면?"
```

---

### 5-5. diversity_rules 재설계

```
현재 문제:
"우선 선택(factoid+definition+how) 합산 50% 초과 금지"
→ 조건부 선택 강제 → list/boolean 억지 삽입 (39.8% 점유)

개선:

① fact + list 합산 40% 초과 금지
   (단순 사실조회·열거 과잉 방지)

② condition 또는 comparison 중 1개 이상 권장
   (컨텍스트에 "다만"/비교 근거 있을 때 필수)

③ 개별 상한
   - fact: 최대 2개/청크
   - list: 최대 1개/청크

④ 전체 QA 수: 컨텍스트 밀도 기반 2~6개
   (내용 없으면 0개 허용)
```

### 5-6. 프롬프트 구성 전략 — 3단계 분리

예시를 시스템 프롬프트에 직접 넣으면 도메인 고착 문제 발생.
(예: 전자정부법 예시 → 다른 문서에서도 동일 패턴 반복)

```
① 시스템 프롬프트  → intent 정의 + reasoning 라벨만 (예시 없음)
② 유저 템플릿     → JSON 출력 스키마 형식 가이드 1개 (내용 비워둠)
③ adaptive 빌더  → domain_profile.key_terms 기반 동적 예시 생성
```

**① 시스템 프롬프트 — 정의만**
```
- fact (사실형): 조문의 적용대상·요건·효과·범위 확인. 식별자 단순 조회 금지.
  reasoning: [근거확인, 범위해석]
- condition (조건형): 단서·예외·조건 분기. "다만", "~의 경우에는" 패턴.
  reasoning: [원칙확인, 조건분석]
...
```

**② 유저 템플릿 — 형식 few-shot (내용 없음)**
```json
<output_example>
{
  "q": "질문 텍스트",
  "reasoning": [
    "근거확인: [조문에서 직접 추출한 핵심 내용]",
    "분석: [조건·범위·차이에 대한 해석]"
  ],
  "a": "답변 텍스트",
  "intent": "condition"
}
</output_example>
```

**③ adaptive 빌더 — 동적 예시**
```python
# domain_profile.key_terms = ["전자정부", "행정기관", "정보시스템"] 기반
fact_example       = f"{key_terms[0]}에서 {key_terms[1]}의 역할은?"
condition_example  = f"{key_terms[0]} 적용 시 예외가 인정되는 조건은?"
comparison_example = f"{key_terms[1]}과 {key_terms[2]}의 의무 차이는?"
# 문서마다 다른 예시 → 도메인 고착 방지
```

### 6-1. 사례1 — 평가 레이어 간 모순 (factual error)

#### 입력 데이터

```
컨텍스트 (YOLOv8 성능 표):
| 클래스 | Precision | Recall | mAP50 | mAP50-95 |
|--------|-----------|--------|-------|----------|
| face   |   0.982   | 0.983  | 0.995 |  0.712   |

Q: "YOLOv8 모델의 face 클래스 mAP50 수치는?"
A: "0.995입니다."
```

#### 평가 결과

| 레이어 | 지표 | 점수 | 판정 |
|--------|------|------|------|
| Layer 2 — RAG Triad | groundedness | 0.667 | "답변의 0.995는 mAP50 컬럼 값과 일치하나 컨텍스트에서 명확히 특정되지 않음" |
| Layer 3 — QA Quality | factuality | 1.000 | "답변 사실과 컨텍스트 일치" |

#### 문제

```
두 레이어가 동일 QA에 대해 모순된 판정:
  - Layer 2(groundedness=0.667): 답변 근거 추적에서 불확실성 탐지
  - Layer 3(factuality=1.000): 값이 컨텍스트에 존재한다는 이유만으로 만점

근본 원인:
  - factuality 루브릭이 "값이 컨텍스트에 존재하는가"만 검증
  - 테이블에서 0.995는 실제로 mAP50 컬럼에 있으므로 사실로 인정됨
  - 그러나 질문에서 요구한 맥락(어느 행/열인지)과 정확히 대응하는지는 검증하지 않음
  - 결과적으로 RAG Triad는 모호성을 잡아냈지만, Quality는 표면적 일치만 확인

부가 문제:
  - 두 레이어 점수가 크게 다를 때(≥0.3 차이) 경고 없음
  - final_score = rag×0.4 + quality×0.4 → 두 모순 점수가 평균 처리됨 (0.667+1.000)/2*0.8 ≒ 0.67
```

#### 개선 방향

```
① factuality 루브릭 강화 — "값 존재 여부"에서 "컨텍스트 내 정확한 대응" 확인으로 변경
   현재: "답변의 모든 사실이 컨텍스트에 있는가?"
   개선: "답변의 각 사실이 컨텍스트의 어느 부분에 근거하는가?
         테이블인 경우 행·열 특정이 가능한가?"

② 레이어 간 모순 감지 로직 추가 (pipeline.py)
   조건: abs(groundedness - factuality) > 0.3
   처리: qa_preview에 contradiction_flag: true 표시
         primary_failure에 "evaluation_contradiction" 추가

③ groundedness 점수가 낮으면 factuality 재평가 트리거 검토
   (단, 추가 LLM 호출 비용 고려 — 선택적 적용)
```

### 6-2. 사례2 — 열거형 trivial 답변 만점

#### 입력 데이터

```
컨텍스트 (성능 지표 테이블 헤더):
| Precision | Recall | mAP50 | mAP50-95 |

Q: "성능 지표의 종류를 모두 기술하십시오."
A: "Precision, Recall, mAP50, mAP50-95입니다."
```

#### 평가 결과

| 레이어 | 지표 | 점수 |
|--------|------|------|
| Layer 2 — RAG Triad | relevance / groundedness / clarity | 1.000 |
| Layer 3 — QA Quality | factuality / completeness / specificity / conciseness | 1.000 |

#### 문제

```
테이블 헤더를 그대로 옮긴 단순 복사 수준 답변이 전 지표 만점:

  - relevance=1.000: 질문과 컨텍스트 모두 "성능 지표" 관련 → 통과
  - groundedness=1.000: 답변의 4개 항목이 컨텍스트 헤더에 그대로 존재 → 통과
  - specificity=1.000: "구체적인 항목 나열" 기준 충족 → 통과
  - completeness=1.000: 열거형 intent는 "모두 포함하면 완전" → 통과

핵심 문제:
  ① 질문 자체가 trivial (테이블 헤더 나열 = 사실 학습 불필요)
  ② 답변이 컨텍스트 복사에 가까워도 평가 지표에서 패널티 없음
  ③ list intent의 completeness 루브릭이 "항목 망라"만 요구 — 이해 수준 무관

list intent의 본질적 한계:
  - 컨텍스트에 항목이 나열되어 있으면 무조건 생성 가능
  - 답변도 컨텍스트 복사로 완성 → 인지 수준 Level 1 고착
  - 5-5의 diversity_rules "list 최대 1개/청크"도 이 패턴을 근본 해결하지 않음
```

#### 개선 방향

```
① list intent 금지 패턴 명시 (prompts.py SYSTEM_PROMPT)
   금지: "헤더/메타 항목(컬럼명, 레이블, 번호 목록) 그대로 나열"
   허용: "의미 있는 기능·요건·절차 항목 열거 (각 항목이 설명 가능한 개념)"

② list completeness 루브릭 개선 (qa_quality.py)
   현재: "열거된 항목이 컨텍스트의 모든 관련 항목을 포함하면 10점"
   개선: "열거된 항목이 각각 독립적인 개념을 나타내며, 단순 레이블이 아닌 경우 10점
         컬럼명·헤더·항목 번호를 그대로 나열한 경우 5점 이하"

③ relevance 루브릭 개선 (rag_triad.py)
   현재: "질문이 컨텍스트와 관련 있는가"
   개선: "질문이 컨텍스트의 핵심 개념을 다루는가 — 메타/식별자 나열은 5점 이하"

④ 생성 단계 차단 (5-4 ① fact 규칙과 동일 원칙 list에도 적용)
```

## 7. 평가 지표 통합 분석

> 2026-03-25 — RAG Triad(3) + Quality(4) → 4개 통합 설계

### 7-1. 현재 7개 지표 기능 분류

| 지표 | 레이어 | 평가 축 | 평가 입력 |
|------|--------|---------|---------|
| **relevance** | RAG | Q ↔ A 주제 일치 (도메인 내) | Q, A, Context |
| **groundedness** | RAG | A ↔ Context 사실 추적 가능성 (CoT) | A, Context |
| **clarity** | RAG | Q+A 표현 명확성 + 구조화 여부 | Q, A |
| **factuality** | Quality | A ↔ Context 사실 일치 여부 | Q, A, Context |
| **completeness** | Quality | 질문 요구사항을 intent별로 완전히 충족했는가 | Q, A, intent |
| **specificity** | Quality | 답변이 구체적인가 vs 모호/generic한가 | Q, A, Context |
| **conciseness** | Quality | 답변 길이가 질문 유형에 적절한가 | Q, A, intent |

### 7-2. 중복 분석

| 제외 지표 | 중복 대상 | 근거 |
|----------|---------|------|
| **factuality** (제외) | groundedness | 둘 다 "답변 ↔ 컨텍스트 사실 검증". groundedness가 CoT 기반으로 더 엄격 |
| **conciseness** (제외) | clarity | clarity가 이미 "properly structured" 포함. 길이 측정의 부가가치 낮음 |
| **specificity** (제외) | — | RAG와 중복 없으나, depth 강제는 생성 프롬프트(depth 원칙 + diversity_rules)에서 이미 처리 |

### 7-3. 최종 확정 4개

```
relevance    — Q ↔ A 주제 일치 (도메인 내)
groundedness — A ↔ Context 사실 추적 (CoT, 가장 엄격)
clarity      — Q+A 표현 명확성 + 구조화 (conciseness 흡수)
completeness — intent별 답변 완전성: how→절차 완비, list→항목 망라, condition→조건 포함 등
```

```python
# rag + quality 완료 후 순회
for q_score in quality_layer["qa_scores"]:
```

## 9. 텍스트 추출·청킹 개선 — opendataloader-pdf 활용 계획

> 상세 설계: `docs/DEV_260324.md` (벤치마크 결과·단계별 착수 조건 포함)

### 8-1. 현재 문제와의 연결점

오늘 적용한 `_PAREN_CONT` 블록 병합 패치 (예: "(Return on Investment)", "Distortion)" 청크 앞 출현)는
PyMuPDF의 **block 단위 텍스트 추출** 구조에서 비롯된 증상이다.
PyMuPDF는 PDF의 렌더링 블록 경계를 그대로 청크 경계로 사용하기 때문에,
괄호 속 영문 병기 구절이 다음 블록 시작에 잔존하는 현상이 발생한다.

opendataloader-pdf는 **XY-Cut++ reading order + MHS heading 감지**로 이런 구조적 절단을 방지한다.

### 8-2. opendataloader-pdf 핵심 우위

| 항목 | PyMuPDF (현재) | opendataloader-pdf |
|------|---------------|-------------------|
| Heading 감지 | 0개 (폰트 heuristic 추정만) | 15개 유효 (노이즈 4종 필터 후) |
| Table 분리 | 수동 `\|` 패턴 의존 | 7개 자동 분리 |
| Header/Footer | heuristic 반복 탐지 | 9/9개 자동 분리 |
| Bounding box | 부분적 | 116/116 100% |
| 이미지 전용 페이지 | 건너뜀 (12/15페이지) | 전 요소 기록 (15/15) |
| 멀티컬럼 reading order | 미보장 | XY-Cut++ 보장 |

> 실측: 테스트데이터\_1.pdf 기준, 텍스트 추출량 PyMuPDF 대비 +5% (재귀 기준 동등)

### 8-3. Hierarchy 개념 혼동 주의

| 구분 | 현재 시스템 H1/H2/H3 | opendataloader heading_level |
|------|---------------------|------------------------------|
| 종류 | Semantic (내용 카테고리) | Structural (레이아웃 수준) |
| 예시 | H1: "서비스소개", H2: "요금정책" | H1: "1장 개요", H2: "1.2 배경" |
| 부여 주체 | LLM (Gemini) | opendataloader 자동 감지 |
| 용도 | QA stratification 필터 | 청크 경계 결정 |

→ **대체 관계 아님 — 보완 관계**: structural heading으로 청크 경계를 잡고, semantic H1/H2/H3로 분류

### 8-4. 단계별 마이그레이션 경로

| 단계 | 내용 | 선행 조건 |
|------|------|---------|
| **1단계** | PyMuPDF → opendataloader 파서 교체 (LLM/DB 파이프라인 유지) | ① p11 이상 현상 원인 확인 (목차 표 흡수 추정) ② Dockerfile JVM(`default-jre-headless`) 빌드 테스트 |
| **2단계** | structural heading 기반 `section_path` 개선, LLM 태깅 입력 품질 향상 | 1단계 완료 + 5개 문서 heading 정확도 실측 |
| **3단계** | heading-aware chunking (고정 크기 RecursiveTextSplitter 대체) | 2단계 완료 + A/B 테스트 |
| **4단계** | opendataloader structural heading을 H1/H2 master 생성 anchor로 활용 | 3단계 완료 |

> **1단계 착수 전 확인 필수**: `test/test_opendataloader_pdf.py` 에서 p11 원시 요소 출력 확인
> (목차 전체가 `table` 타입으로 흡수되어 텍스트 커버리지 36%로 떨어지는 현상 원인 파악)

**completeness 선택 이유**: relevance/groundedness/clarity가 커버하지 않는 유일한 축 — "질문의 요구사항을 얼마나 완전히 충족했는가(breadth)". intent-specific 루브릭과 6종 체계 시너지.
