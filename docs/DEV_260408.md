#### 장기 (선택적 개선) — 우선순위 정리

**1순위 — `api.ts` 반환 타입 구체화** ✅ 완료 (260408)

현재 `apiFetch<T>`가 `ApiResponse<T>` (`{ success, data?: T }`)를 반환하지만 백엔드는 flat JSON을 돌려줌 → 코드 전반에 `as unknown as` 캐스팅이 남아있고, 이번 white screen 버그의 구조적 원인이기도 함.

**방향 결정: Option A (프론트 수정) 채택**

백엔드 전체가 `{ success, ...fields }` flat 패턴으로 일관성 있게 동작 중 (`data` 키 사용하는 엔드포인트 없음). 백엔드를 수정할 명분이 없으므로 프론트 타입만 현실에 맞게 정정.

**수정 계획 (`src/lib/api.ts`)**

| 단계 | 변경 내용 |
|------|-----------|
| 1 | `ApiResponse<T = any>` 제네릭 제거 → `ApiResponse { success, error?, status_code? }` (data 필드 삭제) |
| 2 | `apiFetch<T extends ApiResponse>(...): Promise<T>` — 반환 타입을 `T`로 직접 반환 |
| 3 | `apiFetchWithRetry<T extends ApiResponse>(...): Promise<T>` — 동일 패턴 적용 |
| 4 | 각 엔드포인트 함수의 `as Promise<XxxResponse>` 캐스팅 제거 (제네릭으로 자동 추론) |
| 5 | `getHierarchyList`의 `as unknown as { ... }` 제거 — `apiFetch<HierarchyListResponse>` 직접 지정 |

**기대 효과**
- `as unknown as` / `as Promise<T>` 캐스팅 전체 제거
- 응답 필드 접근 시 tsc가 정확하게 타입 체크
- Zod 도입 시 각 인터페이스를 스키마로 대체하기만 하면 됨

이 작업이 선행돼야 Zod 도입 실익이 생김.

**2순위 — `ErrorBoundary` 세분화**

현재 `<main>` 전체를 단일 boundary로 래핑 → generation 탭 오류가 evaluation 탭까지 날림. 탭별로 분리하면 오류 격리 가능. 작업량 대비 효과가 가장 큼.

**3순위 — Zustand**

`taggingTreeData`, `currentFilename`, `taggingVersion`이 `App.tsx → DataStandardizationPanel → QAGenerationPanel` prop 체인으로 드릴링 중. 컴포넌트 수가 적어 현재는 허용 범위지만, `DataStandardizationPanel` 컴포넌트 분리 작업 전에 먼저 도입하면 분리 후 props 정리 비용이 줄어듦.

**후순위 — 라우팅 / Zod**

| 항목 | 이유 |
|------|------|
| 라우팅 | 딥링크 요구 없이 현재 탭 구조로 충분. 페이지 수 증가 시점에 도입 |
| Zod | api.ts 타입 구체화 선행 필요 + 스키마 정의 비용이 높아 단독 도입 실익 낮음 |
