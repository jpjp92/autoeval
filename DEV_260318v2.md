## 2. DB 테이블 구조 & 연계

### doc_chunks
```
id              uuid  PK
content         text        청크 원문 (최대 1200자)
metadata        jsonb       {filename, document_id, hierarchy_l1/l2/l3,
                             section_title, chunk_type, page, keywords,
                             content_hash, embedding_model, ingested_at}
embedding       vector(1536) Gemini Embedding 2 (output_dimensionality=1536, L2 정규화)
created_at      timestamptz
```
> ※ HNSW 인덱스 2000 dims 제한으로 vector(1536) 유지. 코드도 동일하게 맞춰져 있음.

### qa_gen_results
```

job_id               text        gen_YYYYMMDD_HHMMSS_ffffff
created_at           timestamptz
updated_at           timestamptz
metadata             jsonb       {generation_model, lang, prompt_version, source_doc}
stats                jsonb       {total_qa, total_documents, tokens, cost}
qa_list              jsonb       [{docId, hierarchy:[l1,l2,l3], text, metadata,
                                   qa_list:[{q,a,intent}]}]
linked_evaluation_id uuid        → qa_eval_results.id (평가 완료 후 업데이트)
source_doc           text        doc_chunks 참조 파일명 (직접 컬럼)
```

> ※ `hierarchy` 컬럼 제거됨 (2026-03-18). qa_list JSONB 내부에는 hierarchy 배열 유지.

### qa_eval_results

```
created_at       timestamptz
```

## 3. 세션(Job) 관리 구조

### 3-1. Generation Job (in-memory)

**위치**: `backend/api/generation_api.py` — `JobManager` 클래스

```python
class GenerationJob:
    job_id:       str          # gen_YYYYMMDD_HHMMSS_ffffff
    status:       JobStatus    # pending | running | completed | failed
    progress:     int          # 0–100
    message:      str
    error:        Optional[str]
    result_file:  Optional[str]  # 로컬 JSON 파일명
    result_id:    Optional[str]  # Supabase qa_gen_results.id
    config:       Dict           # 생성 설정 (model, lang, samples, hierarchy)
    started_at:   str
    completed_at: Optional[str]

class JobManager:
    jobs: Dict[str, GenerationJob]  # 메모리 딕셔너리
    lock: threading.Lock            # 스레드 안전
```

**생명주기**:
```
POST /api/generate
  → JobManager.create_job(job_id)
  → BackgroundTasks.add_task(run_generation_job, job_id)
  → 반환: {job_id}

프론트 폴링: GET /api/generate/{job_id}/status (1초 간격)
  → JobManager.get_job(job_id) → {status, progress, message, result_file}

완료 후:
  → result_file → Supabase save → result_id(UUID) 저장
  → Job 메모리에 유지 (서버 재시작 시 소멸)
```

### 3-2. Evaluation Job (in-memory)

**위치**: `backend/evaluators/job_manager.py` — `EvaluationManager` 클래스

```python
class EvalJob:
    job_id:         str           # eval_YYYYMMDD_HHMMSS_ffffff
    result_filename: str
    status:         EvalJobStatus # pending | running | completed | failed
    progress:       int           # 0–100
    message:        str
    error:          Optional[str]
    eval_report:    Optional[Dict]  # 완료 시 전체 평가 결과
    timestamp:      str
    generation_id:  Optional[str]   # qa_gen_results.id (export용)
    layers_status:  Dict            # 4레이어 각각의 상태
      └── {syntax, stats, rag, quality}
              └── {status, progress, message}

class EvaluationManager:
    jobs: Dict[str, EvalJob]
    lock: threading.Lock
```

**생명주기**:
```
POST /api/evaluate
  → EvaluationManager.create_job(job_id)
  → BackgroundTasks.add_task(run_evaluation, job_id, ...)
  → 반환: {job_id}

프론트 폴링: GET /api/evaluate/{job_id}/status (1초 간격)
  → {status, progress, message, layers, eval_report}

완료 후:
  → eval_report → Supabase save → supabase_eval_id
  → link_generation_to_evaluation(generation_id, eval_id)
  → Job 메모리에 유지 (서버 재시작 시 소멸)
```

### 3-3. 세션 관리 한계 (현재)

| 항목 | 현황 | 문제 |
|------|------|------|
| 저장소 | Python 프로세스 메모리 (`Dict`) | 서버 재시작 시 모든 Job 소멸 |
| 영속성 | Supabase (완료된 결과만) | 진행 중 Job은 재시작 후 복구 불가 |
| 확장성 | 단일 프로세스 | 다중 워커(uvicorn workers > 1) 시 Job 공유 불가 |
| 정리 | 없음 | Job 무한 누적 (메모리 누수 가능) |

**Production 고려 시**: Redis + Celery 또는 Supabase jobs 테이블로 이전 필요

## 3-5. Hierarchy L2/L3 개선 계획 — 3-Pass Master 생성 (예정)

### 문제점 (현재)

현재 `apply-granular-tagging`은 청크마다 L2/L3를 **자유 생성**해서 청크 수만큼 L2가 늘어나는 문제 발생.
- L1은 문서 전체 기준 master list 사전 확정 후 선택 → 일관성 유지
- L2/L3는 청크별 개별 생성 → 중복·유사 값 난립

### 개선 방향: Method 3 (L2+L3 master 동시 생성)

```
1단계: analyze-hierarchy (기존 유지)
  ├── 입력: 전체 청크 샘플 (20~30개)
  ├── 처리: LLM → 문서 전체 주제 파악
  └── 출력: L1 master list (3~5개)
           예: ["규제 준수", "기술 개발", "시장 분석"]

           ↓ L1 master list 전달

2단계: analyze-l2-l3 (신규 엔드포인트)
  ├── 입력: L1 master list + 전체 청크 샘플
  ├── 처리: LLM 1회 호출 → L2 목록 + L3 후보 동시 생성
  └── 출력: L1별 L2 목록 + L2별 L3 후보
           {
             "규제 준수": {
               "법적 요건": ["요건 정의", "적용 범위"],
               "심사 기준": ["평가 항목", "합격 기준"],
               "위반 사례": ["사례 유형", "제재 내용"]
             },
             "기술 개발": {
               "핵심 기능": ["기능 설명", "구현 방식"],
               "성능 지표": ["측정 기준", "달성 목표"]
             }
           }
           → 프론트에 반환 (사용자 검토/수정 가능)

           ↓ L1/L2/L3 master dict 전달

3단계: apply-granular-tagging (개선)
  ├── 입력: 각 청크 content + L1/L2/L3 master dict
  ├── 처리: LLM은 생성 X, 선택만
  │         - L1: master list 중 택1
  │         - L2: 선택한 L1의 L2 목록 중 택1
  │         - L3: 선택한 L2의 L3 후보 중 택1
  │              (없으면 가장 유사한 것 선택)
  └── 출력: doc_chunks.metadata 일괄 업데이트
```

### 수량 제한 기준

| 레벨 | 문서당 수량 | 비고 |
|------|------------|------|
| L1 | 3~5개 | 기존과 동일 |
| L2 | L1당 2~5개 | 신규 제약 |
| L3 | L2당 2~4개 | 신규 제약 |

```
Step 3: 5/5 배치, 21/21 청크 전부 매칭 완료
```

### 엣지 케이스 처리

- L3 후보 중 맞는 것 없을 시 → "가장 유사한 것 선택" 프롬프트 지시
- 특정 L2에 청크가 0개 → 정상 (문서에 해당 주제 없는 것)
- L2/L3 후보가 너무 적을 시 → 2단계 재실행 또는 수동 추가

### 프롬프트 XML 태그 전환 완료 (2026-03-18)

Gemini 3 공식 가이드 기준 — `###` 마크다운 + f-string JSON 혼재 구조 → XML 태그 방식 전환.
Claude / GPT-5 모델도 동일 XML 포맷 효과적으로 처리 → 모델 교체 시 프롬프트 재작성 불필요.

**공통 XML 구조:**
```
<role>      — 역할 정의 + 핵심 제약 (맨 앞 배치)
<constraints> — 수량·형식 제약 명시
<l1_master> / <master_hierarchy> / <context> — 데이터 영역 (지시문과 분리)
<chunks>    — 입력 데이터
<task>      — 구체적 지시 + 출력 포맷 (맨 끝 배치)
```

**프롬프트별 변경 내용:**

| 프롬프트 | 핵심 변경 |
|----------|-----------|
| `analyze-hierarchy` | `<role>` + `<constraints>` 앞 배치, 문서 샘플 `<context>`, 지시 `<task>` 끝 |
| `analyze-l2-l3` | `<l1_master>` 태그로 L1 데이터 분리, 수량 제약 `<constraints>` 명시 |
| `_build_prompt` (master 모드) | `<role>`에 grounding 강화 ("verbatim" 선택 명시, 생성 금지), `<master_hierarchy>` / `<chunks>` / `<task>` 완전 분리 |
| `_build_prompt` (fallback 모드) | 동일 XML 구조 적용 (하위 호환 유지) |

**Gemini 3 가이드 적용 포인트:**
- 핵심 제약(`<role>`, `<constraints>`)을 프롬프트 맨 앞 배치
- 데이터와 지시문 완전 분리 → LLM이 데이터를 지시로 오해하는 오류 방지
- grounding 지시 추가: `master_hierarchy` 외 값 생성 금지, verbatim 선택 강제
- `<task>` 맨 끝 배치 (가이드 권장: 긴 컨텍스트 후 지시는 맨 끝)

#### GPT-5 특화 인사이트

GPT-5 prompting guide에서 우리 파이프라인에 적용 가능한 항목.

**1. `reasoning_effort` 파라미터**
GPT-5 API 호출 시 태스크 난이도에 따라 조절.
```python
# 단순 태깅 (apply-granular-tagging)
reasoning_effort = "low"   # 빠른 응답, 비용 절감

# 복잡한 QA 생성 / 평가
reasoning_effort = "high"  # 멀티스텝, 정확도 우선
```

**2. `<context_gathering>` 패턴 — 탐색 범위 제어**
QA 생성 프롬프트에서 LLM이 불필요하게 넓게 추론하지 않도록 제한:
```xml
<context_gathering>
- 청크 내용만 기반으로 QA 생성할 것
- 외부 지식 추론 금지, 제공된 텍스트에서만 근거 추출
- 청크당 최대 QA 생성 수: {qa_per_doc}개
</context_gathering>
```

**3. 태스크 분리 — 턴당 1개 태스크**
GPT-5 공식: 복잡한 태스크는 여러 턴으로 분리 시 성능 최대화.
→ 현재 파이프라인 이미 준수: Ingestion → Generation → Evaluation 각 독립 API 호출 ✅

**4. stop conditions 명시**
평가 프롬프트에서 명확한 중단 조건 정의:
```xml
<stop_conditions>
- QA 쌍이 {qa_per_doc}개 완성되면 즉시 반환
- 근거를 찾을 수 없는 질문은 생성하지 말 것
- 불확실한 경우 생성 중단, 생성된 것만 반환
</stop_conditions>
```

### qa_quality.py 변환 계획

#### 핵심 쟁점 3가지

**쟁점 1 — `_call_llm_combined` Anthropic 분기 버그**

현재 OpenAI는 messages 배열로 system/user 분리되어 있지만, Anthropic(Claude) 분기는 user 프롬프트에 sys_msg를 문자열로 합쳐서 전달합니다.

```python
# OpenAI 분기 — 이미 분리됨 ✅
messages=[
    {"role": "system", "content": sys_msg},
    {"role": "user",   "content": prompt},
]

# Anthropic 분기 — 합쳐서 user로 전달 ❌
response = self.judge_model.invoke(f"{sys_msg}\n\n{prompt}")
# → Claude가 sys_msg를 role 지시가 아닌 일반 텍스트로 처리할 수 있음
```

**쟁점 2 — `evaluate_all` 데이터-지시 혼재**

현재 `[CONTEXT]` 내용과 `[SCORING DIMENSIONS]` 채점 기준이 같은 평문 안에 나열되어 있습니다.
모델이 CONTEXT 내용을 지시로 오인하거나, OUTPUT 포맷 지시를 무시하는 케이스가 발생할 수 있습니다.

XML 분리 기준:
```
<context>       ← 입력 데이터 (모델이 생성/수정 불가 영역)
<question>      ← 평가 대상 데이터
<answer>        ← 평가 대상 데이터
<scoring_dimensions>  ← 채점 기준 (지시 영역)
<task>          ← 출력 포맷 명시 (맨 끝 배치)
```

**쟁점 3 — `sys_msg` XML 구조화 (선택적 강화)**

현재 sys_msg는 단순 평문입니다. `<role>` + `<output_format>` 태그로 구조화하면
Claude / Gemini 모두 system prompt 처리 시 지시와 포맷을 더 명확히 구분합니다.

#### evaluate_all — 프롬프트 본문

```
# 현재
[CONTEXT] {clean_ctx} [QUESTION] {q} [ANSWER] {a}
[SCORING DIMENSIONS] 1. factuality ... 2. completeness ... 3. groundedness ...
[OUTPUT — valid JSON only] {...}

# 변환 후
<context>
{clean_ctx[:3500]}
</context>

<question>{question}</question>
<answer>{answer}</answer>

<scoring_dimensions>
1. factuality (0-10): Are the answer's claims factually consistent with the context?
   - 10: All claims match context semantically
   - 6-7: Core claims match, minor gaps
   - 0-3: Claims contradict or cannot be derived from context

2. completeness (0-10): Does the answer fully address the question?
   - 10: Comprehensive, covers all aspects
   - 6-7: Addresses main point, lacks some detail
   - 0-3: Barely addresses the question

3. groundedness (0-10): Can the answer be derived/inferred from the context?
   - 10: All claims clearly derivable from context
   - 6-7: Core claims derivable, minor unsupported details
   - 0-3: Most claims cannot be traced to context
</scoring_dimensions>

<task>
Evaluate the QA pair on all three dimensions. Score each 0-10.
Return ONLY valid JSON:
{"factuality": <0-10>, "completeness": <0-10>, "groundedness": <0-10>}
</task>
```

#### _call_llm_combined — system 파라미터 분리 (Anthropic + Gemini 통합)

OpenAI는 이미 `messages=[{system}, {user}]` 형태로 분리됨 → 변경 없음.
Anthropic / Gemini 분기를 `[SystemMessage, HumanMessage]` 리스트로 통일합니다.

```python
# 현재 (Anthropic 분기) — 잘못됨 ❌
response = self.judge_model.invoke(f"{sys_msg}\n\n{prompt}")

# 개선 (Anthropic + Gemini 공통 적용) ✅
# langchain_core.messages는 ChatAnthropic / ChatGoogleGenerativeAI 모두 지원
from langchain_core.messages import SystemMessage, HumanMessage

elif self.judge_model:
    response = self.judge_model.invoke([
        SystemMessage(content=sys_msg),
        HumanMessage(content=prompt),
    ])
```

> `ChatGoogleGenerativeAI`도 `SystemMessage` 처리 지원 → 동일 패턴으로 Gemini / Claude 분기 통합 가능

| `USER_TEMPLATE_KO_V1` | `[섹션]` → XML 태그 전환, JSON 8개 반복 블록 → 1개 예시 + 범위 안내 | ✅ |
| `USER_TEMPLATE_EN_V1` | 동일 XML 전환 + 범위 안내 | ✅ |

**핵심 동작 변경:**
- "각 유형 정확히 1개씩" → **근거 있는 유형만 선택** + `<diversity_rules>`로 편중 방지

| 의도 유형 | 컨텍스트 요건 | 없을 경우 |
|-----------|--------------|-----------|

```
5. 컨텍스트 부족 시: ...

# 변환 후
<role>
당신은 문서 기반 QA 데이터셋 생성 전문가입니다.
주어진 컨텍스트만을 근거로 독자가 실제로 물어볼 법한 질문과 답변을 생성합니다.
</role>

<principles>
1. 근거성(Groundedness): 모든 질문은 반드시 제공된 컨텍스트 내에서 명확한 답변이 가능해야 합니다.
   - 금지: "안내에 따르면 그렇습니다" (순환논리)
   - 필수: 컨텍스트에서 직접 인용 또는 명시된 이유/설명 제시
2. 관련성(Relevance): 질문과 답변이 주제적으로 일치해야 합니다.
3. 원자성(Atomicity): 질문 하나는 하나의 개념/과업만 묻습니다. 복합 질문 금지.
4. 명확성: 대명사·광범위한 표현을 피하고, 답의 범위가 분명한 질문을 작성하세요.
</principles>

<intent_types>
- factoid:   구체적인 사실/정보 확인 — 대부분의 컨텍스트에서 생성 가능
- definition: 개념/용어의 정의 설명 — 대부분의 컨텍스트에서 생성 가능
- how:       작동 방식/구체적 방법 — 대부분의 컨텍스트에서 생성 가능
- numeric:   구체적 수치/비율/개수 — 컨텍스트에 숫자/수치가 명시된 경우만
- procedure: 단계별 절차/방법 — 컨텍스트에 순서/단계가 명시된 경우만
- why:       근본적인 이유/원인 — 컨텍스트에 이유/목적이 명시된 경우만 ("정책이다"는 불가)
- list:      전체 목록/옵션 나열 — 컨텍스트에 복수 항목이 나열된 경우만
- boolean:   예/아니오 판단 — 컨텍스트에 조건/규칙이 명시된 경우만
</intent_types>

<constraints>
- 근거 없는 의도 유형은 건너뛰고, 근거 있는 다른 유형으로 대체합니다
- 생성 수량: 최소 {min_qa}개 ~ 최대 {n_qa}개 (컨텍스트 밀도에 따라 조정)
- 8개 유형 모두 포함 불필요. 동일 유형은 최대 2개까지 허용
- 우선 선택 그룹(factoid/definition/how)이 전체의 절반을 초과하면 안 됩니다
- 조건부 그룹 중 컨텍스트에 근거 있는 유형은 반드시 1개 이상 포함
- "N/A" 또는 답변 불가 표시 금지
- 언어: 한국어
</constraints>
```

- `n_qa` 기본값 8 유지 → 충분한 컨텍스트면 최대 8개

#### 2. 의도 유형 선택 규칙 명시 — 유연성 + 다양성 균형

현재 "각 유형 정확히 1개씩" → **"근거 있는 유형만 선택하되, 편중 금지"**로 변경

**설계 원칙:**
- 근거 없는 유형 억지 생성 금지 (유연성)
- 특정 유형에 편중 금지 (다양성)
- 조건부 유형이라도 컨텍스트에 근거가 있으면 반드시 포함 (다양성 강제)

```
<intent_types>
[우선 선택 그룹] 대부분의 컨텍스트에서 생성 가능:
  - factoid:   구체적인 사실/정보 확인
  - definition: 개념/용어의 정의 설명
  - how:       작동 방식/구체적 방법

[조건부 선택 그룹] 컨텍스트에 해당 근거가 있을 때만 선택:
  - numeric:   수치/비율/기간 등 구체적 숫자가 명시된 경우
  - procedure: 단계별 순서/절차가 명시된 경우
  - why:       명시적 이유/목적/근거가 서술된 경우 ("정책이다"는 불가)
  - list:      복수의 항목/유형이 나열된 경우
  - boolean:   예/아니오 판단 가능한 조건/규칙이 명시된 경우
</intent_types>

<diversity_rules>
1. 동일 유형 최대 2개까지 허용 (factoid×3 이상 금지)
2. 우선 선택 그룹(factoid+definition+how)이 전체 QA의 절반을 초과하면 안 됩니다
   예: 6개 생성 시 우선 그룹 최대 3개, 조건부 그룹 최소 3개
3. 조건부 그룹 중 컨텍스트에 근거가 있는 유형은 반드시 1개 이상 포함하세요
</diversity_rules>

<constraints>
- 근거 없는 유형은 건너뛰고 다른 유형으로 대체
- 생성 수량: {min_qa}~{n_qa}개 (컨텍스트 밀도에 따라 조정)
- "N/A" 또는 답변 불가 표시 금지
</constraints>
```

**예시 시나리오:**

| 컨텍스트 특성 | 선택 가능 유형 | 6개 생성 예 |
|--------------|--------------|------------|
| 수치·절차·이유 모두 있음 | 전체 8개 | factoid, numeric, procedure, why, how, definition |
| 수치만 있고 절차·목록 없음 | factoid/def/how/numeric/why/boolean | factoid, definition, how, numeric×2, why |
| 정의·사실 위주 컨텍스트 | factoid/def/how/why | factoid×2, definition×2, how, why |
