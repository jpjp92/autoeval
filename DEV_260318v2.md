  (doc.value ->> 'docId') AS doc_id,
  (doc.value -> 'hierarchy') AS hierarchy,   -- qa_list JSONB 내부 필드 (컬럼 아님)

  (qa.value ->> 'a') AS answer,
  (qa.value ->> 'intent') AS intent,

**evaluation_qa_joined**

## 2. DB 테이블 구조 & 연계

### doc_chunks
```
id              uuid  PK
content         text        청크 원문 (최대 1200자)
metadata        jsonb       {filename, document_id, hierarchy_l1/l2/l3,
                             section_title, chunk_type, page, keywords,
                             content_hash, embedding_model, ingested_at}
embedding       vector(1536) Gemini Embedding 2 (output_dimensionality=1536, L2 정규화)
created_at      timestamptz
```
> ※ HNSW 인덱스 2000 dims 제한으로 vector(1536) 유지. 코드도 동일하게 맞춰져 있음.

### qa_gen_results
```
id                   uuid  PK
job_id               text        gen_YYYYMMDD_HHMMSS_ffffff
created_at           timestamptz
updated_at           timestamptz
metadata             jsonb       {generation_model, lang, prompt_version, source_doc}
stats                jsonb       {total_qa, total_documents, tokens, cost}
qa_list              jsonb       [{docId, hierarchy:[l1,l2,l3], text, metadata,
                                   qa_list:[{q,a,intent}]}]
linked_evaluation_id uuid        → qa_eval_results.id (평가 완료 후 업데이트)
source_doc           text        doc_chunks 참조 파일명 (직접 컬럼)
doc_chunk_ids        uuid[]      사용된 청크 ID 배열 (GIN 인덱스, 역추적용)
```
> ※ `hierarchy` 컬럼 제거됨 (2026-03-18). qa_list JSONB 내부에는 hierarchy 배열 유지.

### qa_eval_results
```
id               uuid  PK
job_id           text        eval_YYYYMMDD_HHMMSS_ffffff
created_at       timestamptz
updated_at       timestamptz
metadata         jsonb       {generation_model, evaluator_model, lang,
                              prompt_version, source_doc, generation_id}
total_qa         int4
valid_qa         int4
scores           jsonb       {syntax.pass_rate, stats.quality_score,
                              rag.avg_*, quality.avg_*}
final_score      float8      0–1 종합 점수 (syntax×0.2 + stats×0.2 + rag×0.3 + quality×0.3)
final_grade      text        A+/A/B+/B/C/F
pipeline_results jsonb       4레이어 전체 상세 결과 (qa_scores 포함)
```
> ※ `interpretation` 컬럼 제거됨 (2026-03-18).

### 연계 관계
```
doc_chunks.id
  ← qa_gen_results.doc_chunk_ids[]    (GIN, 역추적용)
  ← qa_gen_results.qa_list[*].docId  (JSONB 내부, 상세 참조)

qa_gen_results.id
  ← qa_eval_results.metadata.generation_id   (평가→생성 참조)

qa_gen_results.linked_evaluation_id
  → qa_eval_results.id                        (생성→평가 참조)
```

## 3. 세션(Job) 관리 구조

### 3-1. Generation Job (in-memory)

**위치**: `backend/api/generation_api.py` — `JobManager` 클래스

```python
class GenerationJob:
    job_id:       str          # gen_YYYYMMDD_HHMMSS_ffffff
    status:       JobStatus    # pending | running | completed | failed
    progress:     int          # 0–100
    message:      str
    error:        Optional[str]
    result_file:  Optional[str]  # 로컬 JSON 파일명
    result_id:    Optional[str]  # Supabase qa_gen_results.id
    config:       Dict           # 생성 설정 (model, lang, samples, hierarchy)
    started_at:   str
    completed_at: Optional[str]

class JobManager:
    jobs: Dict[str, GenerationJob]  # 메모리 딕셔너리
    lock: threading.Lock            # 스레드 안전
```

**생명주기**:
```
POST /api/generate
  → JobManager.create_job(job_id)
  → BackgroundTasks.add_task(run_generation_job, job_id)
  → 반환: {job_id}

프론트 폴링: GET /api/generate/{job_id}/status (1초 간격)
  → JobManager.get_job(job_id) → {status, progress, message, result_file}

완료 후:
  → result_file → Supabase save → result_id(UUID) 저장
  → Job 메모리에 유지 (서버 재시작 시 소멸)
```

### 3-2. Evaluation Job (in-memory)

**위치**: `backend/evaluators/job_manager.py` — `EvaluationManager` 클래스

```python
class EvalJob:
    job_id:         str           # eval_YYYYMMDD_HHMMSS_ffffff
    result_filename: str
    status:         EvalJobStatus # pending | running | completed | failed
    progress:       int           # 0–100
    message:        str
    error:          Optional[str]
    eval_report:    Optional[Dict]  # 완료 시 전체 평가 결과
    timestamp:      str
    generation_id:  Optional[str]   # qa_gen_results.id (export용)
    layers_status:  Dict            # 4레이어 각각의 상태
      └── {syntax, stats, rag, quality}
              └── {status, progress, message}

class EvaluationManager:
    jobs: Dict[str, EvalJob]
    lock: threading.Lock
```

**생명주기**:
```
POST /api/evaluate
  → EvaluationManager.create_job(job_id)
  → BackgroundTasks.add_task(run_evaluation, job_id, ...)
  → 반환: {job_id}

프론트 폴링: GET /api/evaluate/{job_id}/status (1초 간격)
  → {status, progress, message, layers, eval_report}

완료 후:
  → eval_report → Supabase save → supabase_eval_id
  → link_generation_to_evaluation(generation_id, eval_id)
  → Job 메모리에 유지 (서버 재시작 시 소멸)
```

### 3-3. 세션 관리 한계 (현재)

| 항목 | 현황 | 문제 |
|------|------|------|
| 저장소 | Python 프로세스 메모리 (`Dict`) | 서버 재시작 시 모든 Job 소멸 |
| 영속성 | Supabase (완료된 결과만) | 진행 중 Job은 재시작 후 복구 불가 |
| 확장성 | 단일 프로세스 | 다중 워커(uvicorn workers > 1) 시 Job 공유 불가 |
| 정리 | 없음 | Job 무한 누적 (메모리 누수 가능) |

**Production 고려 시**: Redis + Celery 또는 Supabase jobs 테이블로 이전 필요

## 3-5. Hierarchy L2/L3 개선 계획 — 3-Pass Master 생성 (예정)

### 문제점 (현재)

현재 `apply-granular-tagging`은 청크마다 L2/L3를 **자유 생성**해서 청크 수만큼 L2가 늘어나는 문제 발생.
- L1은 문서 전체 기준 master list 사전 확정 후 선택 → 일관성 유지
- L2/L3는 청크별 개별 생성 → 중복·유사 값 난립

### 개선 방향: Method 3 (L2+L3 master 동시 생성)

```
1단계: analyze-hierarchy (기존 유지)
  ├── 입력: 전체 청크 샘플 (20~30개)
  ├── 처리: LLM → 문서 전체 주제 파악
  └── 출력: L1 master list (3~5개)
           예: ["규제 준수", "기술 개발", "시장 분석"]

           ↓ L1 master list 전달

2단계: analyze-l2-l3 (신규 엔드포인트)
  ├── 입력: L1 master list + 전체 청크 샘플
  ├── 처리: LLM 1회 호출 → L2 목록 + L3 후보 동시 생성
  └── 출력: L1별 L2 목록 + L2별 L3 후보
           {
             "규제 준수": {
               "법적 요건": ["요건 정의", "적용 범위"],
               "심사 기준": ["평가 항목", "합격 기준"],
               "위반 사례": ["사례 유형", "제재 내용"]
             },
             "기술 개발": {
               "핵심 기능": ["기능 설명", "구현 방식"],
               "성능 지표": ["측정 기준", "달성 목표"]
             }
           }
           → 프론트에 반환 (사용자 검토/수정 가능)

           ↓ L1/L2/L3 master dict 전달

3단계: apply-granular-tagging (개선)
  ├── 입력: 각 청크 content + L1/L2/L3 master dict
  ├── 처리: LLM은 생성 X, 선택만
  │         - L1: master list 중 택1
  │         - L2: 선택한 L1의 L2 목록 중 택1
  │         - L3: 선택한 L2의 L3 후보 중 택1
  │              (없으면 가장 유사한 것 선택)
  └── 출력: doc_chunks.metadata 일괄 업데이트
```

### 수량 제한 기준

| 레벨 | 문서당 수량 | 비고 |
|------|------------|------|
| L1 | 3~5개 | 기존과 동일 |
| L2 | L1당 2~5개 | 신규 제약 |
| L3 | L2당 2~4개 | 신규 제약 |

### 구현 계획

**1. 신규 엔드포인트**: `POST /api/ingestion/analyze-l2-l3`
- 요청: `{filename, selected_l1_list}`
- 응답: `{l2_l3_master: {L1: {L2: [L3, ...]}}}`

**2. `apply-granular-tagging` 요청 스키마 변경**
- 기존: `{filename, selected_l1_list}`
- 변경: `{filename, selected_l1_list, l2_l3_master}` 추가

**3. `_build_prompt()` 수정**
- 기존: L1 list만 주입, L2/L3 자유 생성
- 변경: L1/L2/L3 master dict 전체 주입, 선택 지시로 변경

**4. 프론트엔드 `DataStandardizationPanel.tsx`**
- analyze-hierarchy → analyze-l2-l3 → apply-granular-tagging 순서로 호출 추가
- 2단계 결과를 사용자에게 표시 (검토/수정 UI 선택적)

### 엣지 케이스 처리

- L3 후보 중 맞는 것 없을 시 → "가장 유사한 것 선택" 프롬프트 지시
- 특정 L2에 청크가 0개 → 정상 (문서에 해당 주제 없는 것)
- L2/L3 후보가 너무 적을 시 → 2단계 재실행 또는 수동 추가

```

  └── handleStart()

                → completed: startEvaluation(result_file, result_id)

  startEvaluation()
    └── POST /api/evaluate {result_filename, generation_id}
          → evalJobId 저장

  useEffect([evalJobId])
    └── setInterval(1000ms)
          └── GET /api/evaluate/{evalJobId}/status
                → {status, progress, layers, eval_report}
                → completed: setPhase('complete'), onEvalComplete(evalJobId)
```

  └── 히스토리: GET /api/evaluate/history  (Supabase qa_eval_results)

## 6. 주요 설계 결정 & 트레이드오프

| 결정 | 이유 | 트레이드오프 |
|------|------|-------------|
| In-memory Job 관리 | 구현 단순, 빠른 응답 | 서버 재시작 시 Job 소멸 |
| qa_list 비정규화 저장 | doc_chunks 재조회 없이 재현 가능 | doc_chunks 수정 반영 안 됨 |
| doc_chunk_ids uuid[] (신규) | doc_chunks → qa_gen 추적 가능 | 기존 레코드 백필 필요 |
| BackgroundTasks (FastAPI) | 간단한 비동기 실행 | 큐 관리 없음, 동시 실행 수 제한 없음 |
| 1초 폴링 (프론트) | 구현 간단 | WebSocket 대비 네트워크 비효율 |
| final_score float8 0–1 | 정밀도 유지 | 프론트에서 `/100` 표시 시 혼동 가능 |
