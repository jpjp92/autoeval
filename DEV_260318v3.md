# AutoEval — 골든셋(Golden Set) 설계 계획 (2026-03-18)

---

## 1. 골든셋이란

AutoEval 파이프라인에서 **사람 수준의 품질 기준을 통과한 QA 쌍 모음**.
자동 생성된 QA 중 고품질 후보를 추출하여 파이프라인 전반의 기준점으로 활용합니다.

---

## 2. 목적 & 용도

| 용도 | 설명 | 단계 |
|------|------|------|
| **Few-shot 예시** | QA 생성 프롬프트에 고품질 예시 삽입 → 모델이 품질 기준 학습 | 생성 |
| **평가 캘리브레이션** | LLM 평가자(Haiku 등)가 골든셋 채점 → 사람 기준과 비교, 평가 신뢰도 검증 | 평가 |
| **RAG 검색 품질 평가** | 골든셋 질문으로 벡터 검색 → hit rate / recall 측정 | 검색 |
| **회귀 테스트** | 프롬프트·모델 변경 후 골든셋 재평가 → 품질 변화 추적 | 모니터링 |

---

## 3. 진입 기준 (Quality Gate)

### 3-1. 자동 후보 추출 (v1)

평가 완료된 QA 중 아래 기준을 **모두 통과**한 것만 골든셋 후보로 추출:

| 평가 레이어 | 기준 |
|-------------|------|
| Syntax (Layer 1-A) | 100% pass |
| RAG Triad (Layer 2) | relevance / groundedness / clarity **모두 0.85+** |
| Quality (Layer 3) | factuality / completeness / groundedness **모두 0.85+** |

```python
# 자동 후보 추출 조건 (예시)
def is_golden_candidate(qa_score: dict) -> bool:
    rag   = qa_score.get("rag", {})
    qual  = qa_score.get("quality", {})
    return (
        qa_score.get("syntax_pass", False)
        and rag.get("relevance",    0) >= 0.85
        and rag.get("groundedness", 0) >= 0.85
        and rag.get("clarity",      0) >= 0.85
        and qual.get("factuality",   0) >= 0.85
        and qual.get("completeness", 0) >= 0.85
        and qual.get("groundedness", 0) >= 0.85
    )
```

### 3-2. 중복 제거

이미 등록된 골든셋과 **semantic similarity < 0.85** 인 경우만 추가:
- Gemini Embedding 2로 질문 임베딩 → cosine similarity 계산
- 유사도 0.85 이상이면 중복으로 판단 → 기존 골든셋 유지 (새 항목 제외)

### 3-3. 사람 검토 (v2, 선택)

v1 자동 승인 이후 검토 레이어 추가:

```
후보 큐 → 검토자 approve / reject / edit
  - approve: qa_golden_set 확정 등록 (approved_by = 검토자명)
  - reject:  후보에서 제거
  - edit:    질문/답변 수정 후 등록
```

## 4. 다양성 기준

골든셋이 특정 유형에 편중되지 않도록 제약:

| 항목 | 기준 |
|------|------|
| 의도 유형 | 8개 유형 대표 (유형별 최소 2~3개) |
| 계층 | L1별 최소 n개 (문서 L1 구조 커버) |
| 답변 길이 | 단문(1~2문장) / 중문(3~5문장) 혼합 |
| 동일 청크 출처 | 동일 청크에서 최대 2개 (출처 다양성) |

## 5. DB 스키마

### qa_golden_set 테이블

```sql
CREATE TABLE qa_golden_set (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  source_qa_id    text,                           -- 원본 QA ID (qa_gen_results.qa_list[*].qa_id)
  source_gen_id   uuid        REFERENCES qa_gen_results(id),  -- 원본 생성 Job
  source_eval_id  uuid        REFERENCES qa_eval_results(id), -- 평가 결과 참조
  doc_name        text        NOT NULL,
  hierarchy_l1    text,
  hierarchy_l2    text,
  hierarchy_l3    text,
  question        text        NOT NULL,
  answer          text        NOT NULL,
  intent          text,                           -- factoid | numeric | procedure | ...
  context         text,                           -- 청크 원문 (RAG 평가·few-shot용)
  quality_scores  jsonb,                          -- {rag: {...}, quality: {...}} 평가 점수 스냅샷
  approved_by     text        DEFAULT 'auto',     -- 'auto' | 검토자 이름
  is_active       boolean     DEFAULT true,       -- 비활성화 가능 (삭제 대신 soft delete)
  created_at      timestamptz DEFAULT now()
);

-- 인덱스
CREATE INDEX ON qa_golden_set (doc_name);
CREATE INDEX ON qa_golden_set (intent);
CREATE INDEX ON qa_golden_set (hierarchy_l1, hierarchy_l2);
CREATE INDEX ON qa_golden_set (is_active);
```

### 연계 관계

```
qa_gen_results.id     ← qa_golden_set.source_gen_id
qa_eval_results.id    ← qa_golden_set.source_eval_id
doc_chunks (원문)     ← qa_golden_set.context (직접 저장, 검색 불필요)
```

> ※ `context`를 doc_chunks 참조 대신 직접 저장하는 이유:
> - 청크가 삭제/재처리되어도 골든셋 독립성 유지
> - Few-shot / RAG 평가 시 별도 조회 불필요

```
[평가 완료 (qa_eval_results)]
          │

          │
          ▼  중복 제거 (embedding similarity)
[검토 큐]
          │
    ┌─────┴──────┐
    ▼            ▼
 자동 승인      사람 검토
 (v1, 점수만)   (v2, approve/reject/edit)
    │            │
    └─────┬──────┘
          ▼
  [qa_golden_set 테이블]
          │
    ┌─────┼──────────┐
    ▼     ▼          ▼
 Few-shot 평가캘리브  RAG hit-rate
 프롬프트 레이션      측정
```

## 7. 활용 방안 상세

### 7-1. Few-shot 예시 (생성 프롬프트)

```python
# build_user_template()에서 골든셋 예시 삽입
golden_examples = fetch_golden_by_doc(doc_name, intent_types, n=2)

<examples>
{golden_examples[0].question}
{golden_examples[0].answer}
</examples>
```

- 같은 문서·같은 L1에서 예시 우선 선택
- 없으면 동일 intent 유형 골든셋에서 선택

### 7-2. 평가 캘리브레이션

```
골든셋 QA → LLM 평가자로 채점
→ 사람 기준 점수와 비교 (pearson correlation)
→ 평가자 신뢰도 지수(Calibration Score) 산출
→ 평가 결과 해석 시 보정 계수로 활용
```

### 7-3. RAG 검색 품질 평가

```
골든셋 question → 벡터 검색 → 상위 k개 청크 반환
→ 원본 context와 일치 여부 확인
→ Hit@1 / Hit@3 / Hit@5 측정
→ 임베딩 모델·청킹 전략 개선 지표로 활용
```

### 7-4. 회귀 테스트

```
프롬프트·모델 변경 후
→ 골든셋 전체 재평가
→ 이전 평가 점수와 비교 (delta)
→ 품질 저하 감지 시 변경 롤백
```

## 8. 설계 결정 항목

| 결정 항목 | 선택 | 이유 |
|-----------|------|------|
| 승인 방식 | v1 자동 → v2 사람 검토 단계적 도입 | 빠른 구축 후 신뢰도 향상 |
| 저장 단위 | 개별 QA 단위 | Few-shot·캘리브레이션 모두 QA 단위 활용 |
| 컨텍스트 저장 | 청크 원문 직접 저장 | 독립성 보장, 별도 조회 불필요 |
| 중복 기준 | semantic similarity < 0.85 | 의미적 중복 방지 (lexical 중복과 별개) |
| 비활성화 | soft delete (`is_active=false`) | 이력 보존, 롤백 가능 |

## 9. 구현 순서 (예정)

```
Phase 1 (최소 구현):
  └── qa_golden_set 테이블 생성 (Supabase)
  └── supabase_client.py: extract_golden_candidates(), save_golden_set()
  └── 평가 완료 후 자동 후보 추출 & 저장 (pipeline.py 연동)

Phase 2:
  └── 중복 제거 로직 (embedding similarity)
  └── 다양성 기준 적용 (유형별 / L1별 비율 체크)

Phase 3:
  └── Few-shot 프롬프트 연동 (prompts.py)
  └── RAG hit-rate 평가 엔드포인트
  └── 평가 캘리브레이션 리포트
```
