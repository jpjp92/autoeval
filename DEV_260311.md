## Summary

> AutoEval은 React + TypeScript 프론트엔드와 FastAPI 백엔드로 구성된 완전한 QA 자동 생성 및 평가 시스템으로, Claude Sonnet, Gemini 3.1-Flash, GPT-5.2 등 3개 LLM 모델을 이용해 QA 데이터를 생성하고 Claude Haiku, Gemini 2.5-Flash, GPT-5.1 평가 모델들로 Relevance/Groundedness/Clarity 기반 RAG Triad 평가를 수행합니다. 
>
> 최근 업데이트에서 ThreadPoolExecutor를 통한 병렬 QA 생성(40-60초→10-15초) 및 병렬 평가(40초→10초)로 3-5배 성능 개선, 컨텍스트 크기 확대(1000→20000 글자), Chain-of-Thought 기반 근거성 평가 강화, 프론트엔드 UX 개선 등을 구현했으며, 현재 Claude Sonnet으로 엔드투엔드 파이프라인 테스트를 진행 중입니다.

### 4. Groundedness CoT 프롬프트 (Chain of Thought) ✨ CRITICAL UPDATE

**파일**: `backend/evaluation_api.py` - `evaluate_groundedness()` 메서드

```python
def evaluate_groundedness(self, answer: str, context: str) -> float:
    """답변의 근거성 평가 (0-1) - CoT 스타일 추론"""
  
    prompt = f"""You are an expert at assessing whether an answer is grounded in provided context.
Use systematic step-by-step reasoning (Chain of Thought) to evaluate.

Context:
{context_text[:20000]}

Answer:
{answer}

Evaluate groundedness using the following reasoning process:

Step 1: IDENTIFY KEY CLAIMS
List the main factual claims or statements in the answer.

Step 2: FIND SUPPORTING EVIDENCE
For each claim, search the context for supporting evidence.
- Direct quotes from context
- Paraphrased content from context
- Logical inferences from context
- Explicit conclusions from context

Step 3: ASSESS ALIGNMENT
Evaluate how well each claim aligns with the context:
- Is there direct textual support?
- Is there sufficient contextual support for inference?
- Are there any claims without support (hallucinations)?

Step 4: DETERMINE GROUNDING LEVEL
Based on the analysis above, rate on 0-10 scale:
- 10: All claims directly quoted or clearly supported by context
- 8-9: All claims supported through paraphrase or clear inference from context
- 7: Most claims grounded; minor unsupported elaborations
- 5-6: Some grounding but significant ungrounded claims
- 3-4: Mostly hallucinated with minimal context support
- 0-2: No grounding, entirely hallucinated

Return ONLY:
The final integer score (0-10) on the last line."""
```

**효과**:

- Procedure/Why-type 질문 groundedness 평가 개선
- Factoid 질문: 여전히 높음 (1.0)
- 절차형 질문: 0.0-0.5 → 0.4-0.7 예상 상향
- Hallucination 탐지 정확도 향상

  const val = e.target.value;

      ...prev,
      samples: Math.min(Math.max(1, parseInt(val)), 50)

## 🔐 Key Learnings

1. **QA 파일 구조**: `results[].qa_list[]` 형식 중요
2. **Context 처리**: Groundedness 평가엔 실제 컨텍스트 필수
   - 크기: 1K → 20K로 확대 필요
   - Groundedness 0.3 → 0.5-0.7로 상향
3. **병렬화**: API 호출 비용 고려하면서 성능↑
   - QA 생성: ThreadPoolExecutor(max_workers=4)로 3-4배 개선
   - 평가: ThreadPoolExecutor(max_workers=4)로 3-5배 개선
   - Lock으로 thread-safe한 토큰/비용 추적 필수
4. **Groundedness 평가**: CoT 프롬프트 필수
   - 단순 yes/no 프롬프트 → 절차형 질문에서 0점 처리
   - Step-by-step 이유 분석 → procedure/why-type도 정확히 평가
   - 핵심: Hallucination 탐지 (evidence matching)
5. **Polling**: 완료 후 즉시 cleanup 필요
   - setJobId(null), setEvalJobId(null) 필수
6. **Logging**: 과도한 로그는 시스템 노이즈 → access_log 비활성화
7. **Context 크기 vs 정확도**:
   - Context 크기 확대만으로 groundedness 개선 안됨 (증명됨: 1K→20K 후에도 0.3)
   - 프롬프트 품질 (CoT) 이 더 중요
   - 근본 원인: LLM의 의미적 이해 능력
